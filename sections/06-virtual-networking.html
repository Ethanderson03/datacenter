Layer 4 06

Virtual Networking & Overlay Fabrics

Software-defined networking with Open vSwitch, VXLAN overlays, DPDK acceleration, and hardware offload for GPU-dense HPC clusters

This section explains the software-defined network layer that sits alongside the high-speed data fabric. In HPC environments, virtual networking is typically used for management, storage, tenant segmentation, and service traffic, while operators work carefully to avoid adding overhead to the performance-critical GPU data path.

Topics covered here include Open vSwitch, day-to-day OVS operations, VXLAN overlays, OVS with DPDK, virtual topology patterns, SmartNIC and DPU offload, and security controls for virtualized network paths.

1. Open vSwitch (OVS) Deep Dive

Open vSwitch is a production-quality, multilayer, software-defined virtual switch. Licensed under Apache 2.0, it is the de-facto standard virtual switch in datacenter environments running OpenStack, Kubernetes, and bare-metal GPU clusters. OVS implements the OpenFlow protocol, enabling centralized SDN controllers to program forwarding behavior across thousands of hosts from a single control point.

Why OVS in an HPC / GPU Superpod? In NVIDIA DGX SuperPOD deployments, OVS is used on host nodes and inside BlueField DPU ARM cores to steer storage (NVMe-oF), management, and in-band control traffic while keeping the high-speed InfiniBand / RoCE GPU-to-GPU data plane untouched.

Core Architecture

OVS is composed of four tightly-coupled components:

Component Runs In Role
ovs-vswitchd Userspace (daemon) Main forwarding daemon. Implements OpenFlow pipeline, communicates with ovsdb-server for config and with kernel/DPDK datapath for fast-path forwarding. Handles flow table lookups, flow installation, and slow-path packet processing (table-miss packets sent up from kernel).
ovsdb-server Userspace (daemon) Persistent configuration database using the OVSDB protocol (RFC 7047). Stores bridge, port, interface, QoS, mirror, and controller configuration. Accessed via ovs-vsctl CLI or direct JSON-RPC. Supports clustering for HA (raft-based consensus since OVS 2.9).
Kernel Datapath (openvswitch.ko) Kernel module High-speed fast path. Receives packets from NICs and virtual ports, performs exact-match flow lookup in a hash table, and executes cached actions (output, set-field, encap/decap). On a miss, packets are sent to ovs-vswitchd via Netlink for OpenFlow classification; the resulting actions are then cached in the kernel for subsequent packets of the same flow.
Userspace Datapath (DPDK) Userspace (PMD threads) Replaces the kernel datapath entirely. Uses DPDK poll-mode drivers to bypass the kernel networking stack. Packets are read directly from NIC RX queues via memory-mapped DMA, processed in userspace, and written to TX queues — achieving 10–100x throughput improvement with deterministic latency. Requires hugepages and dedicated CPU cores.

Key Concepts

Bridges (br0, br-int, br-ex)

+

An OVS bridge is a virtual Ethernet switch, analogous to a physical L2 switch. Each bridge has its own MAC learning table, STP/RSTP instance, and OpenFlow pipeline. In a typical OpenStack or cloud deployment, three bridges are standard:

  • br-int (integration bridge) — Every VM tap port, container veth, and internal service port lands here. All tenant traffic enters the OVS forwarding pipeline through br-int. OpenFlow rules on this bridge handle VLAN tagging, security group enforcement (via conntrack), and routing to other bridges.
  • br-ex (external bridge) — Connected to the physical NIC(s) that provide upstream connectivity. Floating IPs, SNAT, and provider network traffic egress through br-ex to the physical fabric.
  • br-tun (tunnel bridge) — Handles VXLAN/GRE/Geneve encapsulation and decapsulation. Patch ports connect br-int to br-tun; tunnel ports connect br-tun to remote VTEPs. Flow rules map internal VLAN tags to VNIs for each tenant network.

Ports: internal, system, tap, patch, VXLAN, GRE

+
Port Type Description Typical Use
system Represents a Linux network interface (e.g., eth0, ens3f0). OVS takes over the port and handles all traffic. Uplink to physical network
internal Creates a virtual network interface in the host. The bridge's local port (br0) is always internal. Additional internal ports can serve as management IPs on the bridge. Host management IP on bridge
tap TUN/TAP device created by hypervisor (QEMU/KVM) or container runtime. Appears as a system port to OVS. VM connectivity
patch Software-only port connecting two OVS bridges in-memory (no kernel overhead). Always created in pairs. br-int ↔ br-ex, br-int ↔ br-tun
vxlan VXLAN tunnel endpoint. Encapsulates L2 frames in UDP/IP. Configured with remote_ip and key (VNI). Overlay networking across L3
gre GRE tunnel endpoint. Encapsulates L2 frames in GRE/IP. Lower overhead than VXLAN (no UDP) but no entropy for ECMP. Legacy overlay tunnels
geneve Generic Network Virtualization Encapsulation. Extensible TLV-based metadata. Preferred in modern deployments (OVN default). OVN-based clouds, metadata-rich overlays

OpenFlow Rules: Matching, Actions & Multi-Table Pipeline

+

OVS implements the OpenFlow protocol (versions 1.0 through 1.5). Each bridge has up to 255 flow tables (table 0 through 254). Packets enter at table 0 and traverse tables via resubmit or goto_table actions. Each flow rule has:

  • Priority (0–65535): higher wins on match
  • Match fields: in_port, dl_src, dl_dst, dl_type, dl_vlan, nw_src, nw_dst, nw_proto, tp_src, tp_dst, metadata, tunnel_id, reg0–reg15, and more
  • Actions: output, drop, mod_dl_dst, mod_nw_src, push_vlan, pop_vlan, set_tunnel, set_field, ct (conntrack), group, meter, resubmit, goto_table
  • Counters: packet count, byte count, duration — for monitoring and debugging

A typical multi-table pipeline (as used by OVN or OpenStack Neutron OVS agent):

TablePurpose
0Classify ingress: local vs. tunnel, tag with metadata/registers
1–10Pre-ACL, ingress security group (conntrack zone assignment)
20–30L2 lookup (MAC learning or programmed MACs), unknown unicast handling
40–50L3 routing (if distributed routing is enabled)
60–70Egress ACL / security groups
80+Egress: select output port, VXLAN encap, physical NIC output

2. OVS Commands & Configuration

OVS provides several CLI tools, each targeting a different component. ovs-vsctl manipulates the OVSDB configuration (bridges, ports, interfaces). ovs-ofctl programs OpenFlow rules directly on a bridge. ovs-appctl sends runtime commands to the ovs-vswitchd daemon. ovs-dpctl inspects and manages the kernel datapath.

Bridge & Port Management

# Create a bridge
ovs-vsctl add-br br-int

# List all bridges
ovs-vsctl list-br

# Add a physical NIC as a port (system type, default)
ovs-vsctl add-port br-int eth0

# Add an internal port (creates a Linux netdev for host IP assignment)
ovs-vsctl add-port br-int mgmt0 -- set interface mgmt0 type=internal
ip addr add 10.0.0.1/24 dev mgmt0
ip link set mgmt0 up

# Add a VLAN access port
ovs-vsctl add-port br-int eth1 tag=100

# Add a VLAN trunk port allowing VLANs 100, 200, 300
ovs-vsctl add-port br-int eth2 trunks=100,200,300

# Create patch ports connecting br-int ↔ br-ex
ovs-vsctl add-port br-int patch-to-ex -- set interface patch-to-ex type=patch options:peer=patch-to-int
ovs-vsctl add-port br-ex patch-to-int -- set interface patch-to-int type=patch options:peer=patch-to-ex

# Show full bridge/port/interface hierarchy
ovs-vsctl show

VXLAN Tunnel Configuration

# Add a VXLAN tunnel port with a fixed VNI (key)
ovs-vsctl add-port br-int vxlan0 \
  -- set interface vxlan0 type=vxlan \
     options:remote_ip=10.0.0.2 \
     options:key=1000

# Add a VXLAN tunnel with flow-based VNI (key set per-flow)

  -- set interface vxlan-flow type=vxlan \
     options:remote_ip=flow \
     options:key=flow

# Geneve tunnel (OVN default, supports TLV metadata)
ovs-vsctl add-port br-int geneve0 \
  -- set interface geneve0 type=geneve \
     options:remote_ip=10.0.0.2 \
     options:key=5000

OpenFlow Rule Management

# Dump all flows on br-int (with stats)
ovs-ofctl dump-flows br-int

# Dump flows for a specific table
ovs-ofctl dump-flows br-int table=20

# Add a basic L3 forwarding rule
# Match: packets on port 1, IPv4, destination 10.0.1.0/24
# Action: output on port 2
ovs-ofctl add-flow br-int \
  "table=0,priority=100,in_port=1,dl_type=0x0800,nw_dst=10.0.1.0/24,actions=output:2"

# Add a rule to encapsulate in VXLAN (set tunnel ID, output to tunnel port)
ovs-ofctl add-flow br-int \
  "table=0,priority=90,in_port=1,dl_type=0x0800,actions=set_field:1000->tun_id,output:10"

# Drop broadcast on a specific port
ovs-ofctl add-flow br-int \
  "table=0,priority=200,in_port=3,dl_dst=ff:ff:ff:ff:ff:ff,actions=drop"

# Stateful firewall with conntrack (allow established, track new)
ovs-ofctl add-flow br-int \
  "table=0,priority=100,ct_state=-trk,actions=ct(table=1)"
ovs-ofctl add-flow br-int \
  "table=1,priority=100,ct_state=+trk+est,actions=output:NORMAL"
ovs-ofctl add-flow br-int \
  "table=1,priority=50,ct_state=+trk+new,ip,nw_dst=10.0.1.0/24,actions=ct(commit),output:2"

# Delete a specific flow
ovs-ofctl del-flows br-int "in_port=1,dl_type=0x0800,nw_dst=10.0.1.0/24"

# Delete all flows on the bridge (caution!)
ovs-ofctl del-flows br-int

3. VXLAN (Virtual Extensible LAN)

VXLAN (RFC 7348) is the dominant overlay encapsulation protocol in modern datacenters. It solves the fundamental VLAN scalability problem: 802.1Q provides only a 12-bit VLAN ID field (4,094 usable VLANs), which is wholly inadequate for multi-tenant cloud environments. VXLAN provides a 24-bit VNI (VXLAN Network Identifier), supporting over 16 million logical network segments.

Why VXLAN?

  • Scale: 16,777,216 (224) segments vs. 4,094 VLANs
  • L2 over L3: Extend Layer 2 domains across routed Layer 3 boundaries without spanning tree
  • Workload mobility: VMs/containers can migrate across L3 boundaries while retaining their MAC and IP addresses
  • Decoupled from physical topology: Overlay topology is independent of underlay — no recabling, no VLAN provisioning on every switch
  • ECMP-friendly: UDP encapsulation provides entropy in the source port for equal-cost multi-path load balancing across the underlay fabric

VXLAN Encapsulation Format

An original L2 frame is encapsulated inside a UDP packet destined to port 4789. This adds 50 bytes of overhead (14 outer Ethernet + 20 outer IP + 8 UDP + 8 VXLAN).

Key VXLAN Components

Component Description
VNI (VXLAN Network Identifier) 24-bit segment ID carried in the VXLAN header. Identifies the Layer 2 broadcast domain. Each tenant/network gets a unique VNI. Range: 0–16,777,215.
VTEP (VXLAN Tunnel Endpoint) The entity that performs encapsulation and decapsulation. Can be a software switch (OVS on a hypervisor), a hardware switch (leaf/spine ToR), or a SmartNIC/DPU (BlueField). Each VTEP has at least one IP address on the underlay network used as the outer source/destination IP.
Underlay Network The physical L3 routed fabric (typically a leaf-spine Clos topology). Must support the VXLAN MTU: original frame (1500) + 50 bytes overhead = 1550 minimum. In practice, jumbo frames (MTU 9000 or 9216) are configured on all underlay interfaces to avoid fragmentation.
BUM Traffic Broadcast, Unknown unicast, Multicast. Three approaches: (1) Multicast groups — map VNI to an IP multicast group, flood via PIM/IGMP. (2) Ingress replication — VTEP unicast-copies to all remote VTEPs in the VNI. (3) EVPN — BGP advertises MAC/IP bindings, eliminating flood-and-learn entirely.

MTU Considerations

Critical: MTU Planning VXLAN adds 50 bytes of encapsulation overhead. If your VMs use a standard 1500-byte MTU, the underlay must support at least 1550-byte frames. However, best practice is to configure the underlay fabric (all leaf/spine switch interfaces and host NICs) with jumbo frames (MTU 9000 or 9216). This avoids fragmentation, allows inner jumbo frames for storage traffic (NFS, iSCSI), and accommodates double-encapsulation scenarios (e.g., VXLAN inside GRE). In NVIDIA SuperPOD deployments, the InfiniBand fabric natively supports 4K MTU, and the Ethernet management/storage networks are set to 9000-byte MTU.

VXLAN + EVPN: BGP-Based Control Plane

While basic VXLAN uses flood-and-learn for MAC address discovery (similar to traditional Ethernet switches), this is inefficient at scale. EVPN (Ethernet VPN, RFC 7432 / RFC 8365) provides a BGP-based control plane that distributes MAC and IP reachability information across all VTEPs, eliminating the need for data-plane flooding.

EVPN Route Types

+
Route Type Name Purpose
Type-1 Ethernet Auto-Discovery Multi-homing: advertise Ethernet Segment (ES) membership for fast convergence and split-horizon
Type-2 MAC/IP Advertisement Advertise [MAC, IP, VNI, VTEP-IP] bindings. This is the core route type — replaces flood-and-learn. Supports both MAC-only and MAC+IP for ARP suppression.
Type-3 Inclusive Multicast Ethernet Tag Establishes the ingress replication list for BUM traffic. Each VTEP advertises its interest in a VNI.
Type-4 Ethernet Segment Designated Forwarder (DF) election for multi-homed Ethernet segments
Type-5 IP Prefix Route Advertise IP prefixes (subnets) for inter-VXLAN routing. Enables symmetric IRB (Integrated Routing and Bridging) on leaf switches for efficient L3 forwarding.

With EVPN Type-2 routes, when a VM comes up on VTEP-A, VTEP-A immediately advertises the VM's MAC and IP via BGP to all other VTEPs. When VTEP-B receives a packet destined for that MAC, it already knows the remote VTEP IP — no flooding required. EVPN Type-5 routes extend this to L3: an entire subnet prefix can be advertised, enabling distributed routing where each leaf switch performs local L3 forwarding.

4. OVS with DPDK

The Data Plane Development Kit (DPDK) is a set of libraries and drivers for fast packet processing that bypasses the Linux kernel networking stack entirely. Instead of relying on interrupt-driven packet reception via sk_buff allocations, DPDK maps NIC RX/TX queues directly into userspace memory and uses dedicated poll-mode driver (PMD) threads that continuously spin on the queues, achieving zero-copy, zero-interrupt packet processing.

Performance Comparison

Metric OVS Kernel Datapath OVS-DPDK
Throughput (64-byte packets) ~1–3 Mpps per core ~15–40 Mpps per core
Latency (p99) 20–100 µs 5–15 µs
Jitter High (interrupt coalescing, scheduling) Very low (busy-poll, no context switches)
CPU utilization Low idle, spikes on load 100% on PMD cores (always polling)
Max line-rate (100 GbE) Difficult at small packet sizes Achievable with 2–4 PMD cores

Requirements

  • Hugepages: DPDK requires hugepage memory (2 MB or 1 GB pages) to avoid TLB misses. For a 100 GbE NIC with large flow tables, allocate 4–8 GB of 1 GB hugepages per NUMA node.
  • CPU pinning: PMD threads must be pinned to dedicated CPU cores, isolated from the OS scheduler (isolcpus or cset). Place PMD cores on the same NUMA node as the NIC for optimal performance.
  • IOMMU: Required for VFIO-based NIC binding (vfio-pci driver). Set intel_iommu=on iommu=pt or amd_iommu=on iommu=pt in kernel boot parameters.
  • NIC driver binding: NICs must be unbound from the kernel driver and bound to a DPDK-compatible driver (vfio-pci or uio_pci_generic).

Configuration

# 1. Allocate hugepages (1 GB pages, 8 pages per NUMA node)
echo 8 > /sys/devices/system/node/node0/hugepages/hugepages-1048576kB/nr_hugepages
echo 8 > /sys/devices/system/node/node1/hugepages/hugepages-1048576kB/nr_hugepages
mkdir -p /dev/hugepages
mount -t hugetlbfs none /dev/hugepages

# 2. Enable DPDK in OVS
ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true

# 3. Allocate DPDK memory per NUMA socket (MB)
ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem="4096,4096"

# 4. Set PMD CPU mask (cores 2,3 on NUMA 0; cores 10,11 on NUMA 1)
# Binary: 0000110000001100 = 0x0C0C
ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x0C0C

# 5. Restart OVS to apply DPDK init
systemctl restart openvswitch-switch

# 6. Create a DPDK-backed bridge (datapath_type=netdev is required)
ovs-vsctl add-br br-dpdk -- set bridge br-dpdk datapath_type=netdev

# 7. Bind NIC to DPDK (using dpdk-devargs PCI address)
# First unbind from kernel driver:
dpdk-devbind.py --bind=vfio-pci 0000:03:00.0

# Add as DPDK port:
ovs-vsctl add-port br-dpdk dpdk-p0 \
  -- set Interface dpdk-p0 type=dpdk \
     options:dpdk-devargs=0000:03:00.0

# 8. Add a vhost-user port for VM connectivity
ovs-vsctl add-port br-dpdk vhost-vm1 \
  -- set Interface vhost-vm1 type=dpdkvhostuserclient \
     options:vhost-server-path=/var/run/openvswitch/vhost-vm1

# 9. Set interface MTU (e.g., 9000 for jumbo frames)
ovs-vsctl set Interface dpdk-p0 mtu_request=9000

# 10. Configure RX/TX queues per PMD
ovs-vsctl set Interface dpdk-p0 options:n_rxq=4
ovs-vsctl set Interface dpdk-p0 options:n_txq=4
NUMA Affinity Matters In a dual-socket server (common in DGX-class nodes), placing PMD threads on the same NUMA node as the NIC's PCIe slot avoids cross-socket QPI/UPI traffic, which can add 100+ ns of latency per packet. Use lstopo or lspci -vmms to determine the NIC's NUMA node, then pin PMD cores accordingly via pmd-cpu-mask.

5. Virtual Network Topologies

Provider Networks (VLAN-based)

In a provider network model, VMs are connected directly to the physical network using VLAN tagging. The hypervisor host's OVS bridge tags/untags traffic on the appropriate VLAN, and the physical ToR switch carries the tagged frames upstream. This model provides the best performance (no encapsulation overhead) and simplest troubleshooting, but requires VLAN provisioning on every physical switch in the path.

Tenant / Overlay Networks (VXLAN/GRE)

Overlay networks provide full tenant isolation without any physical switch configuration. Each tenant gets one or more VNIs. Traffic between VMs on the same VNI is encapsulated in VXLAN and tunneled over the L3 underlay fabric. The physical switches see only regular IP/UDP traffic between VTEP addresses — completely unaware of the overlay topology.

Network Namespaces for Routing Isolation

Linux network namespaces (ip netns) provide complete routing table isolation. Each namespace has its own interfaces, routing table, iptables rules, and ARP cache. In OpenStack, each tenant router runs in its own namespace (qrouter-*), and each DHCP agent runs in a namespace (qdhcp-*). Namespaces connect to OVS via veth pairs or OVS internal ports. 

# Create a network namespace for tenant routing
ip netns add tenant-router-1

# Create a veth pair and place one end in the namespace
ip link add veth-host type veth peer name veth-ns
ip link set veth-ns netns tenant-router-1

# Configure the namespace side
ip netns exec tenant-router-1 ip addr add 10.0.1.1/24 dev veth-ns
ip netns exec tenant-router-1 ip link set veth-ns up
ip netns exec tenant-router-1 ip link set lo up

# Attach host side to OVS bridge
ovs-vsctl add-port br-int veth-host
ip link set veth-host up

# Enable NAT in the namespace for external access
ip netns exec tenant-router-1 iptables -t nat -A POSTROUTING \
  -o veth-ext -j MASQUERADE

Security: iptables/nftables for Security Groups and NAT

In virtualized environments, security groups (per-port ACLs) are enforced at the hypervisor level. Two primary implementation strategies exist:

  • OVS conntrack (preferred): Stateful firewall rules implemented directly in OVS OpenFlow using the ct() action. Connection tracking is performed in the kernel's Netfilter conntrack subsystem. This approach avoids the per-packet iptables overhead and scales better with port count.
  • iptables/nftables bridge: Legacy approach where traffic is diverted through a Linux bridge with iptables rules before reaching OVS. Each security group rule becomes an iptables chain entry. Functional but slower and harder to manage at scale.
# OVS conntrack-based security group example:

# Table 0: Send all traffic to connection tracker
ovs-ofctl add-flow br-int \
  "table=0,priority=100,ip,actions=ct(table=1,zone=1)"

# Table 1: Allow established/related connections
ovs-ofctl add-flow br-int \
  "table=1,priority=100,ct_state=+est+trk,actions=NORMAL"
ovs-ofctl add-flow br-int \
  "table=1,priority=100,ct_state=+rel+trk,actions=NORMAL"

# Table 1: Allow new SSH connections to VM (port 22)
ovs-ofctl add-flow br-int \
  "table=1,priority=90,ct_state=+new+trk,tcp,nw_dst=192.168.1.10,tp_dst=22,actions=ct(commit,zone=1),NORMAL"

# Table 1: Allow new ICMP (ping)
ovs-ofctl add-flow br-int \
  "table=1,priority=90,ct_state=+new+trk,icmp,actions=ct(commit,zone=1),NORMAL"

# Table 1: Drop everything else
ovs-ofctl add-flow br-int \
  "table=1,priority=0,actions=drop"

6. Hardware Offload with SmartNICs & DPUs

Even with DPDK, OVS consumes host CPU cores for packet processing — cores that could otherwise drive GPU computation, run ML frameworks, or handle storage I/O. Hardware offload moves the OVS forwarding pipeline into the NIC/DPU silicon, achieving line-rate performance (100/200/400 Gbps) while freeing all host CPU cores for application workloads.

NVIDIA BlueField DPU & ASAP²

ASAP² (Accelerated Switching and Packet Processing) is NVIDIA's OVS hardware offload technology for ConnectX and BlueField SmartNICs/DPUs. It works by offloading OVS datapath flows to the NIC's embedded switch (eSwitch), which performs matching and actions entirely in hardware.

TC Flower Offload

The Linux Traffic Control (TC) subsystem provides the framework for hardware offload. When OVS installs a flow in the kernel datapath, the TC flower classifier translates it into a hardware-compatible rule and programs it into the NIC's eSwitch via the switchdev driver model. The flow lifecycle:

  1. ovs-vswitchd classifies a packet miss and installs an OVS kernel flow
  2. The kernel OVS module translates the flow to a TC flower rule
  3. The NIC driver (mlx5_core) receives the TC flower rule and programs it into the eSwitch hardware tables
  4. Subsequent packets matching this flow are forwarded entirely in hardware — they never reach the host CPU
  5. Flow statistics are read back from hardware counters for OVS accounting

Enabling Hardware Offload

# 1. Enable switchdev mode on the NIC (e.g., ConnectX-6 Dx / BlueField-3)
devlink dev eswitch set pci/0000:03:00.0 mode switchdev

# 2. Enable hardware offload in OVS
ovs-vsctl set Open_vSwitch . other_config:hw-offload=true

# 3. Enable TC offload on the representor ports
ethtool -K enp3s0f0 hw-tc-offload on
ethtool -K enp3s0f0_0 hw-tc-offload on  # VF representor

# 4. Restart OVS
systemctl restart openvswitch-switch

# 5. Create bridge and add representor ports
ovs-vsctl add-br br-offload
ovs-vsctl add-port br-offload enp3s0f0       # PF uplink representor
ovs-vsctl add-port br-offload enp3s0f0_0     # VF 0 representor (VM port)
ovs-vsctl add-port br-offload enp3s0f0_1     # VF 1 representor

# 6. Verify offloaded flows
ovs-appctl dpctl/dump-flows type=offloaded
tc -s filter show dev enp3s0f0 ingress
BlueField DPU Architecture In a BlueField-3 DPU deployment, the DPU's ARM cores run a full Linux OS with OVS. The host sees only SR-IOV Virtual Functions (VFs) — the entire switching, security, and encapsulation pipeline runs on the DPU hardware. This provides host isolation: even a compromised host OS cannot bypass network security policies enforced by the DPU. This is the foundation of NVIDIA's "infrastructure on a chip" model used in SuperPOD deployments.

7. Network Security in Virtualized Environments

Security Groups: Per-Port Stateful Firewall

Security groups are the primary network access control mechanism in virtualized datacenters. Each virtual port (VM NIC, container interface) has an associated set of ingress and egress rules specifying allowed protocols, ports, and source/destination CIDR ranges. All traffic not explicitly allowed is dropped (default-deny).

Connection Tracking (conntrack) in OVS

OVS implements stateful firewall functionality through integration with the kernel's Netfilter connection tracking subsystem. The ct() OpenFlow action sends packets through conntrack, which tracks connection state (NEW, ESTABLISHED, RELATED, INVALID). This enables rules like "allow outbound traffic and automatically permit return traffic" without explicitly programming return-path rules.

Conntrack Zones and Packet Processing

+

Conntrack zones provide per-port isolation of connection tracking state. Each virtual port is assigned a unique zone number, ensuring that connections from one VM cannot interfere with the conntrack state of another VM — even if they share the same 5-tuple (src IP, dst IP, protocol, src port, dst port). 

# Assign conntrack zone per port using OVS registers
# Port 1 (VM-A) → zone 1; Port 2 (VM-B) → zone 2

# Tag incoming traffic with zone based on ingress port
ovs-ofctl add-flow br-int \
  "table=0,priority=100,in_port=1,actions=load:1->NXM_NX_CT_ZONE[],resubmit(,1)"
ovs-ofctl add-flow br-int \
  "table=0,priority=100,in_port=2,actions=load:2->NXM_NX_CT_ZONE[],resubmit(,1)"

# Send to conntrack in the assigned zone
ovs-ofctl add-flow br-int \
  "table=1,priority=100,ip,actions=ct(table=2,zone=NXM_NX_CT_ZONE[])"

# Table 2: enforcement
ovs-ofctl add-flow br-int \
  "table=2,priority=100,ct_state=+est+trk,actions=NORMAL"
ovs-ofctl add-flow br-int \
  "table=2,priority=50,ct_state=+new+trk,tcp,tp_dst=80,actions=ct(commit),NORMAL"
ovs-ofctl add-flow br-int \
  "table=2,priority=0,actions=drop"

Microsegmentation

Microsegmentation implements fine-grained, per-workload security policies that restrict lateral (east-west) traffic within the datacenter. Unlike traditional perimeter firewalls that only inspect north-south traffic, microsegmentation policies are enforced at the virtual switch level on every host. In a GPU cluster, this prevents a compromised management VM from accessing the high-speed storage network or GPU interconnect control plane.

  • Identity-based policies: Rules based on workload labels/tags, not just IP addresses (e.g., "allow training-job pods to reach storage-server group on NFS port 2049")
  • Zero-trust model: All east-west traffic is denied by default; each flow must be explicitly permitted
  • Distributed enforcement: Policies are programmed as OVS flows on every hypervisor, ensuring enforcement even for intra-host VM-to-VM traffic
  • Policy engines: NSX-T, Calico, Cilium, OVN ACLs, or custom OpenFlow pipelines

Network Policies (Kubernetes / Container Context)

In Kubernetes-based GPU clusters, NetworkPolicy resources define allowed ingress and egress traffic for pods using label selectors. The CNI plugin (Calico, Cilium, OVN-Kubernetes) translates these policies into datapath rules:

Security in GPU Superpod Deployments In NVIDIA DGX SuperPOD configurations, the network is segmented into distinct security domains: (1) In-band management network for BMC/IPMI, (2) Out-of-band provisioning network, (3) Storage network (NVMe-oF over RoCE), and (4) High-speed compute fabric (InfiniBand for NCCL/GPU-Direct RDMA). Each domain uses separate VLANs or VNIs with strict ACLs preventing cross-domain traffic. The BlueField DPU enforces these boundaries in hardware, ensuring that even a compromised host cannot breach network segmentation.

Knowledge Check: Virtual Networking

Question 1: What is the maximum number of unique VXLAN segments (VNIs) supported, and why is this a significant improvement over 802.1Q VLANs?

Correct answer: B. VXLAN uses a 24-bit VNI field in its 8-byte header, providing 224 = 16,777,216 unique segments. 802.1Q VLANs use a 12-bit VLAN ID (with 0 and 4095 reserved), yielding only 4,094 usable segments. This ~4,000x increase is essential for multi-tenant cloud datacenters where each tenant may need dozens of isolated network segments.

Question 2: In an OVS-DPDK deployment, what is the role of PMD (Poll Mode Driver) threads, and why do they consume 100% CPU even when idle?

Correct answer: C. DPDK Poll Mode Driver threads operate in a busy-wait loop, continuously polling NIC RX descriptor rings for new packets. Unlike the traditional interrupt-driven model (where the NIC triggers an IRQ, the kernel wakes the softirq handler, allocates an sk_buff, and passes it through the network stack), PMD threads read packets directly from memory-mapped DMA buffers with zero kernel involvement. This eliminates all interrupt, context-switch, and memory allocation overhead. The trade-off is that PMD cores show 100% utilization at all times, regardless of traffic load, because the polling loop never sleeps.

Question 3: How does ASAP² hardware offload on a BlueField DPU differ from OVS-DPDK, and what is the primary advantage for GPU compute nodes?

Correct answer: B. ASAP² (Accelerated Switching and Packet Processing) offloads OVS datapath flows from the host entirely into the ConnectX/BlueField NIC's embedded switch hardware (eSwitch). When OVS installs a kernel datapath flow, the TC flower classifier translates it into a hardware rule and programs it into the NIC's match-action tables (TCAM/SRAM). Subsequent packets matching that flow are forwarded at line rate (100/200/400 Gbps) in hardware, never touching the host CPU. Unlike OVS-DPDK, which dedicates 2–4 host CPU cores to PMD polling, ASAP² consumes zero host CPU cores. In GPU compute nodes where every CPU core and PCIe lane is valuable for feeding GPU workloads (data loading, preprocessing, NCCL coordination), freeing network processing from the host is a critical optimization.
