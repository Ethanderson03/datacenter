GPU Monitoring with NVIDIA DCGM

NVIDIA Data Center GPU Manager (DCGM) is the foundational monitoring framework for GPU clusters. It provides a unified API and CLI for health monitoring, diagnostics, policy management, and telemetry collection across hundreds or thousands of GPUs. DCGM runs as a persistent host-level daemon (nv-hostengine) and exposes metrics via dcgmi CLI, C/Python APIs, and the DCGM Exporter for Prometheus integration.

Core DCGM Architecture

The DCGM stack consists of the nv-hostengine daemon (runs on each node with GPUs), the dcgmi command-line interface, and optional DCGM Exporter (a lightweight Go binary that queries nv-hostengine and serves Prometheus-formatted metrics on port :9400/metrics). In Kubernetes, DCGM Exporter typically runs as a DaemonSet on every GPU node.

Essential dcgmi Commands

Command Purpose Typical Use
dcgmi discovery -l List all GPUs and their group IDs Initial inventory and topology verification
dcgmi diag -r 1 Quick diagnostic (deployment-level, ~30 seconds) Pre-job health check, PCIe bandwidth verification
dcgmi diag -r 2 Medium diagnostic (~2 minutes) Targeted stress test, memory checks
dcgmi diag -r 3 Long diagnostic (~15 minutes) Full stress test: memory, compute, PCIe, NVLink, power
dcgmi health -c -g 1 Check health of GPU group 1 Continuous health monitoring via scripts/cron
dcgmi stats -e -g 1 Enable stats recording for group Job-level GPU utilization profiling
dcgmi stats -v -g 1 View recorded stats Post-job analysis of GPU utilization
dcgmi policy -s -g 1 Set policy watches (ECC, thermal, power) Automated alerts when thresholds exceeded
dcgmi topo -g 1 Display GPU topology (NVLink, PCIe affinity) Validating NVLink mesh connectivity
dcgmi nvlink -s -g 1 Show NVLink status and error counts Diagnosing NVLink degradation

DCGM Diagnostics Levels

Level 1 (Quick / Deployment): ~30 seconds. Validates GPU presence, PCIe bandwidth, basic NVLink connectivity. Run this before every job or as a node health gate in Slurm prolog scripts. Catches obvious hardware failures (dead GPU, broken PCIe link).
Level 2 (Medium / Validation): ~2 minutes. Adds targeted stress tests for GPU compute and memory subsystems. Suitable for post-maintenance validation or when a node returns from repair. Detects marginal memory errors and reduced compute throughput.
Level 3 (Long / Extended): ~15 minutes. Full burn-in: extended stress on every GPU subsystem including memory, compute SMs, NVLink bandwidth, PCIe bandwidth, and power delivery. Run during commissioning of new hardware, after GPU/NVLink cable replacement, or quarterly as preventive maintenance. This is the gold standard for confirming a node is fully operational.

GPU XID Errors — Critical SRE Knowledge

XID errors are NVIDIA GPU fault codes logged by the kernel driver to dmesg and /var/log/syslog. Every SRE operating GPU infrastructure must be fluent in XID interpretation. These codes appear as kernel messages in the format: NVRM: Xid (PCI:0000:41:00): 48, pid=12345, .... The XID number is the key to diagnosis.

XID Name Severity Root Cause Action
13 GR (Graphics Engine) Fault Medium Usually a driver or CUDA runtime bug; sometimes bad application code accessing freed memory Update driver. If persistent across driver versions, check application. Rare hardware root cause.
31 GPU Memory Page Retirement High Correctable ECC errors exceeded threshold on a memory page; driver retires the page proactively Monitor frequency. >5 retirements/week = schedule GPU replacement. Check nvidia-smi -q -d RETIRED_PAGES.
43 GPU Stopped Processing Critical GPU context watchdog detected hang. Often thermal throttling, power delivery issue, or firmware bug. Check thermals, power. Reset GPU with nvidia-smi -r. If recurring, drain and RMA.
48 Double-Bit ECC Error Critical Uncorrectable memory error. Data corruption has occurred. Cannot be repaired in the field. Drain node immediately. Kill affected jobs. RMA the GPU. Do NOT attempt to continue operations.
63 Row Remapping Failure Critical GPU attempted to remap a faulty memory row but ran out of spare rows (Ampere+/Hopper). GPU memory is permanently degraded. Drain node and RMA. Indicates extensive memory wear.
64 Fallen off the Bus Critical GPU is no longer visible on the PCIe bus. Often a power or physical seating issue. Power cycle node. If recurs, reseat GPU physically. Check PCIe riser card and power cables. Possible motherboard or GPU hardware failure.
74 NVLink Error High NVLink CRC or protocol error. Cable degradation, connector issue, or NVLink circuitry fault. Check nvidia-smi nvlink -s. Reseat NVLink cables. If errors persist, replace cable then GPU.
79 GPU Access to Invalid Address Medium Application or driver attempted to access an invalid GPU virtual address. Usually an application bug. Enable CUDA_LAUNCH_BLOCKING=1 to get precise stack trace. Update CUDA toolkit and driver.
94 Contained ECC Error High ECC error was contained to the faulting application (Hopper+ feature). Process killed but GPU is OK. Job will fail; can be restarted. Monitor for frequency. If repeated on same GPU, treat as XID 48 path.
95 Uncontained ECC Error Critical ECC error could not be contained. Potential cross-process data corruption. Drain node. Reset GPU. If recurring, RMA. Similar severity to XID 48.

Checking for XID Errors

# Check kernel messages for XID errors
dmesg | grep -i "xid"

# Check specific GPU by PCI bus address
dmesg | grep "NVRM: Xid (PCI:0000:41:00)"

# Query retired pages (indicates prior XID 31 events)
nvidia-smi -q -d RETIRED_PAGES

# Check ECC error counts
nvidia-smi -q -d ECC

# Query remapped rows (Ampere+/Hopper)
nvidia-smi -q -d REMAPPED_ROWS

# View XID errors in system journal (systemd)
journalctl -k --since "1 hour ago" | grep -i xid

# DCGM health check that catches XID-worthy conditions
dcgmi health -c -g 1

# Continuous monitoring: watch for new XID in real-time
dmesg -wH | grep --line-buffered "Xid"

XID Error Troubleshooting Flowchart

SRE Rule: Any XID 48 (double-bit ECC) event is a drain-immediately situation. The GPU has experienced uncorrectable memory corruption. All running jobs on that GPU are suspect and their outputs should be treated as potentially incorrect. The node must be drained from the scheduler, the GPU replaced, and a Level 3 DCGM diagnostic run before returning to service.

Prometheus + Grafana Observability Stack

The standard observability stack for GPU clusters is Prometheus for metrics collection and storage, Grafana for visualization and dashboards, and AlertManager for alert routing. This stack is deployed either directly on the cluster or as a dedicated monitoring infrastructure, and it forms the nerve center for all operational visibility.

# /etc/prometheus/rules/gpu_alerts.yml
groups:
  - name: gpu-health
    interval: 30s
    rules:
      # Temperature alerts
      - alert: GPUTemperatureWarning
        expr: DCGM_FI_DEV_GPU_TEMP > 80
        for: 5m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "GPU {{ $labels.gpu }} on {{ $labels.instance }} at {{ $value }}°C"
          description: "GPU temperature above 80°C for 5 minutes. Thermal throttling imminent at 83°C."
          runbook_url: "https://wiki.internal/runbooks/gpu-thermal"

      - alert: GPUTemperatureCritical
        expr: DCGM_FI_DEV_GPU_TEMP > 85
        for: 2m
        labels:
          severity: critical
          team: gpu-infra
        annotations:
          summary: "GPU {{ $labels.gpu }} on {{ $labels.instance }} OVERHEATING at {{ $value }}°C"
          description: "GPU temperature above 85°C. Active thermal throttling. Check cooling system."

      # ECC error alerts
      - alert: GPUDoublebitECC
        expr: DCGM_FI_DEV_ECC_DBE_VOL_TOTAL > 0
        for: 0m
        labels:
          severity: critical
          team: gpu-infra
          page: "true"
        annotations:
          summary: "FATAL: Double-bit ECC error on GPU {{ $labels.gpu }} on {{ $labels.instance }}"
          description: "Uncorrectable memory error detected. Drain node immediately and RMA GPU."

      - alert: GPUSingleBitECCHigh
        expr: rate(DCGM_FI_DEV_ECC_SBE_VOL_TOTAL[1h]) > 100
        for: 10m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "High single-bit ECC rate on GPU {{ $labels.gpu }} on {{ $labels.instance }}"
          description: "{{ $value }} correctable ECC errors/hour. Memory degrading. Plan replacement."

      # XID error alert
      - alert: GPUXidError
        expr: DCGM_FI_DEV_XID_ERRORS != 0
        for: 0m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "XID {{ $value }} on GPU {{ $labels.gpu }} on {{ $labels.instance }}"
          description: "GPU reported XID error. Check dmesg on the node for details."

      # GPU utilization anomalies
      - alert: GPUIdleDuringJob
        expr: |
          DCGM_FI_DEV_GPU_UTIL < 5
          and on(instance) slurm_node_state{state="allocated"} == 1
        for: 15m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "GPU {{ $labels.gpu }} idle on allocated node {{ $labels.instance }}"
          description: "GPU shows <5% utilization but node is allocated. Job may be stuck or misconfigured."

      # Power alerts
      - alert: GPUPowerExceedsTDP
        expr: DCGM_FI_DEV_POWER_USAGE > 700
        for: 5m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "GPU {{ $labels.gpu }} power {{ $value }}W exceeds TDP on {{ $labels.instance }}"

      # NVLink errors
      - alert: NVLinkCRCErrors
        expr: rate(DCGM_FI_DEV_NVLINK_CRC_FLIT_ERROR_COUNT_TOTAL[5m]) > 0
        for: 2m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "NVLink CRC errors on GPU {{ $labels.gpu }} on {{ $labels.instance }}"
          description: "NVLink experiencing CRC errors. Check cables and connectors."

      # PCIe replay
      - alert: PCIeReplayHigh
        expr: rate(DCGM_FI_DEV_PCIE_REPLAY_COUNTER[1h]) > 100
        for: 10m
        labels:
          severity: warning
          team: gpu-infra
        annotations:
          summary: "High PCIe replay count on GPU {{ $labels.gpu }} on {{ $labels.instance }}"
          description: "PCIe link quality degraded. Check physical seating and riser card."

      # Node-level alerts
      - alert: GPUNodeDown
        expr: up{job="dcgm-exporter"} == 0
        for: 3m
        labels:
          severity: critical
          team: gpu-infra
        annotations:
          summary: "DCGM Exporter down on {{ $labels.instance }}"
          description: "Cannot scrape GPU metrics. Node may be down or DCGM daemon failed."

Useful PromQL Queries for GPU Operations

# Average GPU utilization across the entire cluster
avg(DCGM_FI_DEV_GPU_UTIL)

# GPU utilization heatmap: per-node, per-GPU
DCGM_FI_DEV_GPU_UTIL

# Top 10 hottest GPUs in the cluster
topk(10, DCGM_FI_DEV_GPU_TEMP)

# Total power consumption across all GPUs (watts)
sum(DCGM_FI_DEV_POWER_USAGE)

# GPUs with any ECC errors in the last hour
sum by (instance, gpu) (increase(DCGM_FI_DEV_ECC_SBE_VOL_TOTAL[1h])) > 0

# GPU memory utilization percentage
(DCGM_FI_DEV_FB_USED / (DCGM_FI_DEV_FB_USED + DCGM_FI_DEV_FB_FREE)) * 100

# Nodes with GPUs above 80°C
count by (instance) (DCGM_FI_DEV_GPU_TEMP > 80)

# Average GPU utilization by rack (requires rack label)
avg by (rack) (DCGM_FI_DEV_GPU_UTIL)

# NVLink bandwidth utilization rate of change
rate(DCGM_FI_DEV_NVLINK_BANDWIDTH_TOTAL[5m])

# Cluster-wide GPU availability (fraction of healthy GPUs)
count(DCGM_FI_DEV_GPU_TEMP > 0) / count(DCGM_FI_DEV_GPU_TEMP >= 0)

BMC / IPMI / Redfish — Out-of-Band Management

Every server in a GPU cluster has a Baseboard Management Controller (BMC): an independent embedded processor with its own dedicated network interface that provides out-of-band management. The BMC operates independently of the host OS, meaning you can power cycle a completely frozen node, access its serial console, and read hardware sensors even when the operating system has crashed. This is an indispensable tool for SRE operations.

BMC Architecture

The BMC is a small ARM-based SoC (commonly AST2600 from ASPEED) soldered onto the server motherboard. It has its own RAM (typically 512 MB - 1 GB), flash storage for firmware, and a dedicated Ethernet port (the "IPMI port" or "management port") on a physically separate network (the out-of-band management network, often a /24 VLAN like 10.10.100.0/24).

IPMI Commands (ipmitool)

IPMI (Intelligent Platform Management Interface) is the legacy protocol for BMC communication. While being replaced by Redfish, ipmitool remains widely used. 

# ===== Power Management =====
# Check power status
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password power status
# Output: Chassis Power is on

# Power cycle (hard reboot - last resort)
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password power cycle

# Graceful shutdown
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password power soft

# Force power off (equivalent to holding power button)
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password power off

# Power on
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password power on

# ===== Serial Over LAN (Console Access) =====
# Access serial console of a frozen/unresponsive node
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password sol activate
# Use ~. to disconnect from SOL session

# ===== Sensor Readings =====
# List all sensor data records
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password sdr list
# Example output:
# CPU0 Temp        | 62 degrees C      | ok
# CPU1 Temp        | 58 degrees C      | ok
# Inlet Temp       | 24 degrees C      | ok
# System Fan 1     | 8400 RPM          | ok
# PSU1 Input       | 1648 Watts        | ok
# PSU2 Input       | 1632 Watts        | ok

# Get a specific sensor reading
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password sdr get "Inlet Temp"

# ===== System Event Log =====
# List hardware events (PSU failures, temp events, boot errors)
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password sel list
# Example output:
#    1 | 02/15/2026 | 14:23:05 | Temperature #0x30 | Upper Critical going high
#    2 | 02/15/2026 | 14:23:06 | Fan #0x45 | Lower Critical going low

# Clear event log (after review)
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password sel clear

# ===== Chassis Status =====
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password chassis status

# ===== BMC Info =====
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password bmc info

# ===== Boot Device Override =====
# Force next boot from PXE (for OS re-imaging)
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password chassis bootdev pxe

# Force next boot from BIOS setup
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password chassis bootdev bios

# ===== Fan Control (vendor-specific, use with caution) =====
# Read fan mode
ipmitool -I lanplus -H bmc-gpu-001.mgmt -U admin -P password raw 0x30 0x45 0x00

Redfish — Modern REST API for BMC

Redfish (DMTF standard) is the modern replacement for IPMI. It provides a RESTful JSON-over-HTTPS API, proper authentication, a rich data model, and is significantly more secure and extensible than IPMI. NVIDIA DGX systems use Redfish extensively for management, including GPU-specific telemetry exposed through OEM extensions. 

# ===== Redfish API Examples (using curl) =====

# Get system information
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Systems/1 | jq '.'
# Returns: Model, SerialNumber, PowerState, BiosVersion, ProcessorSummary, MemorySummary

# Get BMC (Manager) information
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Managers/1 | jq '.'
# Returns: FirmwareVersion, DateTime, NetworkProtocol

# Get thermal data (temperatures and fans)
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Chassis/1/Thermal | jq '.'
# Returns: Temperatures[] (ReadingCelsius, UpperThresholdCritical), Fans[] (Reading, Status)

# Get power data (PSU status, power consumption)
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Chassis/1/Power | jq '.'
# Returns: PowerControl[] (PowerConsumedWatts), PowerSupplies[] (Status, LineInputVoltage)

# Perform a graceful restart via Redfish
curl -sk -u admin:password \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"ResetType": "GracefulRestart"}' \
  https://bmc-gpu-001.mgmt/redfish/v1/Systems/1/Actions/ComputerSystem.Reset

# Force power off
curl -sk -u admin:password \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"ResetType": "ForceOff"}' \
  https://bmc-gpu-001.mgmt/redfish/v1/Systems/1/Actions/ComputerSystem.Reset

# Get system event log entries
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Systems/1/LogServices/Log1/Entries | jq '.'

# Get storage controller info
curl -sk -u admin:password \
  https://bmc-gpu-001.mgmt/redfish/v1/Systems/1/Storage | jq '.'

# DGX-specific: Get GPU info via Redfish OEM extensions
curl -sk -u admin:password \
  https://bmc-dgx-001.mgmt/redfish/v1/Systems/1/Oem/Nvidia/GPUs | jq '.'
Management Network Security: BMC/IPMI/Redfish interfaces must be isolated on a dedicated management VLAN with strict firewall rules. IPMI has known security vulnerabilities (cipher zero, weak default credentials). Always change default passwords, disable IPMI if Redfish is available, use HTTPS for Redfish, and restrict management network access to authorized jump hosts only.

Network Monitoring

GPU clusters rely on high-performance networks (InfiniBand and/or high-speed Ethernet) for inter-GPU communication (NCCL all-reduce, all-gather) and storage access. Network issues directly translate into training slowdowns, job failures, and wasted GPU-hours. Comprehensive fabric monitoring is therefore a top-tier SRE priority.

InfiniBand Monitoring

# ===== InfiniBand CLI Diagnostics =====

# Show local HCA (Host Channel Adapter) port status

NVIDIA UFM (Unified Fabric Manager)

UFM is NVIDIA's commercial InfiniBand fabric management platform. It provides a GUI and REST API for monitoring and managing the entire IB fabric: topology discovery, health monitoring, event alerting, adaptive routing configuration, SHARP (Scalable Hierarchical Aggregation and Reduction Protocol) management, and telemetry collection. UFM is deployed as a server application (or VM) connected to the IB fabric's management plane.

UFM Feature Description
Topology View Visual fabric map showing all switches, links, and host connections with real-time status
Telemetry Per-port bandwidth, packet rates, error counters at 1-second granularity
Events & Alerts Link down/up events, error threshold crossings, SM failovers
Adaptive Routing Dynamic path selection to avoid congested or failed links
Cable Diagnostics Identify degraded cables via BER (Bit Error Rate) monitoring
REST API /ufmRest/app/ endpoints for integration with Prometheus, custom tooling
SHARP Management Configure and monitor in-network compute for collective operations

Network Error Investigation Priority: In GPU clusters, even small increases in network error rates can cause massive training performance degradation. A single flapping InfiniBand link can slow down a 256-GPU NCCL all-reduce by 10-100x because the collective operation runs at the speed of the slowest participant. When investigating job performance issues, always check perfquery error counters and UFM events as one of the first steps.

Incident Response

Runbook: GPU Node Down (No Metrics)

# RUNBOOK: GPU Node Not Responding
# Severity: High | On-call: GPU Infra Team
# Trigger: GPUNodeDown alert (DCGM Exporter unreachable for >3 min)

# Step 1: Verify node is actually unreachable
ping -c 3 gpu-node-042
ssh gpu-node-042 hostname 2>/dev/null

# Step 2: If unreachable via in-band network, use BMC/IPMI
ipmitool -I lanplus -H bmc-gpu-042.mgmt -U admin -P $BMC_PASS power status

# Step 3: Check BMC system event log for hardware events
ipmitool -I lanplus -H bmc-gpu-042.mgmt -U admin -P $BMC_PASS sel list | tail -20

# Step 4: Try serial console to see kernel panic or hang
ipmitool -I lanplus -H bmc-gpu-042.mgmt -U admin -P $BMC_PASS sol activate
# Look for: kernel panic, OOM killer, GPU XID errors, NVLink errors
# Use ~. to disconnect

# Step 5: If OS is hung, attempt graceful reboot first
ipmitool -I lanplus -H bmc-gpu-042.mgmt -U admin -P $BMC_PASS power soft
sleep 120  # wait 2 minutes for graceful shutdown

# Step 6: If still unresponsive, hard power cycle
ipmitool -I lanplus -H bmc-gpu-042.mgmt -U admin -P $BMC_PASS power cycle

# Step 7: After node boots, verify GPU health
ssh gpu-node-042 nvidia-smi
ssh gpu-node-042 dcgmi diag -r 2

# Step 8: If node passes diagnostics, return to scheduler
scontrol update NodeName=gpu-node-042 State=RESUME Reason="Recovered from outage"

# Step 9: If diagnostics fail, drain and escalate
scontrol update NodeName=gpu-node-042 State=DRAIN Reason="Hardware failure - ticket HW-12345"

Runbook: High ECC Error Rate

# RUNBOOK: High ECC Error Rate
# Severity: Warning -> Critical | On-call: GPU Infra Team
# Trigger: GPUSingleBitECCHigh or GPUDoublebitECC alert

# Step 1: Identify the affected GPU
# From the alert: instance=gpu-node-017, gpu=3

# Step 2: Check current ECC error counts
ssh gpu-node-017 nvidia-smi -i 3 -q -d ECC
# Look for:
#   Volatile ECC Errors -> Single Bit -> count
#   Volatile ECC Errors -> Double Bit -> count (ANY = critical)
#   Aggregate ECC Errors -> counts (lifetime, persists across reboots)

# Step 3: Check retired pages
ssh gpu-node-017 nvidia-smi -i 3 -q -d RETIRED_PAGES
# Retired pages: Single Bit ECC = correctable retirements
# Pending page retirement = reboot needed to complete retirement

# Step 4: Check row remapping (Ampere+/Hopper)
ssh gpu-node-017 nvidia-smi -i 3 -q -d REMAPPED_ROWS
# Remapping Failure = critical (no more spare rows)

# Step 5: Decision
# IF double-bit errors (XID 48) OR row remapping failure (XID 63):
#   -> Drain node immediately, kill jobs, open RMA ticket
scontrol update NodeName=gpu-node-017 State=DRAIN \
  Reason="DBE/Row remap failure GPU 3 - RMA ticket HW-12346"

# IF single-bit only, moderate rate (<50/hour):
#   -> Monitor. Schedule replacement during next maintenance window.
#   -> Reset volatile counters after noting them
ssh gpu-node-017 nvidia-smi -i 3 -rac  # reset aggregate counters (CAUTION: lose history)

# IF single-bit, high rate (>100/hour) or accelerating:
#   -> Drain node within 24 hours, schedule GPU replacement

Escalation Path & Post-Incident Review

Escalation Tiers:

Tier Response Time Role Actions
L1 15 minutes On-call SRE Acknowledge alert, triage, follow runbook, drain node if needed
L2 1 hour Senior SRE / Specialist Complex diagnosis (NVLink topology issues, multi-node failures, firmware bugs)
L3 4 hours NVIDIA Support (Enterprise) GPU/NVSwitch firmware issues, hardware design defects, RMA coordination
Incident Commander 30 minutes (major) Engineering Manager Coordinate multi-team response, stakeholder communication

Post-Incident Review (PIR) Template:

  • Summary: What happened, duration, impact (GPU-hours lost, jobs affected)
  • Timeline: Detection time, response time, mitigation time, resolution time
  • Root Cause: 5-whys analysis. Hardware failure? Software bug? Config error?
  • Impact: Number of GPUs affected, jobs killed, GPU-hours wasted, SLO impact
  • What Went Well: Automation that worked, fast detection, effective runbooks
  • What Can Be Improved: Missing monitoring, gaps in runbooks, slow escalation
  • Action Items: Specific, assigned, time-bound improvements

Reviews are blameless. The goal is systemic improvement, not individual fault-finding. Focus on the system: How did the system allow this failure to have this impact? How can the system be made more resilient?

# Example: Rolling NVIDIA driver update workflow
# Step 1: Drain node from scheduler
scontrol update NodeName=gpu-node-001 State=DRAIN Reason="Driver update"

# Step 2: Wait for running jobs to complete

# Step 3: Stop GPU-dependent services

scontrol update NodeName=gpu-node-001 State=RESUME Reason="Driver update complete"

Troubleshooting Playbook

These are the most common failure scenarios an SRE will encounter in GPU cluster operations, along with systematic diagnostic procedures.

GPU Not Visible (nvidia-smi shows fewer GPUs)

# SCENARIO: Node should have 8 GPUs but nvidia-smi shows 7 (or 0)

# Step 1: Check how many GPUs the kernel sees on PCIe bus
lspci | grep -i nvidia
# Count the entries. If fewer than expected, GPU is not on the PCIe bus.

# Step 2: Check if it's a driver issue vs hardware issue
lspci -vvv -s [missing_gpu_pci_slot] 2>/dev/null
# If lspci sees the device but nvidia-smi doesn't, it's a driver issue
# If lspci doesn't see it, it's a hardware/PCIe issue

# Step 3: Check IOMMU settings (can hide devices)
dmesg | grep -i iommu
cat /proc/cmdline | grep iommu
# For GPU passthrough: intel_iommu=on iommu=pt
# For bare-metal GPU: intel_iommu=off (or iommu=pt)

# Step 4: Check for XID 64 (GPU fell off bus)
dmesg | grep -i "Xid.*64\|fell off"

# Step 5: Check BMC system event log for PCIe errors
ipmitool -I lanplus -H bmc-node.mgmt -U admin sel list | grep -i pci

# Step 6: Hardware actions (requires physical access or remote hands)
# - Power off node completely (not just reboot - full power cycle)
# - Reseat the GPU in its PCIe slot
# - Check PCIe riser card connection
# - Check GPU power cables (8-pin/12-pin connectors)
# - Power on and verify

# Step 7: If GPU returns, run diagnostics
nvidia-smi
dcgmi diag -r 3  # full diagnostic

# Step 8: If GPU still missing after reseat + power cycle = RMA

Training Job Running Slow

# SCENARIO: Distributed training throughput (samples/sec) below expected

# Step 1: Check for thermal throttling
nvidia-smi -q -d PERFORMANCE
# Look for: "SW Thermal Slowdown: Active" or "HW Thermal Slowdown: Active"
nvidia-smi --query-gpu=temperature.gpu,clocks_throttle_reasons.active \
  --format=csv -l 5

# Step 2: Check GPU clock speeds (should be at max for training)
nvidia-smi -q -d CLOCK
# Compare "Graphics" and "Memory" clocks vs max clocks
# If clocks are low: check power limit, thermal, or driver issue

# Step 3: Check NVLink health and bandwidth
nvidia-smi nvlink -s    # NVLink status
nvidia-smi nvlink -e    # NVLink error counters
# Any non-zero CRC or replay errors = degraded NVLink

# Step 4: Check NCCL topology and algorithm selection
# Set NCCL debug logging:
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=ALL
# Re-run training and check logs for:
# - "Using network" -> should show IB or NVLink, not Socket
# - "Tree" or "Ring" algorithm selection
# - Any warnings about topology or speed mismatches

# Step 5: Check InfiniBand for errors
ibstat          # verify port is Active, rate matches expected (e.g., 400 Gb/s)
perfquery       # check error counters
ibdiagnet -ls 30  # check port counters over 30 seconds

# Step 6: Check for PCIe bandwidth bottleneck
nvidia-smi -q -d PCIE
# Verify: Link Gen = 5 (PCIe 5.0), Link Width = x16
# If degraded (e.g., Gen3 x8), reseat GPU or check riser

# Step 7: Check CPU/memory bottleneck (data loading)
htop  # CPU utilization
iostat -xz 1 5  # disk I/O (is data loading starving GPUs?)
# If CPU at 100% with GPUs idle: increase data loader workers

# Step 8: Check storage throughput
dd if=/dev/zero of=/lustre/scratch/test.tmp bs=1M count=4096 oflag=direct
# Should match expected Lustre throughput

# Step 9: Profile the training job
# Use NVIDIA Nsight Systems:
nsys profile -o profile_output python train.py --epochs 1
# Use PyTorch Profiler:
# torch.profiler.profile() in code

Network Partition (Nodes Cannot Communicate)

# SCENARIO: Some nodes cannot reach other nodes over InfiniBand

# Step 1: Check local IB port status
ibstat
# State should be: Active
# Physical state should be: LinkUp
# Rate should match expected (e.g., 400 for NDR)

# Step 2: Check subnet manager status
sminfo
# Verify SM is in MASTER state
# If no SM: check if OpenSM or UFM is running on the SM node

# Step 3: Check if we can see the destination
ibping -G [destination_port_guid]
# If timeout: path is broken

# Step 4: Trace the path to see where it breaks
ibtracert [source_LID] [dest_LID]
# Identify which hop fails

# Step 5: Check the switch port at the failing hop
smpquery portinfo [switch_LID] [port]
# Look for: PhysState, LinkSpeedActive, errors

# Step 6: Check for cable issues on the suspect link
# In UFM: check BER (Bit Error Rate) for the link
# Via CLI:
perfquery [switch_LID] [port]
# Look for non-zero: SymbolErrorCounter, LinkErrorRecoveryCounter,
#                     LinkDownedCounter, PortRcvErrors

# Step 7: Check if issue is transient (flapping link)
# Watch for link up/down events:
journalctl -f -u opensm  # or check UFM events

# Step 8: Resolution
# Bad cable: Replace the cable, verify with ibdiagnet
# Bad switch port: Move cable to spare port, report HW issue
# SM issue: Restart OpenSM or UFM, verify failover works
# Configuration: Check routing tables, LID assignment, P_Key config

Storage Performance Degraded

# SCENARIO: Lustre filesystem showing slow I/O

# Step 1: Check Lustre filesystem status
lctl dl                    # list Lustre devices
lfs df -h                  # show filesystem capacity and usage per OST
lfs check servers          # verify connectivity to all MDTs and OSTs

# Step 2: Check for OST imbalance
lfs df -h | sort -k5 -rn  # sort by usage percentage
# If some OSTs >95% while others are <50%, data is imbalanced
# Fix: lfs migrate to rebalance, or lfs setstripe for new files

# Step 3: Check network path to storage
# From a GPU node:
ping -c 5 lustre-oss-001            # basic connectivity
ib_write_bw lustre-oss-001          # IB bandwidth to storage server
# Compare against expected (e.g., 200 Gb/s per link)

# Step 4: Check for degraded RAID on OSS (storage servers)
ssh lustre-oss-001 "mdadm --detail /dev/md0"    # if using mdraid
ssh lustre-oss-001 "zpool status"                 # if using ZFS
# Degraded RAID = reduced throughput + rebuild I/O competing

# Step 5: Check for heavy metadata operations (small file storm)
lctl get_param mdt.*.stats | grep -i open
# High open/close rates = metadata bottleneck
# Solution: increase MDT capacity, use DNE (Distributed Namespace)

# Step 6: Check client-side Lustre tuning
lctl get_param osc.*.max_rpcs_in_flight   # should be 16-64
lctl get_param osc.*.max_pages_per_rpc     # should be 4096 (16MB)
lctl get_param llite.*.max_read_ahead_mb   # should be 256+

# Step 7: Check for network congestion between compute and storage
# On IB fabric: check ISL (Inter-Switch Link) utilization in UFM
# Congested ISLs = need more bandwidth or better routing

# Step 8: Temporary mitigation
# If one OST is the bottleneck, mark it unavailable for new writes:
# lctl set_param osc.*.active=0  (CAUTION: makes existing data on that OST unreachable)
# Better: Contact storage team for targeted investigation
Troubleshooting Principle: Always start with the most common and easily checked causes before moving to exotic explanations. In GPU clusters, the most frequent root causes of performance issues are, in order: (1) thermal throttling, (2) degraded NVLink/PCIe links, (3) network cable issues, (4) application misconfiguration (wrong NCCL settings, wrong topology), (5) storage bottleneck. Hardware failure (dead GPU, bad memory) is less common but more severe.

Knowledge Check: Monitoring & SRE

Question 1: A monitoring alert fires for XID 48 on GPU 3 of node gpu-node-017. What is the correct immediate response?

A) Reboot the node and re-run the failed job. XID 48 is a transient driver error that clears on restart.
B) Update the NVIDIA driver to the latest version. XID 48 indicates a driver compatibility issue.
C) Drain the node from the scheduler immediately, kill jobs on the affected GPU, and open an RMA ticket for GPU replacement. XID 48 is a double-bit (uncorrectable) ECC error.
D) Run dcgmi diag -r 3 to assess the severity. XID 48 may or may not require hardware replacement depending on the diagnostic result.

Question 2: Which PromQL expression correctly alerts when a GPU is experiencing a sustained high rate of correctable (single-bit) ECC errors?

A) DCGM_FI_DEV_ECC_SBE_VOL_TOTAL > 100 — alert when total count exceeds 100.
B) rate(DCGM_FI_DEV_ECC_SBE_VOL_TOTAL[1h]) > 100 with for: 10m — alert when error rate exceeds 100/hour for 10 minutes.
C) delta(DCGM_FI_DEV_ECC_SBE_VOL_TOTAL[5m]) == 0 — alert when errors stop accumulating (GPU is failing silently).
D) DCGM_FI_DEV_ECC_DBE_VOL_TOTAL > 0 — this catches all ECC errors including single-bit ones.

Question 3: A GPU node is completely unresponsive — it does not reply to ping, SSH connections time out, and there is no in-band network connectivity. Which management path allows you to diagnose the issue and potentially recover the node without physical access?

A) Use kubectl exec to access a debugging container on the node and run diagnostics from within Kubernetes.
B) Use IPMI/Redfish via the out-of-band BMC management network to access the serial console (ipmitool sol activate), check the system event log, and power cycle the node.
C) SSH to an adjacent node in the same rack and use ipmitool locally over the data network to reach the unresponsive node's IPMI interface.
D) Wait for the hardware watchdog timer to automatically reboot the node. All GPU nodes have hardware watchdogs enabled by default that will recover within 5 minutes.
