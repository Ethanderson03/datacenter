Orchestration & Cluster Management

Kubernetes, GPU Operators, RDMA Networking, and Job Scheduling for GPU Superpods

1. Kubernetes for GPU Clusters

Why Kubernetes for HPC / GPU Infrastructure?

Kubernetes has become the dominant orchestration platform for GPU clusters because it solves three fundamental challenges at scale: scheduling (placing workloads on the right hardware), scaling (elastically growing from 1 to 1,000+ nodes), and multi-tenancy (isolating teams sharing expensive GPU resources). In a Superpod environment with 32+ DGX nodes each containing 8 GPUs, manual placement of distributed training jobs is impractical. Kubernetes automates this while respecting hardware topology constraints critical for NCCL collective communication performance.

Unlike traditional HPC schedulers (Slurm, PBS), Kubernetes provides a declarative API, a rich ecosystem of operators and controllers, container-native isolation, and seamless integration with CI/CD pipelines. Many organizations run Kubernetes alongside Slurm, using K8s for inference/serving and interactive workloads, and Slurm for batch training. However, with the maturation of gang scheduling (Volcano, Kueue) and training operators (Kubeflow), Kubernetes is increasingly used for large-scale training as well.

GPU-Aware Scheduling

At the core of GPU scheduling in Kubernetes is the Extended Resource mechanism. GPUs are advertised by the NVIDIA Device Plugin as nvidia.com/gpu resources on each node. Pods request GPUs in their resource spec, and the Kubernetes scheduler only places pods on nodes with sufficient available GPUs. Unlike CPU and memory, GPU resources are whole-number only and cannot be overcommitted — if a node has 8 GPUs and 8 are allocated, no more GPU pods can be scheduled there. 

# Pod requesting all 8 GPUs on a DGX H100 node
apiVersion: v1
kind: Pod
metadata:
  name: training-worker-0
  labels:
    app: llm-training
    role: worker
spec:
  containers:
  - name: pytorch-trainer
    image: nvcr.io/nvidia/pytorch:24.01-py3
    resources:
      requests:
        cpu: "64"
        memory: "512Gi"
        nvidia.com/gpu: 8          # Request all 8 H100 GPUs
      limits:
        cpu: "128"
        memory: "1024Gi"
        nvidia.com/gpu: 8          # Limits must equal requests for GPUs
    env:
    - name: NCCL_DEBUG
      value: "INFO"
    - name: NCCL_IB_HCA
      value: "mlx5"               # Use InfiniBand HCAs for NCCL
    - name: NCCL_TOPO_DUMP_FILE
      value: "/var/run/nvidia/topo.xml"
    volumeMounts:
    - name: shm
      mountPath: /dev/shm
  volumes:
  - name: shm
    emptyDir:
      medium: Memory
      sizeLimit: "128Gi"           # Large shared memory for NCCL
Topology-Aware Scheduling

For distributed training with NCCL, pod placement matters enormously. Two pods on the same NVSwitch-connected DGX node communicate at 900 GB/s (NVLink). Two pods on different nodes in the same leaf switch communicate at 400 Gb/s (InfiniBand). Two pods crossing spine switches see higher latency and potential congestion. The scheduler must be topology-aware to co-locate pods in the same rack or pod (network pod) whenever possible. This is achieved via topology spread constraints and pod affinity rules, combined with node labels that encode the physical topology (rack, row, network pod).

Node Labels and Taints for GPU Types

GPU Feature Discovery (GFD), a component of the NVIDIA GPU Operator, automatically labels nodes with detailed GPU information. These labels enable precise scheduling decisions:

Pod Scheduling Strategies

nodeSelector is the simplest constraint — a map of key-value label pairs that the target node must match. nodeAffinity provides richer expressions including In, NotIn, Exists, Gt, Lt operators, and supports both required (hard) and preferred (soft) rules. 

affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: nvidia.com/gpu.product
          operator: In
          values:
          - "NVIDIA-H100-SXM5-80GB"
          - "NVIDIA-H200-SXM-141GB"
        - key: nvidia.com/gpu.count
          operator: Gt
          values:
          - "4"                    # Nodes with more than 4 GPUs
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
        - key: topology.kubernetes.io/rack
          operator: In
          values:
          - "rack-01"              # Prefer rack-01 (soft)

Pod affinity co-locates pods relative to other pods. For distributed training, you want all workers of the same job on nodes in the same rack (same topologyKey). Pod anti-affinity spreads replicas across failure domains for high availability. 

affinity:
  podAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: training-job
          operator: In
          values:
          - "llm-gpt-run-042"
      topologyKey: topology.kubernetes.io/rack  # Same rack
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 50
      podAffinityTerm:
        labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - "inference-server"
        topologyKey: kubernetes.io/hostname  # Different nodes

Topology spread constraints distribute pods evenly across topology domains. This is useful for spreading inference replicas across racks for fault tolerance, or concentrating training workers within a network pod for NCCL performance. 

topologySpreadConstraints:
- maxSkew: 1                      # Max difference in pod count across domains
  topologyKey: topology.kubernetes.io/rack
  whenUnsatisfiable: DoNotSchedule  # Hard constraint
  labelSelector:
    matchLabels:
      app: inference-triton
- maxSkew: 2
  topologyKey: topology.kubernetes.io/zone
  whenUnsatisfiable: ScheduleAnyway  # Soft constraint (best-effort)
  labelSelector:
    matchLabels:
      app: inference-triton

2. NVIDIA GPU Operator

Automating the GPU Software Stack

The NVIDIA GPU Operator uses the Kubernetes operator framework to automate the management of all NVIDIA software components needed for GPU provisioning. Instead of manually installing drivers, container toolkits, and device plugins on every node, the GPU Operator deploys and manages these as containerized workloads via DaemonSets. When a new GPU node joins the cluster, the operator automatically installs the full stack — from kernel-mode driver to monitoring exporters — without any manual intervention. This is essential in cloud-native environments where nodes are ephemeral and auto-scaled.

Components Deployed by the GPU Operator

Component DaemonSet Name Function
NVIDIA Driver nvidia-driver-daemonset Container-based driver installation. Compiles and loads the kernel module (nvidia.ko, nvidia-uvm.ko, nvidia-modeset.ko) on the host. Supports pre-compiled drivers for specific kernel versions to avoid compilation overhead. Version is pinned in ClusterPolicy (e.g., 535.129.03).
NVIDIA Container Toolkit nvidia-container-toolkit-daemonset Formerly nvidia-docker2. Installs the nvidia-container-runtime and nvidia-container-cli on the host. Configures containerd to use the NVIDIA runtime as a handler for GPU containers. Handles injecting GPU device nodes (/dev/nvidia*) and driver libraries into containers.
NVIDIA Device Plugin nvidia-device-plugin-daemonset Implements the Kubernetes Device Plugin API. Discovers GPUs on the node and advertises them to the kubelet as nvidia.com/gpu extended resources. Handles GPU allocation, health checking, and device topology reporting. Supports time-slicing for GPU sharing and MIG device advertisement.
DCGM Exporter nvidia-dcgm-exporter Exports GPU metrics via the NVIDIA Data Center GPU Manager (DCGM) in Prometheus format. Metrics include GPU utilization, memory usage, temperature, power draw, ECC errors, NVLink throughput, PCIe throughput, and SM clock frequencies. Default scrape endpoint: :9400/metrics.
GPU Feature Discovery nvidia-gpu-feature-discovery Labels nodes with GPU properties (product name, driver version, CUDA version, compute capability, MIG capability, GPU family). These labels power nodeSelector and nodeAffinity scheduling decisions. Re-labels periodically to detect hardware changes.
MIG Manager nvidia-mig-manager Manages Multi-Instance GPU (MIG) partitioning on A100/H100 GPUs. Applies MIG profiles declaratively via ConfigMaps. Can reconfigure MIG partitions by draining pods, applying the new profile, and uncordoning the node. Supports per-node MIG configurations.
DCGM (standalone) nvidia-dcgm Runs the DCGM daemon for deep GPU health monitoring, diagnostics, and policy-based management. Provides GPU health watches, job statistics, and active health checks (memory stress tests, PCIe bandwidth tests). Used by DCGM Exporter as a data source.
Node Status Exporter nvidia-operator-validator Validates that all GPU operator components are functioning correctly. Runs post-install validation (can the container see GPUs? Are drivers loaded?). Reports node readiness status back to the operator.

ClusterPolicy CRD

The GPU Operator is configured through a single ClusterPolicy Custom Resource. This CRD defines the desired state of every component — image versions, runtime configs, feature flags, and node selectors:

3. Network Operator & RDMA in Kubernetes

NVIDIA Network Operator

The NVIDIA Network Operator complements the GPU Operator by managing the high-performance networking stack. In a Superpod, each DGX node has 8 ConnectX-7 NICs (400 Gb/s each) providing 3.2 Tb/s of aggregate InfiniBand or RoCE bandwidth for NCCL collective communication. The Network Operator automates the deployment of MLNX_OFED drivers, SR-IOV configuration, RDMA device plugins, and secondary CNI configuration — ensuring that every GPU pod can access RDMA-capable network interfaces for GPUDirect RDMA data paths.

Multus CNI: Multiple Network Interfaces per Pod

Standard Kubernetes pods get a single network interface from the cluster CNI. For GPU workloads, this is insufficient — you need both a management network (for K8s API, logging, metrics) and high-performance data-plane networks (for NCCL traffic over RDMA). Multus CNI is a meta-CNI plugin that attaches multiple network interfaces to a single pod:

  • Primary interface (eth0) — Provided by the cluster CNI (Calico or Cilium). Carries management traffic, pod-to-pod communication, service discovery, and DNS.
  • Secondary interfaces (net1..netN) — Provided by SR-IOV CNI or host-device CNI. Each interface maps to a ConnectX-7 VF (Virtual Function) or PF (Physical Function) and supports RDMA operations. Typically one per GPU for optimal NCCL topology.
# NetworkAttachmentDefinition for SR-IOV RDMA network
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: sriov-rdma-network
  namespace: gpu-workloads
  annotations:
    k8s.v1.cni.cncf.io/resourceName: nvidia.com/rdma_shared_device
spec:
  config: |
    {
      "cniVersion": "0.3.1",
      "name": "sriov-rdma",
      "type": "sriov",
      "vlan": 0,
      "spoofchk": "off",
      "trust": "on",
      "rdma": {
        "enabled": true
      },
      "ipam": {
        "type": "whereabouts",
        "range": "192.168.100.0/24",
        "exclude": ["192.168.100.0/32", "192.168.100.255/32"]
      }
    }

SR-IOV Device Plugin & RDMA Shared Device Plugin

The SR-IOV Device Plugin discovers SR-IOV Virtual Functions on ConnectX NICs and exposes them as schedulable Kubernetes resources (e.g., nvidia.com/sriov_rdma_vf). The RDMA Shared Device Plugin exposes RDMA-capable devices as nvidia.com/rdma_shared_device, allowing multiple pods on the same node to share RDMA resources. The "shared" mode is typically used because each DGX node runs a single training pod consuming all 8 GPUs and all 8 NICs.

4. Job Scheduling for Distributed Training

The Gang Scheduling Problem

Distributed training jobs require all their workers to start simultaneously. A 256-GPU training job (32 nodes x 8 GPUs) using data-parallel + tensor-parallel strategies cannot make progress if only 31 of 32 workers are running — NCCL initialization hangs waiting for all ranks. Standard Kubernetes scheduling places pods independently, which can lead to deadlock: two jobs each get half their pods scheduled, but neither can complete because the remaining pods are blocked by the other job. Gang scheduling solves this by treating all pods in a job as an atomic unit — either all pods are placed, or none are.

Volcano Scheduler

Volcano is a CNCF project that provides batch scheduling for Kubernetes. It introduces PodGroup and Queue CRDs, implements gang scheduling, fair-share, priority, and backfill policies. It replaces (or supplements) the default kube-scheduler for batch/HPC workloads: 

# Volcano Job for distributed training
apiVersion: batch.volcano.sh/v1alpha1
kind: Job
metadata:
  name: llm-training-gpt4
spec:
  minAvailable: 32                  # Gang scheduling: all 32 tasks required
  schedulerName: volcano
  queue: high-priority-training
  policies:
  - event: PodEvicted
    action: RestartJob              # Restart entire job if any pod evicted
  - event: TaskCompleted
    action: CompleteJob
  plugins:
    sla:
    - --max-waiting-time=30m        # Fail if not scheduled in 30 minutes
  tasks:
  - replicas: 32
    name: worker
    template:
      spec:
        containers:
        - name: pytorch
          image: nvcr.io/nvidia/pytorch:24.01-py3
          command: ["torchrun"]
          args:
          - "--nproc_per_node=8"
          - "--nnodes=32"
          - "--rdzv_backend=c10d"
          - "--rdzv_endpoint=$(MASTER_ADDR):29500"
          - "train.py"
          resources:
            limits:
              nvidia.com/gpu: 8
              nvidia.com/rdma_shared_device: 8
          volumeMounts:
          - name: shm
            mountPath: /dev/shm
          - name: training-data
            mountPath: /data
        volumes:
        - name: shm
          emptyDir:
            medium: Memory
            sizeLimit: "128Gi"
        - name: training-data
          persistentVolumeClaim:
            claimName: lustre-training-data

Kueue: K8s-Native Job Queueing

Kueue is a Kubernetes-native job queueing system (SIG-scheduling project) designed for multi-tenant GPU clusters. Unlike Volcano (which provides its own scheduler), Kueue works alongside the default scheduler by controlling when workloads are admitted to the cluster. It provides:

  • ClusterQueues — Define resource pools (e.g., 256 GPUs for team-A, 128 for team-B)
  • LocalQueues — Namespace-scoped queues that map to ClusterQueues
  • ResourceFlavors — Differentiate GPU types (H100 vs A100) within a queue
  • Cohorts — Allow queues to borrow unused resources from peer queues
  • Preemption — Lower-priority workloads can be preempted when higher-priority work arrives
# Kueue ClusterQueue with resource flavors and borrowing
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: team-research
spec:
  namespaceSelector: {}
  queueingStrategy: BestEffortFIFO
  preemption:
    reclaimWithinCohort: Any
    borrowWithinCohort:
      policy: LowerPriority
      maxPriorityThreshold: 100
  resourceGroups:
  - coveredResources: ["cpu", "memory", "nvidia.com/gpu"]
    flavors:
    - name: h100-sxm
      resources:
      - name: "nvidia.com/gpu"
        nominalQuota: 256            # Team's guaranteed 256 GPUs
        borrowingLimit: 128          # Can borrow up to 128 more from cohort
      - name: "cpu"
        nominalQuota: 2048
      - name: "memory"
        nominalQuota: "16Ti"
  cohort: research-cohort            # Share unused GPUs with peer queues
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: LocalQueue
metadata:
  name: training-queue
  namespace: team-research
spec:
  clusterQueue: team-research

5. Container Runtime for GPUs

How GPU Access Works in Containers

GPU containers in Kubernetes rely on a chain of components that bridge the gap between the Kubernetes resource model and the Linux device layer. The process works as follows:

  1. Pod requests nvidia.com/gpu: N — The Kubernetes scheduler finds a node with N available GPUs (tracked by the NVIDIA Device Plugin). The pod is bound to that node.
  2. Device Plugin allocates GPUs — The kubelet calls the NVIDIA Device Plugin's Allocate() RPC. The device plugin returns the specific GPU device IDs (e.g., GPU-a1b2c3d4) and the environment variable NVIDIA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7.
  3. nvidia-container-runtime hook fires — When containerd creates the container, the NVIDIA container runtime (configured as a runtime handler in containerd) intercepts the OCI spec. The nvidia-container-runtime-hook (now called nvidia-container-toolkit) modifies the container spec to:
    • Bind-mount GPU device files: /dev/nvidia0../dev/nvidia7, /dev/nvidiactl, /dev/nvidia-uvm, /dev/nvidia-uvm-tools
    • Mount NVIDIA driver libraries from the host into the container (libcuda.so, libnvidia-ml.so, etc.)
    • Set NVIDIA_VISIBLE_DEVICES and NVIDIA_DRIVER_CAPABILITIES environment variables
  4. Container sees GPUs — Inside the container, nvidia-smi shows the allocated GPUs, CUDA libraries work transparently, and GPU workloads run as if on bare metal. 
# containerd configuration (/etc/containerd/config.toml)
# The NVIDIA container runtime is registered as a handler

CDI: Container Device Interface

CDI is a newer standard (part of the CNCF container ecosystem) that provides a vendor-neutral, declarative way to make devices available to containers. Instead of runtime hooks that modify the OCI spec at launch time, CDI uses JSON specification files (e.g., /etc/cdi/nvidia.yaml) that describe which device nodes, mounts, and environment variables should be injected. CDI is supported by containerd 1.7+, CRI-O 1.27+, and Podman 4.1+. The NVIDIA Container Toolkit generates CDI specs via nvidia-ctk cdi generate. CDI is the future direction for device exposure in containers, replacing the hook-based approach. 

# Generate CDI spec for NVIDIA GPUs
nvidia-ctk cdi generate --output=/etc/cdi/nvidia.yaml

6. Service Mesh & Networking in Kubernetes

The management-plane network in a GPU cluster handles API server communication, pod-to-pod connectivity for non-RDMA traffic, service discovery, ingress, and observability. While the RDMA data plane carries the heavy NCCL traffic, the management network must still be reliable and performant.

Component Options Notes for GPU Clusters
CNI Plugin Calico (BGP-based, mature, NetworkPolicy support)
Cilium (eBPF-based, high performance, L7 policies) 		Calico is most common in on-prem HPC. Uses BGP peering with ToR switches for native pod routing without overlay (VXLAN overhead avoided). Cilium offers eBPF datapath for lower latency and kernel-bypass networking. Both support NetworkPolicy.
NetworkPolicy Microsegmentation rules per namespace/pod Isolate team namespaces. Allow GPU pods to communicate within a training job (same labels) but block cross-team traffic. Example: allow all traffic within app=llm-training-042 pods, deny ingress from other namespaces.
Load Balancer MetalLB (bare-metal LB using L2/BGP) Bare-metal clusters lack cloud LB integration. MetalLB assigns external IPs from a configured pool and announces them via BGP or ARP/NDP. Used for exposing inference endpoints, Jupyter notebooks, and monitoring dashboards.
Ingress Controller NGINX Ingress Controller
Istio Gateway 		Routes external HTTP/gRPC traffic to services. For Triton Inference Server, the ingress terminates TLS and routes to the correct model endpoint. Istio adds mTLS, traffic splitting (canary deployments), and observability.
Service Types ClusterIP, NodePort, LoadBalancer ClusterIP: Default, internal-only. For inter-service communication.
NodePort: Exposes on static port on all nodes. For debugging.
LoadBalancer: External IP via MetalLB. For production inference APIs.
DNS CoreDNS Cluster-internal DNS for service discovery. Training pods discover the master via DNS (llm-training-master-0.default.svc.cluster.local). Ensure CoreDNS scales to handle large clusters (100+ nodes).

7. Cluster Management at Scale

Day-0 through Day-2 Operations

Managing a Superpod with hundreds of nodes requires automation at every layer: from bare-metal provisioning (Day 0) through initial cluster setup (Day 1) to ongoing operations, upgrades, and scaling (Day 2). The toolchain spans hardware lifecycle management, OS provisioning, Kubernetes cluster creation, and GitOps-driven application deployment.

DGX nodes are provisioned via PXE (Preboot Execution Environment) boot. The process:

  1. Node powers on and BMC (Baseboard Management Controller) network boots via DHCP/TFTP
  2. PXE server delivers boot image (iPXE chain-loads to provisioning system)
  3. MAAS (Metal as a Service) or Foreman manages the full lifecycle:
    • Hardware discovery and commissioning (detect CPUs, GPUs, NICs, storage)
    • OS deployment (Ubuntu 22.04 LTS with NVIDIA-optimized kernel)
    • Network configuration (bonding, VLANs, IP assignment)
    • Post-deploy hooks for NVIDIA driver and MLNX_OFED installation
  4. Ansible takes over for post-OS configuration:
    • NVIDIA driver installation (or GPU Operator handles this)
    • MLNX_OFED installation and configuration
    • containerd installation and configuration
    • kubelet installation and cluster join
    • BIOS/firmware settings (IOMMU, SR-IOV, ACS, power profile)

Knowledge Check

Question 1: GPU Resource Scheduling

In a Kubernetes cluster with the NVIDIA Device Plugin, a pod spec requests nvidia.com/gpu: 4 in its resource limits. A DGX H100 node currently has 3 GPUs free (5 are allocated to other pods). What happens when the scheduler tries to place this pod?

Correct Answer: B. Extended resources like nvidia.com/gpu are integer-only and cannot be overcommitted. The pod stays Pending until a node has at least 4 allocatable GPUs. Unlike CPU and memory, GPUs have no overcommit or fractional scheduling (unless MIG or time-slicing is explicitly configured). Preemption (option D) only occurs if the pod has a higher PriorityClass than existing pods AND preemptionPolicy: PreemptLowerPriority is set — it is not automatic without these configurations.

Question 2: Gang Scheduling

A distributed PyTorchJob requires 32 workers, each needing 8 GPUs (256 total GPUs). The cluster currently has 240 free GPUs across 30 nodes. Without gang scheduling (using the default kube-scheduler), what problem is most likely to occur?

Correct Answer: C. This is the classic partial-scheduling deadlock problem that gang scheduling solves. The default kube-scheduler places pods independently — it schedules 30 of 32 worker pods on available nodes. These 30 workers start and call torch.distributed.init_process_group(), which blocks waiting for all 32 ranks to join. The 2 remaining pods stay Pending because there aren't enough GPUs. The result: 240 GPUs are locked up doing nothing. Gang schedulers like Volcano or Kueue enforce all-or-nothing: either all 32 pods can be placed, or none are admitted.

Question 3: GPU Operator Components

A new DGX H100 node is added to a Kubernetes cluster running the NVIDIA GPU Operator. Which sequence correctly describes what happens automatically?

Correct Answer: B. This is the core value of the GPU Operator — fully automated, zero-touch GPU stack provisioning. When a new node joins the cluster and matches the GPU Operator's node selector, each DaemonSet component rolls out in dependency order: the driver container first (it compiles the kernel module against the running kernel or uses a pre-compiled driver), then the toolkit configures the container runtime, then the device plugin exposes GPUs to the scheduler, GFD labels the node, and DCGM begins monitoring. The entire process is governed by the ClusterPolicy CRD and takes 2-5 minutes depending on whether drivers need compilation. Option D is partially true — you can configure the operator with driver.enabled: false if you pre-install drivers, but by default, the operator manages the full stack including drivers.
