Orchestration & Cluster Management

Kubernetes Fundamentals, Slurm, GPU Operators, RDMA Networking, and Job Scheduling for GPU Superpods

1. Kubernetes Fundamentals

What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform originally developed by Google and now maintained by the Cloud Native Computing Foundation (CNCF). It automates the deployment, scaling, and management of containerized applications across clusters of machines. In a GPU datacenter context, Kubernetes serves as the control plane that decides which workloads run on which nodes, ensures they stay running, and provides networking and storage abstractions that let researchers and engineers focus on their models rather than infrastructure plumbing.

Before Kubernetes, deploying a distributed training job meant SSH-ing into individual machines, manually starting processes, configuring environment variables, and monitoring each node independently. Kubernetes replaces all of that with a declarative API: you describe your desired state in YAML, and Kubernetes continuously works to make reality match your specification. If a node fails, pods are automatically rescheduled. If you scale from 8 to 32 GPUs, you change one number in a manifest.

Control Plane Components

The Kubernetes control plane runs on dedicated master nodes (typically 3 for high availability) and manages the overall cluster state. Understanding these components is essential for debugging scheduling issues, API timeouts, and cluster instability:

Component What It Does Why It Matters for GPU Clusters
kube-apiserver The front door to the cluster. Every kubectl command, every operator controller, and every kubelet heartbeat goes through the API server. It validates and processes RESTful API requests, persists state to etcd, and serves as the hub for all cluster communication. Must be scaled for large clusters (100+ nodes). GPU operators and device plugins make frequent API calls. High API latency = slow scheduling = idle GPUs.
etcd A distributed key-value store that holds all cluster state: node registrations, pod specs, ConfigMaps, Secrets, resource quotas, and custom resources. It uses the Raft consensus algorithm for consistency across replicas. etcd performance directly limits cluster size. Each GPU node with 8 GPUs, 8 RDMA devices, and dozens of labels generates significant state. Use SSDs for etcd, monitor latency, and compact regularly.
kube-scheduler Watches for unscheduled pods and assigns them to nodes. Runs through a series of filter (hard constraints) and score (soft preferences) plugins to find the best node. Filters eliminate nodes that don't meet resource or affinity requirements; scoring ranks the remaining candidates. The default scheduler understands nvidia.com/gpu as an extended resource but lacks GPU topology awareness. Gang scheduling requires additional schedulers (Volcano) or admission controllers (Kueue).
kube-controller-manager Runs a collection of control loops that watch cluster state and make changes to move toward the desired state. Includes the Node Controller (detects node failures), Job Controller (manages batch Jobs), ReplicaSet Controller (maintains pod counts), and many others. The Node Controller's --node-monitor-grace-period determines how quickly a failed GPU node is marked NotReady. Too aggressive = false positives during driver updates. Too slow = wasted GPU time on dead nodes.

Worker Node Components

Every worker node (each DGX system in a SuperPOD) runs these components:

Component What It Does
kubelet The node agent that communicates with the API server. It receives pod specs (via the API server), instructs the container runtime to start containers, reports node status (capacity, allocatable resources, conditions), and manages pod lifecycle (liveness/readiness probes, restarts). The kubelet is also responsible for calling Device Plugin Allocate() RPCs when pods request extended resources like GPUs.
kube-proxy Maintains network rules on each node for Kubernetes Services. Implements the virtual IP (ClusterIP) abstraction by programming iptables or IPVS rules that route traffic to the correct backend pods. In GPU clusters with Cilium, kube-proxy is often replaced by Cilium's eBPF-based implementation for better performance.
Container Runtime The software that actually runs containers. Kubernetes uses the Container Runtime Interface (CRI) to communicate with the runtime. containerd is the standard runtime, with the NVIDIA Container Toolkit providing GPU-aware runtime hooks that inject GPU devices and driver libraries into containers transparently.

Core Kubernetes Objects

Everything in Kubernetes is represented as an object — a persistent entity in the cluster that describes your desired state. Objects are defined in YAML manifests and managed through the API server. Here are the foundational objects you'll work with daily:

Pods

+

A Pod is the smallest deployable unit in Kubernetes — a wrapper around one or more containers that share the same network namespace (IP address), storage volumes, and lifecycle. In GPU workloads, a pod typically contains a single container running a training or inference process. Pods are ephemeral by design: they can be killed, rescheduled, or replaced at any time. 

# Simplest possible pod manifest
apiVersion: v1
kind: Pod
metadata:
  name: my-first-pod
  namespace: default
  labels:
    app: hello-gpu              # Labels are key-value pairs for selection
spec:
  containers:
  - name: cuda-test
    image: nvcr.io/nvidia/cuda:12.3.1-base-ubuntu22.04
    command: ["nvidia-smi"]     # Run nvidia-smi and exit
    resources:
      requests:                 # Minimum resources guaranteed
        cpu: "1"
        memory: "4Gi"
      limits:                   # Maximum resources allowed
        cpu: "4"
        memory: "8Gi"
        nvidia.com/gpu: 1       # Request 1 GPU

Key concept: requests vs limits. Requests are what the scheduler uses for placement decisions — the node must have this much available. Limits are enforced at runtime — the container is throttled (CPU) or killed (memory) if it exceeds them. For GPUs, requests and limits must be equal (no overcommit).

Deployments & ReplicaSets

+

A Deployment manages a set of identical pod replicas. It ensures the specified number of pods are always running, handles rolling updates (gradually replacing old pods with new ones), and supports rollbacks. Deployments create ReplicaSets under the hood, which maintain the actual pod count. In GPU clusters, Deployments are commonly used for inference servers (e.g., Triton Inference Server replicas) where you want N copies of a model-serving container behind a load balancer. 

# Deployment for GPU inference servers
apiVersion: apps/v1
kind: Deployment
metadata:
  name: triton-inference
  namespace: inference-prod
spec:
  replicas: 4                   # Run 4 inference server replicas
  selector:
    matchLabels:
      app: triton               # Must match template labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1         # At most 1 pod down during update
      maxSurge: 1               # At most 1 extra pod during update
  template:                     # Pod template (same as Pod spec)
    metadata:
      labels:
        app: triton
    spec:
      containers:
      - name: triton
        image: nvcr.io/nvidia/tritonserver:24.01-py3
        ports:
        - containerPort: 8000   # HTTP
        - containerPort: 8001   # gRPC
        - containerPort: 8002   # Metrics
        resources:
          limits:
            nvidia.com/gpu: 1   # 1 GPU per replica
        readinessProbe:         # Don't send traffic until ready
          httpGet:
            path: /v2/health/ready
            port: 8000
          initialDelaySeconds: 30
          periodSeconds: 10

Services

+

A Service provides a stable network endpoint (IP address and DNS name) for a set of pods. Since pods are ephemeral and get new IPs when rescheduled, you can't rely on pod IPs directly. A Service uses a label selector to find its backend pods and load-balances traffic across them.

Service Type Behavior Use Case
ClusterIP Internal-only virtual IP, reachable only within the cluster Inter-service communication (e.g., app → database)
NodePort Exposes the service on a static port (30000-32767) on every node's IP Quick external access for debugging/testing
LoadBalancer Provisions an external load balancer (cloud) or uses MetalLB (bare-metal) Production inference API endpoints
Headless No ClusterIP; DNS returns individual pod IPs directly StatefulSets, distributed training master discovery 
# Service exposing Triton inference servers
apiVersion: v1
kind: Service
metadata:
  name: triton-service
  namespace: inference-prod
spec:
  type: ClusterIP
  selector:
    app: triton                 # Routes to pods with this label
  ports:
  - name: http
    port: 8000                  # Service port
    targetPort: 8000            # Container port
  - name: grpc
    port: 8001
    targetPort: 8001

Namespaces

+

Namespaces provide logical isolation within a cluster. They partition resources, apply access controls (RBAC), and scope resource quotas. In a multi-tenant GPU cluster, each team or project typically gets its own namespace with a defined GPU quota: 

# Team namespace with GPU resource quota
apiVersion: v1
kind: Namespace
metadata:
  name: team-nlp
  labels:
    team: nlp
    gpu-tier: high-priority
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: gpu-quota
  namespace: team-nlp
spec:
  hard:
    requests.nvidia.com/gpu: "64"   # Team can use up to 64 GPUs
    requests.cpu: "512"
    requests.memory: "4Ti"
    pods: "200"

Default namespaces: default (user workloads if no namespace specified), kube-system (core K8s components), kube-public (publicly readable), kube-node-lease (node heartbeat leases).

ConfigMaps & Secrets

+

ConfigMaps store non-sensitive configuration data as key-value pairs. They decouple configuration from container images, allowing the same image to be used across environments. Secrets store sensitive data (tokens, passwords, TLS certificates) with base64 encoding and optional encryption at rest. 

# ConfigMap for training hyperparameters
apiVersion: v1
kind: ConfigMap
metadata:
  name: training-config
  namespace: team-nlp
data:
  BATCH_SIZE: "32"
  LEARNING_RATE: "0.0001"
  MAX_EPOCHS: "100"
  NCCL_DEBUG: "WARN"
  NCCL_IB_HCA: "mlx5"
---
# Secret for NGC API key (used to pull containers from nvcr.io)
apiVersion: v1
kind: Secret
metadata:
  name: ngc-secret
  namespace: team-nlp
type: kubernetes.io/dockerconfigjson
data:
  .dockerconfigjson: <base64-encoded-docker-config>

ConfigMaps can be mounted as files or injected as environment variables. For GPU workloads, they commonly hold NCCL tuning parameters, model configurations, and training hyperparameters.

Persistent Volumes & Claims

+

PersistentVolumes (PVs) represent physical storage resources in the cluster (a Lustre mount, an NFS share, a local SSD). PersistentVolumeClaims (PVCs) are user requests for storage — pods reference PVCs, and Kubernetes binds them to suitable PVs. This abstraction lets pods consume storage without knowing the underlying implementation. 

# PV pointing to a Lustre filesystem
apiVersion: v1
kind: PersistentVolume
metadata:
  name: lustre-training-data
spec:
  capacity:
    storage: 100Ti
  accessModes:
  - ReadWriteMany                 # Multiple pods can read/write simultaneously
  persistentVolumeReclaimPolicy: Retain
  storageClassName: lustre
  csi:
    driver: lustre.csi.hpe.com
    volumeHandle: lustre-fs-01
    volumeAttributes:
      mgs: 10.0.1.100@tcp
      fsName: scratch
---
# PVC requesting storage from the Lustre PV
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: training-data
  namespace: team-nlp
spec:
  accessModes:
  - ReadWriteMany
  storageClassName: lustre
  resources:
    requests:
      storage: 10Ti               # Request 10 TiB from the Lustre pool

Access modes: ReadWriteOnce (single node), ReadOnlyMany (many nodes read-only), ReadWriteMany (many nodes read-write). Distributed training requires ReadWriteMany for shared datasets and checkpoints.

Jobs & CronJobs

+

A Job creates one or more pods that run to completion (unlike Deployments, which run indefinitely). Jobs are the natural fit for training runs, data preprocessing, and batch inference. A CronJob creates Jobs on a schedule (e.g., nightly retraining). 

# Job for a single-node GPU training run
apiVersion: batch/v1
kind: Job
metadata:
  name: finetune-bert
  namespace: team-nlp
spec:
  backoffLimit: 3                 # Retry up to 3 times on failure
  activeDeadlineSeconds: 86400    # Kill after 24 hours max
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: trainer
        image: nvcr.io/nvidia/pytorch:24.01-py3
        command: ["python", "train.py", "--model=bert-large"]
        resources:
          limits:
            nvidia.com/gpu: 8     # Use all 8 GPUs on one node
            memory: "512Gi"
        volumeMounts:
        - name: data
          mountPath: /data
      volumes:
      - name: data
        persistentVolumeClaim:
          claimName: training-data

DaemonSets & StatefulSets

+

DaemonSets ensure that a copy of a pod runs on every node (or a subset of nodes matching a selector). The NVIDIA GPU Operator uses DaemonSets extensively — the device plugin, DCGM exporter, GPU feature discovery, and driver container all run as DaemonSets to guarantee one instance per GPU node.

StatefulSets manage stateful applications that need stable network identities and persistent storage. Each pod gets a predictable hostname (pod-name-0, pod-name-1, etc.) and its own PVC. StatefulSets are used for databases (etcd itself), distributed storage systems, and occasionally for distributed training where stable pod identities simplify rank assignment.

Declarative vs. Imperative

Kubernetes strongly favors the declarative model: you write YAML describing the desired end state and kubectl apply it. Kubernetes figures out what changes are needed. The imperative approach (kubectl create, kubectl run) is useful for quick tests but should not be used for production workloads. Always version-control your YAML manifests in Git — this is the foundation of GitOps.

How the Scheduler Makes Decisions

When a pod is created, the scheduler runs a multi-step process to find the best node:

  1. Filtering: Eliminate nodes that cannot run the pod. Reasons include: insufficient CPU/memory/GPU resources, node taints without matching tolerations, node affinity rules that don't match, and pod anti-affinity conflicts.
  2. Scoring: Rank remaining nodes by desirability. Factors include: resource balance (spread vs. pack), topology preferences (preferred node/rack affinity), and custom scoring plugins.
  3. Binding: The highest-scoring node is selected, and the pod is bound to it. The kubelet on that node receives the pod spec and starts containers.
# Check why a pod is stuck in Pending state
kubectl describe pod my-training-pod

# Look for events like:
# "0/32 nodes are available: 32 Insufficient nvidia.com/gpu"
#   → No node has enough free GPUs
# "0/32 nodes are available: 32 node(s) had untolerated taint {nvidia.com/gpu: present}"
#   → Pod is missing the GPU toleration
# "0/32 nodes are available: 8 node(s) didn't match Pod's node affinity/selector"
#   → nodeSelector or nodeAffinity excludes available nodes

2. Kubernetes for GPU Clusters

Why Kubernetes for HPC / GPU Infrastructure?

Kubernetes has become the dominant orchestration platform for GPU clusters because it solves three fundamental challenges at scale: scheduling (placing workloads on the right hardware), scaling (elastically growing from 1 to 1,000+ nodes), and multi-tenancy (isolating teams sharing expensive GPU resources). In a Superpod environment with 32+ DGX nodes each containing 8 GPUs, manual placement of distributed training jobs is impractical. Kubernetes automates this while respecting hardware topology constraints critical for NCCL collective communication performance.

Unlike traditional HPC schedulers (Slurm, PBS), Kubernetes provides a declarative API, a rich ecosystem of operators and controllers, container-native isolation, and seamless integration with CI/CD pipelines. Many organizations run Kubernetes alongside Slurm, using K8s for inference/serving and interactive workloads, and Slurm for batch training. However, with the maturation of gang scheduling (Volcano, Kueue) and training operators (Kubeflow), Kubernetes is increasingly used for large-scale training as well.

GPU-Aware Scheduling

At the core of GPU scheduling in Kubernetes is the Extended Resource mechanism. GPUs are advertised by the NVIDIA Device Plugin as nvidia.com/gpu resources on each node. Pods request GPUs in their resource spec, and the Kubernetes scheduler only places pods on nodes with sufficient available GPUs. Unlike CPU and memory, GPU resources are whole-number only and cannot be overcommitted — if a node has 8 GPUs and 8 are allocated, no more GPU pods can be scheduled there. 

# Pod requesting all 8 GPUs on a DGX H100 node
apiVersion: v1
kind: Pod
metadata:
  name: training-worker-0
  labels:
    app: llm-training
    role: worker
spec:
  containers:
  - name: pytorch-trainer
    image: nvcr.io/nvidia/pytorch:24.01-py3
    resources:
      requests:
        cpu: "64"
        memory: "512Gi"
        nvidia.com/gpu: 8          # Request all 8 H100 GPUs
      limits:
        cpu: "128"
        memory: "1024Gi"
        nvidia.com/gpu: 8          # Limits must equal requests for GPUs
    env:
    - name: NCCL_DEBUG
      value: "INFO"
    - name: NCCL_IB_HCA
      value: "mlx5"               # Use InfiniBand HCAs for NCCL
    - name: NCCL_TOPO_DUMP_FILE
      value: "/var/run/nvidia/topo.xml"
    volumeMounts:
    - name: shm
      mountPath: /dev/shm
  volumes:
  - name: shm
    emptyDir:
      medium: Memory
      sizeLimit: "128Gi"           # Large shared memory for NCCL
Topology-Aware Scheduling

For distributed training with NCCL, pod placement matters enormously. Two pods on the same NVSwitch-connected DGX node communicate at 900 GB/s (NVLink). Two pods on different nodes in the same leaf switch communicate at 400 Gb/s (InfiniBand). Two pods crossing spine switches see higher latency and potential congestion. The scheduler must be topology-aware to co-locate pods in the same rack or pod (network pod) whenever possible. This is achieved via topology spread constraints and pod affinity rules, combined with node labels that encode the physical topology (rack, row, network pod).

Node Labels and Taints for GPU Types

GPU Feature Discovery (GFD), a component of the NVIDIA GPU Operator, automatically labels nodes with detailed GPU information. These labels enable precise scheduling decisions:

nvidia.com/gpu.count 8 Number of GPUs on the node
nvidia.com/gpu.memory 81920 GPU memory in MiB per GPU
nvidia.com/cuda.driver.major 535 CUDA driver major version

Pod Scheduling Strategies

nodeSelector & nodeAffinity

+

nodeSelector is the simplest constraint — a map of key-value label pairs that the target node must match. nodeAffinity provides richer expressions including In, NotIn, Exists, Gt, Lt operators, and supports both required (hard) and preferred (soft) rules. 

affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: nvidia.com/gpu.product
          operator: In
          values:
          - "NVIDIA-H100-SXM5-80GB"
          - "NVIDIA-H200-SXM-141GB"
        - key: nvidia.com/gpu.count
          operator: Gt
          values:
          - "4"                    # Nodes with more than 4 GPUs
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
        - key: topology.kubernetes.io/rack
          operator: In
          values:
          - "rack-01"              # Prefer rack-01 (soft)

podAffinity / podAntiAffinity

+

Pod affinity co-locates pods relative to other pods. For distributed training, you want all workers of the same job on nodes in the same rack (same topologyKey). Pod anti-affinity spreads replicas across failure domains for high availability. 

affinity:
  podAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: training-job
          operator: In
          values:
          - "llm-gpt-run-042"
      topologyKey: topology.kubernetes.io/rack  # Same rack
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 50
      podAffinityTerm:
        labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - "inference-server"
        topologyKey: kubernetes.io/hostname  # Different nodes

Topology Spread Constraints

+

Topology spread constraints distribute pods evenly across topology domains. This is useful for spreading inference replicas across racks for fault tolerance, or concentrating training workers within a network pod for NCCL performance. 

topologySpreadConstraints:
- maxSkew: 1                      # Max difference in pod count across domains
  topologyKey: topology.kubernetes.io/rack
  whenUnsatisfiable: DoNotSchedule  # Hard constraint
  labelSelector:
    matchLabels:
      app: inference-triton
- maxSkew: 2
  topologyKey: topology.kubernetes.io/zone
  whenUnsatisfiable: ScheduleAnyway  # Soft constraint (best-effort)
  labelSelector:
    matchLabels:
      app: inference-triton

3. NVIDIA GPU Operator

Automating the GPU Software Stack

The NVIDIA GPU Operator uses the Kubernetes operator framework to automate the management of all NVIDIA software components needed for GPU provisioning. Instead of manually installing drivers, container toolkits, and device plugins on every node, the GPU Operator deploys and manages these as containerized workloads via DaemonSets. When a new GPU node joins the cluster, the operator automatically installs the full stack — from kernel-mode driver to monitoring exporters — without any manual intervention. This is essential in cloud-native environments where nodes are ephemeral and auto-scaled.

Components Deployed by the GPU Operator

Component DaemonSet Name Function
NVIDIA Driver nvidia-driver-daemonset Container-based driver installation. Compiles and loads the kernel module (nvidia.ko, nvidia-uvm.ko, nvidia-modeset.ko) on the host. Supports pre-compiled drivers for specific kernel versions to avoid compilation overhead. Version is pinned in ClusterPolicy (e.g., 535.129.03).
NVIDIA Container Toolkit nvidia-container-toolkit-daemonset Formerly nvidia-docker2. Installs the nvidia-container-runtime and nvidia-container-cli on the host. Configures containerd to use the NVIDIA runtime as a handler for GPU containers. Handles injecting GPU device nodes (/dev/nvidia*) and driver libraries into containers.
NVIDIA Device Plugin nvidia-device-plugin-daemonset Implements the Kubernetes Device Plugin API. Discovers GPUs on the node and advertises them to the kubelet as nvidia.com/gpu extended resources. Handles GPU allocation, health checking, and device topology reporting. Supports time-slicing for GPU sharing and MIG device advertisement.
DCGM Exporter nvidia-dcgm-exporter Exports GPU metrics via the NVIDIA Data Center GPU Manager (DCGM) in Prometheus format. Metrics include GPU utilization, memory usage, temperature, power draw, ECC errors, NVLink throughput, PCIe throughput, and SM clock frequencies. Default scrape endpoint: :9400/metrics.
GPU Feature Discovery nvidia-gpu-feature-discovery Labels nodes with GPU properties (product name, driver version, CUDA version, compute capability, MIG capability, GPU family). These labels power nodeSelector and nodeAffinity scheduling decisions. Re-labels periodically to detect hardware changes.
MIG Manager nvidia-mig-manager Manages Multi-Instance GPU (MIG) partitioning on A100/H100 GPUs. Applies MIG profiles declaratively via ConfigMaps. Can reconfigure MIG partitions by draining pods, applying the new profile, and uncordoning the node. Supports per-node MIG configurations.
DCGM (standalone) nvidia-dcgm Runs the DCGM daemon for deep GPU health monitoring, diagnostics, and policy-based management. Provides GPU health watches, job statistics, and active health checks (memory stress tests, PCIe bandwidth tests). Used by DCGM Exporter as a data source.
Node Status Exporter nvidia-operator-validator Validates that all GPU operator components are functioning correctly. Runs post-install validation (can the container see GPUs? Are drivers loaded?). Reports node readiness status back to the operator.

ClusterPolicy CRD

The GPU Operator is configured through a single ClusterPolicy Custom Resource. This CRD defines the desired state of every component — image versions, runtime configs, feature flags, and node selectors:

spec:
  operator:

    enabled: true

4. Network Operator & RDMA in Kubernetes

NVIDIA Network Operator

The NVIDIA Network Operator complements the GPU Operator by managing the high-performance networking stack. In a Superpod, each DGX node has 8 ConnectX-7 NICs (400 Gb/s each) providing 3.2 Tb/s of aggregate InfiniBand or RoCE bandwidth for NCCL collective communication. The Network Operator automates the deployment of MLNX_OFED drivers, SR-IOV configuration, RDMA device plugins, and secondary CNI configuration — ensuring that every GPU pod can access RDMA-capable network interfaces for GPUDirect RDMA data paths.

Multus CNI: Multiple Network Interfaces per Pod

Standard Kubernetes pods get a single network interface from the cluster CNI. For GPU workloads, this is insufficient — you need both a management network (for K8s API, logging, metrics) and high-performance data-plane networks (for NCCL traffic over RDMA). Multus CNI is a meta-CNI plugin that attaches multiple network interfaces to a single pod:

  • Primary interface (eth0) — Provided by the cluster CNI (Calico or Cilium). Carries management traffic, pod-to-pod communication, service discovery, and DNS.
  • Secondary interfaces (net1..netN) — Provided by SR-IOV CNI or host-device CNI. Each interface maps to a ConnectX-7 VF (Virtual Function) or PF (Physical Function) and supports RDMA operations. Typically one per GPU for optimal NCCL topology.

5. Job Scheduling for Distributed Training

The Gang Scheduling Problem

Distributed training jobs require all their workers to start simultaneously. A 256-GPU training job (32 nodes x 8 GPUs) using data-parallel + tensor-parallel strategies cannot make progress if only 31 of 32 workers are running — NCCL initialization hangs waiting for all ranks. Standard Kubernetes scheduling places pods independently, which can lead to deadlock: two jobs each get half their pods scheduled, but neither can complete because the remaining pods are blocked by the other job. Gang scheduling solves this by treating all pods in a job as an atomic unit — either all pods are placed, or none are.

Volcano Scheduler

Volcano is a CNCF project that provides batch scheduling for Kubernetes. It introduces PodGroup and Queue CRDs, implements gang scheduling, fair-share, priority, and backfill policies. It replaces (or supplements) the default kube-scheduler for batch/HPC workloads:

  name: llm-training-gpt4

  schedulerName: volcano
  queue: high-priority-training
  policies:
  - event: PodEvicted
    action: RestartJob              # Restart entire job if any pod evicted
  - event: TaskCompleted
    action: CompleteJob
  plugins:
    sla:
    - --max-waiting-time=30m        # Fail if not scheduled in 30 minutes
  tasks:
  - replicas: 32
    name: worker
    template:
      spec:
        containers:
        - name: pytorch
          image: nvcr.io/nvidia/pytorch:24.01-py3

          - "--nproc_per_node=8"

Kueue: K8s-Native Job Queueing

Kueue is a Kubernetes-native job queueing system (SIG-scheduling project) designed for multi-tenant GPU clusters. Unlike Volcano (which provides its own scheduler), Kueue works alongside the default scheduler by controlling when workloads are admitted to the cluster. It provides:

  • ClusterQueues — Define resource pools (e.g., 256 GPUs for team-A, 128 for team-B)
  • LocalQueues — Namespace-scoped queues that map to ClusterQueues
  • ResourceFlavors — Differentiate GPU types (H100 vs A100) within a queue
  • Cohorts — Allow queues to borrow unused resources from peer queues
  • Preemption — Lower-priority workloads can be preempted when higher-priority work arrives
# Kueue ClusterQueue with resource flavors and borrowing
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: team-research
spec:
  namespaceSelector: {}
  queueingStrategy: BestEffortFIFO
  preemption:
    reclaimWithinCohort: Any
    borrowWithinCohort:
      policy: LowerPriority
      maxPriorityThreshold: 100
  resourceGroups:
  - coveredResources: ["cpu", "memory", "nvidia.com/gpu"]
    flavors:
    - name: h100-sxm
      resources:
      - name: "nvidia.com/gpu"
        nominalQuota: 256            # Team's guaranteed 256 GPUs
        borrowingLimit: 128          # Can borrow up to 128 more from cohort
      - name: "cpu"
        nominalQuota: 2048
      - name: "memory"
        nominalQuota: "16Ti"
  cohort: research-cohort            # Share unused GPUs with peer queues

6. Slurm Workload Manager

The HPC Standard

Slurm (Simple Linux Utility for Resource Management) is the dominant workload manager in High-Performance Computing, used by the majority of the world's TOP500 supercomputers. While Kubernetes has become the standard for cloud-native GPU workloads, Slurm remains deeply entrenched in traditional HPC, national labs, and many GPU training clusters — including NVIDIA's own DGX SuperPOD reference architectures. Many large AI organizations run both Slurm and Kubernetes: Slurm for large-scale batch training (where its mature gang scheduling, backfill algorithms, and MPI integration excel) and Kubernetes for inference serving, interactive notebooks, and CI/CD pipelines.

Slurm Architecture

Slurm uses a centralized controller model with lightweight daemons on each compute node:

Component Daemon Role
Controller slurmctld Central management daemon that runs on the head node(s). Monitors resources, accepts job submissions, makes scheduling decisions, and manages the job queue. Can run in HA mode with a backup controller for failover. Reads the cluster configuration from slurm.conf.
Compute Daemon slurmd Runs on every compute node. Accepts work from slurmctld, launches and monitors job steps, reports node status (CPU, memory, GPU utilization), and manages job cleanup. Analogous to Kubernetes' kubelet.
Database Daemon slurmdbd Optional but recommended. Provides a database interface for job accounting, usage tracking, and fair-share scheduling history. Stores data in MySQL/MariaDB. Enables sacct queries for historical job analysis and sshare for fair-share reporting.
REST API slurmrestd REST API daemon (Slurm 20.02+) that exposes job submission, queue status, and node information via HTTP. Enables integration with web portals, monitoring systems, and custom job submission tools.

Slurm Cluster Architecture

Key Concepts

Partitions & QOS

+

A partition is a logical grouping of nodes, similar to a queue. Each partition defines which nodes are available, time limits, priority, and access controls. Think of partitions as different "lanes" for different types of work:

Partition Nodes Time Limit Use Case
batch dgx[001-032] 72 hours Long-running training jobs
interactive dgx[001-004] 8 hours Debugging, Jupyter, interactive dev
preemptible dgx[001-032] 24 hours Low-priority, can be preempted
inference dgx[033-040] Unlimited Model serving (long-running)

Quality of Service (QOS) adds another layer of control on top of partitions, allowing per-user or per-group limits on the number of jobs, GPUs, wall time, and priority. QOS policies can also enable preemption: a high-QOS job can preempt a lower-QOS job to access GPUs. 

# slurm.conf partition definitions
PartitionName=batch Nodes=dgx[001-032] MaxTime=3-00:00:00 Default=YES State=UP
PartitionName=interactive Nodes=dgx[001-004] MaxTime=08:00:00 State=UP
PartitionName=preemptible Nodes=dgx[001-032] MaxTime=1-00:00:00 PreemptMode=REQUEUE State=UP

# QOS definitions (via sacctmgr)
sacctmgr add qos high-priority set Priority=1000 MaxTRESPerUser=gpu=128
sacctmgr add qos normal set Priority=500 MaxTRESPerUser=gpu=64
sacctmgr add qos low-priority set Priority=100 MaxTRESPerUser=gpu=16 Preempt=normal,high-priority

GRES: GPU Scheduling in Slurm

+

Generic Resources (GRES) is Slurm's mechanism for scheduling GPUs and other specialized hardware. Each node declares its GPU inventory, and users request GPUs in their job submissions. Slurm tracks GPU allocation per node, preventing oversubscription. 

# gres.conf — declares GPU resources on each node
# Placed on every compute node at /etc/slurm/gres.conf

# AutoDetect lets Slurm discover GPUs via NVML
AutoDetect=nvml

# Or manually specify (useful when AutoDetect isn't available):
# Name=gpu Type=h100 File=/dev/nvidia[0-7] Count=8

# slurm.conf — enable GRES and configure GPU types
GresTypes=gpu
NodeName=dgx[001-032] Gres=gpu:h100:8 CPUs=128 RealMemory=2048000 State=UNKNOWN

# This tells Slurm: each dgx node has 8 H100 GPUs, 128 CPUs, ~2TB RAM
# Requesting GPUs in job submissions
sbatch --gres=gpu:8 train.sh          # Request 8 GPUs (any type)
sbatch --gres=gpu:h100:4 train.sh     # Request 4 H100 GPUs specifically
sbatch --gres=gpu:8 -N 4 train.sh     # 8 GPUs per node, across 4 nodes (32 total)

# GPU binding options for NUMA/topology awareness
srun --gres=gpu:8 --gpu-bind=closest python train.py
# --gpu-bind=closest: bind each task to GPUs on the same NUMA node
# --gpu-bind=map_gpu:0,1,2,3: explicitly map tasks to GPU indices

There is no partial-scheduling deadlock problem. Slurm's backfill scheduler also excels at packing smaller jobs into gaps while large jobs wait, maximizing overall GPU utilization.

7. Container Runtime for GPUs

How GPU Access Works in Containers

GPU containers in Kubernetes rely on a chain of components that bridge the gap between the Kubernetes resource model and the Linux device layer. The process works as follows:

  1. Pod requests nvidia.com/gpu: N — The Kubernetes scheduler finds a node with N available GPUs (tracked by the NVIDIA Device Plugin). The pod is bound to that node.
  2. Device Plugin allocates GPUs — The kubelet calls the NVIDIA Device Plugin's Allocate() RPC. The device plugin returns the specific GPU device IDs (e.g., GPU-a1b2c3d4) and the environment variable NVIDIA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7.
  3. nvidia-container-runtime hook fires — When containerd creates the container, the NVIDIA container runtime (configured as a runtime handler in containerd) intercepts the OCI spec. The nvidia-container-runtime-hook (now called nvidia-container-toolkit) modifies the container spec to:
    • Bind-mount GPU device files: /dev/nvidia0../dev/nvidia7, /dev/nvidiactl, /dev/nvidia-uvm, /dev/nvidia-uvm-tools
    • Mount NVIDIA driver libraries from the host into the container (libcuda.so, libnvidia-ml.so, etc.)
    • Set NVIDIA_VISIBLE_DEVICES and NVIDIA_DRIVER_CAPABILITIES environment variables
  4. Container sees GPUs — Inside the container, nvidia-smi shows the allocated GPUs, CUDA libraries work transparently, and GPU workloads run as if on bare metal.

CDI: Container Device Interface

CDI is a newer standard (part of the CNCF container ecosystem) that provides a vendor-neutral, declarative way to make devices available to containers. Instead of runtime hooks that modify the OCI spec at launch time, CDI uses JSON specification files (e.g., /etc/cdi/nvidia.yaml) that describe which device nodes, mounts, and environment variables should be injected. CDI is supported by containerd 1.7+, CRI-O 1.27+, and Podman 4.1+. The NVIDIA Container Toolkit generates CDI specs via nvidia-ctk cdi generate. CDI is the future direction for device exposure in containers, replacing the hook-based approach.

Knowledge Check

Question 1: GPU Resource Scheduling

In a Kubernetes cluster with the NVIDIA Device Plugin, a pod spec requests nvidia.com/gpu: 4 in its resource limits. A DGX H100 node currently has 3 GPUs free (5 are allocated to other pods). What happens when the scheduler tries to place this pod?

Correct Answer: B. Extended resources like nvidia.com/gpu are integer-only and cannot be overcommitted. The pod stays Pending until a node has at least 4 allocatable GPUs. Unlike CPU and memory, GPUs have no overcommit or fractional scheduling (unless MIG or time-slicing is explicitly configured). Preemption (option D) only occurs if the pod has a higher PriorityClass than existing pods AND preemptionPolicy: PreemptLowerPriority is set — it is not automatic without these configurations.

Question 2: Gang Scheduling

A distributed PyTorchJob requires 32 workers, each needing 8 GPUs (256 total GPUs). The cluster currently has 240 free GPUs across 30 nodes. Without gang scheduling (using the default kube-scheduler), what problem is most likely to occur?

Correct Answer: C. This is the classic partial-scheduling deadlock problem that gang scheduling solves. The default kube-scheduler places pods independently — it schedules 30 of 32 worker pods on available nodes. These 30 workers start and call torch.distributed.init_process_group(), which blocks waiting for all 32 ranks to join. The 2 remaining pods stay Pending because there aren't enough GPUs. The result: 240 GPUs are locked up doing nothing. Gang schedulers like Volcano or Kueue enforce all-or-nothing: either all 32 pods can be placed, or none are admitted.

Question 3: GPU Operator Components

A new DGX H100 node is added to a Kubernetes cluster running the NVIDIA GPU Operator. Which sequence correctly describes what happens automatically?

Correct Answer: B. This is the core value of the GPU Operator — fully automated, zero-touch GPU stack provisioning. When a new node joins the cluster and matches the GPU Operator's node selector, each DaemonSet component rolls out in dependency order: the driver container first (it compiles the kernel module against the running kernel or uses a pre-compiled driver), then the toolkit configures the container runtime, then the device plugin exposes GPUs to the scheduler, GFD labels the node, and DCGM begins monitoring. The entire process is governed by the ClusterPolicy CRD and takes 2-5 minutes depending on whether drivers need compilation. Option D is partially true — you can configure the operator with driver.enabled: false if you pre-install drivers, but by default, the operator manages the full stack including drivers.
