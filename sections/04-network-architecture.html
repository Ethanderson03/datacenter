Network Architecture

The network fabric is the nervous system of a GPU Superpod. Every collective operation, every gradient exchange, every activation transfer flows through this infrastructure. A poorly designed network turns a $200M compute investment into idle silicon. This section covers the topology, protocols, and redundancy mechanisms that make large-scale distributed GPU training possible.

1. Spine-Leaf (Clos) Topology

Why Spine-Leaf?

Traditional three-tier architectures (access → aggregation → core) create bottlenecks, require Spanning Tree Protocol (STP) to prevent loops, and produce unpredictable latency. The spine-leaf (Clos) topology eliminates these problems:

  • Predictable latency: Every server-to-server path traverses exactly one leaf → spine → leaf hop (or leaf → spine → super-spine → spine → leaf in 3-tier). Maximum hop count is fixed regardless of fabric size.
  • No STP: All links are active simultaneously. ECMP (Equal-Cost Multi-Path) provides loop-free multi-pathing at L3. No blocked ports, no wasted bandwidth.
  • Horizontal scalability: Adding capacity means adding more leaf switches (for ports) or more spine switches (for bandwidth). The architecture scales linearly.
  • Full bisection bandwidth: In a non-blocking configuration, the aggregate bandwidth between any two halves of the fabric equals the total server-facing bandwidth. Every server can communicate at line rate simultaneously.

Fat-Tree Topology for HPC

Charles Leiserson's fat-tree (1985) is the theoretical basis for modern Clos fabrics. In a fat-tree, links get "fatter" (higher bandwidth) as you move up the tree. In practice, this is achieved by increasing the number of equal-bandwidth links at each tier rather than using physically wider links.

For GPU clusters, a non-blocking fat-tree is essential. During AllReduce operations, every GPU sends data to every other GPU simultaneously. Any oversubscription at the spine layer creates a bottleneck that slows the entire collective, wasting GPU cycles across the cluster.

Leaf Switches (Top-of-Rack / ToR)

  • Positioned at the top of each server rack
  • Downlink ports connect to servers, GPUs (via ConnectX-7 NICs), and storage nodes
  • Uplink ports connect to every spine switch in the fabric
  • Typical: NVIDIA Spectrum-4 SN5600 with 64x 400GbE ports, or Quantum-2 QM9700 with 64x NDR (400Gb/s) InfiniBand ports
  • Each leaf has a unique /31 or unnumbered link to every spine
  • Performs L2 switching locally (within rack) and L3 routing to the fabric

Spine Switches

  • Every spine connects to every leaf switch in the pod
  • No servers connect directly to spine switches
  • Purpose: provide full mesh connectivity between all leaves
  • For a non-blocking fabric: total spine uplink bandwidth from each leaf must equal total server-facing bandwidth on that leaf
  • Example: 32 server ports at 400G = 12.8 Tb/s down → need 32 uplinks at 400G = 12.8 Tb/s to spines

Super-Spine (3-Tier Clos)

  • Required when the fabric exceeds what a single spine tier can interconnect
  • A pod = one complete leaf+spine fabric. Super-spines connect pods together.
  • NVIDIA DGX SuperPOD with 256+ DGX nodes typically requires super-spine for the InfiniBand compute fabric
  • Each spine connects to every super-spine; each super-spine connects to every spine across all pods
  • Maximum 3-tier fat-tree with 64-port switches: 64 pods × 32 leaves/pod × 32 servers/leaf = 65,536 endpoints

Oversubscription Ratios

Ratio Meaning Use Case Cost Impact
1:1 (Non-blocking) Spine bandwidth = leaf server bandwidth. Every port can run at line rate simultaneously. GPU training clusters, HPC, InfiniBand fabrics. Mandatory for AllReduce-heavy workloads. Highest cost. Requires maximum spine switch count.
2:1 Half the spine bandwidth vs server bandwidth. Statistical multiplexing assumed. Storage networks, inference serving, enterprise workloads where not all hosts communicate simultaneously. ~50% fewer spine switches and uplinks.
3:1 One-third spine bandwidth vs server bandwidth. General enterprise, web serving, microservices. Acceptable when traffic is bursty and not all-to-all. Significant savings on spine infrastructure.

Rail-Optimized Topology

In NVIDIA's rail-optimized design for DGX H100/B200 clusters, each of the 8 GPU NICs in a server connects to a different leaf switch (a different "rail"). GPU 0 across all servers connects to Rail 0 leaf switches, GPU 1 to Rail 1 leaf switches, and so on.

This means intra-rail traffic (GPU 0 on server A talking to GPU 0 on server B) stays within a single leaf switch or traverses one leaf-spine-leaf path, while cross-rail traffic uses NVSwitch/NVLink within the node. This aligns the physical topology with NCCL's ring/tree algorithms, reduces the number of spine switches needed, and confines most collective traffic to a single rail, dramatically simplifying traffic engineering.

For a 32-node DGX H100 pod with 8 rails: 8 leaf switches (one per rail), each with 32 downlinks (one per server's GPU N) and uplinks to shared spine switches. A 256-node SuperPOD uses 8 rails × 8 leaf switches per rail = 64 leaf switches.

Spine-Leaf Topology: Compute Fabric (InfiniBand) & Management/Storage Fabric (Ethernet)

2. East-West Traffic (Data Plane)

What Is East-West Traffic?

East-west traffic flows laterally between servers, GPUs, and storage nodes within the datacenter. In a GPU training cluster, east-west traffic constitutes 90-99% of all network traffic. This is fundamentally different from traditional enterprise datacenters where north-south (client-server) traffic dominates.

During distributed training of a large language model across 2,048 GPUs, every GPU must exchange gradient data with every other GPU during each training step. A single AllReduce operation on a 175B-parameter model moves approximately 350 GB of gradient data across the fabric in under 1 second. This is why non-blocking fabric design is non-negotiable.

NCCL: NVIDIA Collective Communications Library

NCCL is the communication backbone for distributed GPU training. It abstracts the underlying network transport (InfiniBand Verbs, RoCE, TCP sockets) and implements optimized collective operations:

Collective Operation Training Use Case Traffic Pattern
AllReduce Reduce data across all ranks, result available on all ranks Data parallelism: aggregate gradients from all workers All-to-all; each GPU sends and receives ~2 × (N-1)/N × data_size
AllGather Gather data from all ranks to all ranks FSDP (Fully Sharded Data Parallel): reconstruct full model weights All-to-all; total data moved = N × shard_size
ReduceScatter Reduce and scatter result chunks across ranks FSDP: reduce gradients and distribute shards All-to-all; each GPU ends with 1/N of reduced result
Broadcast Send data from one rank to all others Weight initialization, hyperparameter sync One-to-all; tree-shaped distribution
Send/Recv (P2P) Point-to-point data transfer between two specific ranks Model/pipeline parallelism: activation exchange between stages Point-to-point; creates predictable flows

Ring Algorithm

GPUs are arranged in a logical ring. Data is chunked and pipelined around the ring in two phases: scatter-reduce (each GPU reduces a chunk and passes it on) and allgather (the fully reduced chunks circulate). For N GPUs and data size D:

  • Each GPU sends: 2 × (N-1)/N × D total bytes
  • Latency: 2 × (N-1) × α (where α is per-hop latency)
  • Bandwidth-optimal: approaches theoretical limit for large D
  • Latency-suboptimal: O(N) latency terms; slow at large GPU counts with small messages
  • NCCL creates multiple rings across NVLink and network to maximize bandwidth

Tree Algorithm

GPUs are arranged in a binary tree (or double binary tree for full-duplex utilization). Data flows up the tree (reduce) and down the tree (broadcast).

  • Latency: 2 × log2(N) × α — dramatically better than ring for large N
  • Bandwidth: each GPU sends 2 × D total bytes (less efficient than ring for large D)
  • NCCL auto-selects ring for large messages, tree for small messages (crossover ~256KB-4MB depending on GPU count)

Mapping to InfiniBand / RoCE

NCCL uses RDMA verbs directly over InfiniBand or RoCE v2. Key transport mechanisms:

  • GPUDirect RDMA: NIC reads/writes GPU memory directly via PCIe BAR, bypassing CPU and system memory. Reduces latency by ~5μs per transfer.
  • GPUDirect Async: GPU kernel triggers RDMA operations without CPU involvement, enabling full overlap of compute and communication.
  • SHARP (Scalable Hierarchical Aggregation and Reduction Protocol): Offloads reduction operations to InfiniBand switch ASICs. The network performs the math during transit, halving the data traversal. Available on Quantum-2 switches.

Traffic Patterns During Distributed Training

Data Parallelism (AllReduce-dominant)

Each GPU holds a full copy of the model and trains on different data. After the backward pass, gradients are synchronized via AllReduce. For a model with P parameters in FP16, each AllReduce moves approximately 2 × P bytes. A 70B-parameter model: 2 × 70×109 = 140 GB per AllReduce. With BF16 gradients, this must complete within the backward pass time (~500ms for typical LLM training) to avoid becoming the bottleneck.

Model (Tensor) Parallelism (P2P-dominant)

The model is split across GPUs within a node (typically 8-way tensor parallelism over NVLink). Each layer requires two AllReduce operations on activations (forward + backward). Traffic volume per layer: 2 × batch_size × sequence_length × hidden_dim × 2 bytes. This traffic stays within the NVSwitch domain (900 GB/s per GPU on H100) and does not touch the InfiniBand fabric.

Pipeline Parallelism (Sequential P2P)

The model is split into stages across nodes. Micro-batches flow between stages via point-to-point Send/Recv. Traffic is sequential and relatively low-bandwidth compared to AllReduce, but latency-sensitive. Typical activation size per micro-batch: batch_size × seq_len × hidden_dim × 2 bytes (~50-200 MB).

Fully Sharded Data Parallel (FSDP / ZeRO-3)

Model parameters, gradients, and optimizer states are sharded across GPUs. Before each forward/backward layer, an AllGather reconstructs parameters; after backward, a ReduceScatter distributes gradient shards. This trades 3x more communication for N-fold memory savings. Communication volume per layer: 2 × layer_params × 2 bytes (AllGather + ReduceScatter).

Bandwidth Is Non-Negotiable

In a 2,048-GPU cluster running AllReduce over a ring, each GPU NIC must sustain near line-rate throughput (~385 Gb/s usable out of 400 Gb/s NDR) for the entire duration of the gradient synchronization phase. Even a 10% bandwidth reduction on a single link cascades into a proportional training slowdown across all 2,048 GPUs, because the collective cannot complete until the slowest participant finishes.

ECMP and Adaptive Routing

ECMP (Equal-Cost Multi-Path) is the standard mechanism for distributing traffic across multiple spine links. When a leaf switch has 32 equal-cost paths to another leaf (through 32 spine switches), ECMP selects a path using a hash of the packet header fields (5-tuple: src IP, dst IP, src port, dst port, protocol).

The Elephant Flow Problem

NCCL generates a small number of very large, long-lived flows ("elephant flows"). With 5-tuple hashing, multiple elephant flows can hash to the same spine link, creating congestion on that path while other paths are idle. In a 32-spine fabric, the probability of at least one collision with just 8 flows is approximately 75% (birthday problem).

Solutions
  • Adaptive Routing (NVIDIA Quantum-2): InfiniBand switches dynamically reroute packets based on real-time port congestion. Each packet can take a different path, achieving near-perfect load balancing. This is the preferred solution for InfiniBand fabrics.
  • Packet Spraying: Each packet is independently routed across all paths. Achieves perfect load balance but requires reordering at the receiver. Supported by some Ethernet switches (e.g., Memory-based Memory Fabrics, Barefoot/Intel Tofino).
  • Flowlet Switching: Splits flows into "flowlets" (bursts separated by idle gaps) and assigns each flowlet to a different path. Balances load without reordering within a burst. Used by SONiC's dynamic hashing.
  • Larger Hash Entropy: Using NCCL's multi-port/multi-QP feature to create more parallel connections, increasing the number of flows and reducing collision probability.

3. North-South Traffic (Control Plane + Ingress/Egress)

What Is North-South Traffic?

North-south traffic flows vertically between the datacenter and external networks (the internet, other datacenters, cloud providers). In a GPU training cluster, this is typically less than 5% of total traffic but is critical for operations:

  • Model and dataset ingestion: Downloading training data from S3/GCS/Azure Blob, pulling model checkpoints (can be 100s of GB but infrequent)
  • Checkpoint upload: Saving model checkpoints to remote storage during training
  • API serving: If the cluster also serves inference traffic, client requests enter via north-south paths
  • Management: SSH access, monitoring (Prometheus/Grafana), log shipping (Elasticsearch/Splunk), LDAP/SSO authentication
  • Software updates: OS patches, driver updates, container image pulls

North-South vs East-West Traffic Flows

North-South Infrastructure Components

  • Border / Edge Routers: Terminate BGP sessions with ISPs and transit providers. Typically deployed in redundant pairs (active-active with ECMP or active-standby). Handle route filtering, AS-path prepending, and community tagging. Common hardware: Arista 7280R3, Cisco 8000 series, Juniper MX304.
  • Firewalls: Stateful packet inspection, ACLs, and DDoS mitigation at the datacenter edge. For GPU clusters, the primary concern is protecting management interfaces (SSH, IPMI/BMC) from external access. Training traffic (InfiniBand) never touches the firewall because it stays on the compute fabric. Typical throughput: 100Gbps+ (Palo Alto PA-7000, Fortinet 4400F).
  • Load Balancers: Required if the cluster serves inference APIs.
    • L4 (Transport): Distributes TCP/UDP connections based on 5-tuple hash. Low overhead, does not inspect payload. Good for distributing inference requests across GPU servers. Examples: NVIDIA BlueField DPU, DPDK-based (Katran/Facebook), IPVS/LVS.
    • L7 (Application): Inspects HTTP headers, URLs, cookies. Can route based on model name, API version, or request content. Higher overhead. Examples: Envoy, HAProxy, NGINX, AWS ALB equivalent.

4. BGP (Border Gateway Protocol)

BGP in the Datacenter

BGP (RFC 4271) was originally designed for inter-domain routing on the internet, but has become the dominant routing protocol for datacenter fabrics, replacing OSPF and IS-IS in modern deployments. The key insight: treat each switch in the spine-leaf fabric as its own "autonomous system" and run eBGP between them (RFC 7938: "Use of BGP for Routing in Large-Scale Data Centers").

eBGP for External Peering

Border routers establish eBGP sessions with upstream ISPs, transit providers, and peering partners. Each peer advertises its routes; the border router selects the best path based on BGP's path selection algorithm:

  1. Highest local preference (prefer one ISP over another)
  2. Shortest AS path (fewest autonomous systems traversed)
  3. Lowest origin type (IGP < EGP < incomplete)
  4. Lowest MED (Multi-Exit Discriminator, hint from peer)
  5. eBGP over iBGP (prefer external routes)
  6. Lowest IGP metric to next-hop
  7. Oldest route (stability preference)
  8. Lowest router ID (tiebreaker)

Typical configuration: the datacenter's own public ASN (e.g., AS 396356 for NVIDIA), advertising the datacenter's IP prefixes to upstream peers with appropriate communities for traffic engineering.

iBGP Within the Datacenter

Traditional iBGP (same ASN on all routers) requires either a full mesh of iBGP sessions (N×(N-1)/2 sessions) or route reflectors to propagate routes. This creates scaling problems:

  • Full mesh with 128 switches = 8,128 iBGP sessions
  • Route reflectors add complexity and single points of failure
  • iBGP does not modify the AS path, making loop detection harder
  • Next-hop-self configuration required on every device

For these reasons, modern datacenter designs prefer eBGP with private ASNs (see BGP as DC IGP tab) over traditional iBGP.

BGP as an IGP (RFC 7938)

The most common approach in modern datacenter fabrics: assign each switch its own private ASN (from the 4-byte range 4200000000-4294967294) and run eBGP on every link.

Design Pattern
  • Each leaf switch: unique ASN (e.g., AS 4200000001, 4200000002, ...)
  • Each spine switch: unique ASN (e.g., AS 4200001001, 4200001002, ...)
  • eBGP session on every leaf-spine link (point-to-point /31 or unnumbered)
  • No route reflectors needed (eBGP propagates routes naturally)
  • Loop prevention via AS-path (inherent in eBGP)
  • ECMP across all spine paths (multipath relax with bestpath as-path multipath-relax)
Advantages Over OSPF/IS-IS
  • No complex area design or LSA flooding
  • Policy control at every hop (route-maps, communities)
  • Slower convergence is mitigated by BFD (sub-second detection)
  • Battle-tested at massive scale (every major cloud provider uses this)
  • Simpler troubleshooting: show bgp summary immediately shows peer state
Unnumbered Interfaces (RFC 5765)

Instead of assigning /31 subnets to every point-to-point link (which consumes IP addresses and creates management overhead), BGP unnumbered uses IPv6 link-local addresses for peering and advertises IPv4 routes with the link-local next-hop. This is supported by FRRouting (FRR) on Cumulus/SONiC and NVIDIA Spectrum switches natively.

EVPN-VXLAN with BGP

When Layer 2 connectivity across racks is needed (e.g., for VM migration, legacy applications, or container networking), EVPN (Ethernet VPN) provides an overlay solution using VXLAN encapsulation with BGP as the control plane:

  • VXLAN: Encapsulates L2 frames in UDP (port 4789) for transport over the L3 spine-leaf fabric. 24-bit VNI (VXLAN Network Identifier) supports ~16 million virtual networks.
  • BGP EVPN (Address Family L2VPN EVPN): Distributes MAC addresses, IP addresses, and VNI-to-VTEP mappings via BGP rather than flood-and-learn. Eliminates BUM (Broadcast, Unknown unicast, Multicast) flooding.
  • Type-2 routes: MAC/IP advertisement (host reachability)
  • Type-5 routes: IP prefix routes (inter-VXLAN routing)
  • Distributed Anycast Gateway: Every leaf switch acts as the default gateway for its local VNIs using the same virtual MAC and IP. No VRRP/HSRP needed.

In GPU clusters, EVPN-VXLAN is primarily used on the Ethernet management/storage fabric, not on the InfiniBand compute fabric. It provides isolated network segments for BMC/IPMI management, storage traffic, and container networking (e.g., Kubernetes pod networks).

BFD, Graceful Restart, and Route Dampening

  • BFD (Bidirectional Forwarding Detection): Lightweight protocol running independently of BGP to detect link/node failures in ~50ms (configurable: 3 × transmit interval, typically 50ms × 3 = 150ms). When BFD detects a failure, it notifies BGP to immediately withdraw routes rather than waiting for the BGP hold timer (default 180 seconds). Essential for sub-second failover in spine-leaf fabrics.
  • Graceful Restart (GR): Allows a router to maintain forwarding during BGP process restart (e.g., software upgrade). The restarting router signals GR capability; peers maintain stale routes for a configurable period (restart-time, typically 120s). Critical for maintenance without traffic disruption. However, GR conflicts with BFD (BFD detects the restart as a failure), so Long-Lived Graceful Restart (LLGR) is sometimes used instead.
  • Route Dampening: Suppresses routes that flap repeatedly (go up/down/up/down). Assigns a penalty on each flap; when the penalty exceeds a threshold, the route is suppressed until the penalty decays below a reuse threshold. Prevents instability from propagating across the fabric. Typically used on external BGP sessions, not internal DC fabric links.
  • BGP Communities: 32-bit tags attached to routes for policy signaling. Standard communities (e.g., NO_EXPORT, NO_ADVERTISE), extended communities (for EVPN), and large communities (RFC 8092, 12-byte) for complex traffic engineering. Examples: tag routes from ISP-A for backup-only use, tag internal routes by datacenter region.

5. Blended ISP / Transit

What Is Blended Transit?

Blended transit means connecting the datacenter to multiple upstream ISPs and transit providers simultaneously to achieve redundancy, performance, and cost optimization. Rather than relying on a single provider, traffic is distributed across multiple paths based on BGP routing policy.

Transit vs Peering vs IXP

Type Definition Cost Model Typical Use
Transit Paying an upstream provider to carry your traffic to the entire internet. They advertise your prefixes to their peers and provide you with a full routing table (~950K IPv4 routes as of 2025). Per-Mbps commit + 95th percentile overage. $0.30-$2.00/Mbps/month depending on volume and location. Primary internet access. Every datacenter needs at least one transit provider (usually two for redundancy).
Peering (PNI) Direct private interconnection with another network (e.g., a cloud provider or CDN). Settlement-free (no payment) or paid. Each party only advertises its own routes. Often free (settlement-free). Cross-connect fee: $200-500/month. Paid peering: negotiated rate. High-volume traffic exchange with specific networks. Reduces transit costs and latency. E.g., peering with AWS for model checkpoint uploads.
IXP (Internet Exchange) A shared switching fabric (usually Ethernet) where multiple networks peer with each other. Route servers simplify multilateral peering. Port fee: $500-$5,000/month for 10G-100G. No per-Mbps charge. Each peer relationship is settlement-free. Peer with many networks simultaneously. Major IXPs: DE-CIX, AMS-IX, LINX, Equinix IX. Reduces transit usage significantly.

Multi-Homed Active-Active Setup

A properly multi-homed datacenter has:

  • At least 2 transit providers with physically diverse paths (different fiber routes, different buildings for upstream POPs)
  • Own ASN and Provider-Independent (PI) IP space (at minimum a /24 IPv4 and /48 IPv6 from an RIR allocation)
  • Active-active traffic engineering: Both ISPs carry traffic simultaneously. BGP local-preference determines outbound ISP selection; AS-path prepending and MED influence inbound traffic from the internet.
  • Automatic failover: If one ISP fails, BFD detects the link down in <200ms, BGP withdraws routes via that ISP, and all traffic shifts to the remaining provider within 1-5 seconds.
Anycast for Global Services

If inference endpoints are served from multiple datacenter locations, anycast allows the same IP address to be advertised via BGP from multiple sites. Users are routed to the nearest datacenter based on BGP path selection, providing automatic geographic load balancing and failover.

DDoS Scrubbing

GPU clusters serving public inference APIs are targets for volumetric DDoS attacks. Solutions include:

  • Upstream scrubbing: Providers like Cloudflare, Akamai Prolexic, or AWS Shield proxy traffic and filter attacks before they reach the datacenter. Traffic is routed through the scrubbing center via BGP (either always-on or on-demand via BGP community signaling).
  • On-premise mitigation: Dedicated DDoS appliances (Arbor/Netscout, A10 Thunder TPS) at the datacenter edge for lower-latency filtering.
  • Flowspec: BGP Flowspec (RFC 5575) allows border routers to install packet-level filtering rules distributed via BGP, enabling upstream providers to filter attack traffic at their edge.

Typical ISP Uplink Sizing for GPU Datacenters

Cluster Size ISP Bandwidth Billing Model Notes
64-256 GPUs (single pod) 2 × 10 GbE per ISP (2 ISPs) 95th percentile, 1-5 Gbps commit Sufficient for dataset ingestion + management. Training traffic stays on IB fabric.
512-2048 GPUs (SuperPOD) 2 × 100 GbE per ISP (2 ISPs) 95th percentile, 10-40 Gbps commit Higher bandwidth needed for checkpoint upload to cloud storage and multi-site training sync.

95th Percentile Billing

Most transit providers bill based on the 95th percentile of traffic measured in 5-minute samples over a month. The top 5% of samples are discarded (allows for short bursts). The remaining highest sample determines the bill. For a GPU cluster, large dataset downloads create temporary spikes, but steady-state north-south traffic is typically low. The commit rate is the minimum guaranteed bandwidth and minimum monthly charge. Overages above the commit are billed per-Mbps at a negotiated rate.

6. Redundancy & High Availability

Link Redundancy: MLAG / VPC

MLAG (Multi-Chassis Link Aggregation Group), also called VPC (Virtual Port Channel) on Cisco or MC-LAG, allows two physical switches to appear as a single logical switch to downstream devices. This is used on the Ethernet fabric (not InfiniBand, which uses its own redundancy mechanisms).

MLAG Components
  • Peer Link: A high-bandwidth link (typically 2-4 × 100GbE LAG) between the two MLAG peer switches. Carries orphan traffic (traffic destined for a host connected to only one peer) and control-plane synchronization. This is not the primary data path — in a well-designed MLAG, the peer link carries minimal data traffic.
  • Keepalive Link: A low-bandwidth out-of-band link (1GbE management port) used solely for heartbeat detection between peers. If the keepalive is lost but the peer link is up, the secondary switch disables its MLAG ports to prevent split-brain.
  • Orphan Ports: Server ports connected to only one of the two MLAG peers (not dual-homed). Traffic to/from orphan ports must traverse the peer link to reach the other peer's uplinks. Minimize orphan ports in production designs.
  • Dual-Homed Servers: Each server's bonded NIC (LACP bond) connects one link to each MLAG peer. From the server's perspective, it has a single LAG to a single switch. If one peer fails, the server's traffic flows entirely through the surviving peer with no reconfiguration.

LACP (802.3ad / 802.1AX)

Link Aggregation Control Protocol dynamically negotiates and manages link aggregation groups (LAGs / bonds). Key aspects:

  • LACP negotiation: Both endpoints exchange LACPDUs (every 1 second in fast mode, 30 seconds in slow mode) to agree on LAG membership. If a link fails to exchange LACPDUs, it is removed from the LAG.
  • Active vs Passive: Active mode sends LACPDUs immediately; Passive mode only responds. Best practice: one side Active, one Passive (or both Active).
  • Hash-based distribution: Outbound traffic is distributed across member links using a hash (typically L3+L4: src/dst IP + src/dst port). This means a single flow uses a single link; aggregate bandwidth is achieved across multiple flows.
  • Min-links: Configure the minimum number of active links required for the LAG to remain up. If active links drop below this threshold, the entire LAG is brought down, triggering BFD/BGP failover to alternate paths.

Active-Active vs Active-Standby Comparison

Aspect Active-Active Active-Standby
Bandwidth utilization Both links carry traffic simultaneously — full aggregate bandwidth available Standby link carries no traffic — 50% bandwidth wasted
Failover time Sub-second (traffic already flowing on surviving link, just redistributed) 1-5 seconds (standby must activate, MAC tables update)
Complexity Higher: requires ECMP or MLAG, consistent hashing, state sync between peers Lower: clear primary/secondary roles, simpler troubleshooting
Use in GPU DC Preferred for all fabric links. Spine-leaf inherently active-active via ECMP. Sometimes used for management/OOB networks where simplicity is prioritized.

Additional HA Mechanisms

  • BFD (Bidirectional Forwarding Detection): As covered in the BGP section, BFD provides sub-second failure detection (50-300ms). It runs independently of routing protocols and notifies BGP/OSPF immediately upon detecting a link failure. Critical for maintaining training job continuity — a 3-minute BGP hold-timer expiry would cause NCCL timeouts and kill the training job.
  • VRRP/HSRP (Gateway Redundancy): Virtual Router Redundancy Protocol (VRRP, RFC 5798) and Hot Standby Router Protocol (HSRP, Cisco proprietary) provide a virtual IP that floats between two routers. One is active (responds to ARP), one is standby. Less common in modern spine-leaf designs because anycast gateway (via EVPN) is preferred — every leaf advertises the same gateway IP with the same virtual MAC, so there is no single active router.
  • Redundant Control Planes: High-end switches (Cisco Nexus 9000, Arista 7500R) have dual supervisor modules. If the active supervisor fails, the standby takes over with stateful switchover (SSO) — forwarding continues uninterrupted during the transition. On InfiniBand, the Subnet Manager (OpenSM or NVIDIA UFM) runs on redundant management nodes in active-standby configuration.
  • Anycast Gateway (Distributed Gateway): In an EVPN-VXLAN fabric, every leaf switch is configured with the same virtual gateway IP and MAC for each VLAN/VNI. Hosts always reach their gateway locally on their own leaf switch — no hairpinning to a centralized gateway. If a leaf fails, hosts on surviving leaves are unaffected. This replaces VRRP/HSRP entirely.

7. Routing Protocols Inside the DC

OSPF in the Datacenter

OSPF (Open Shortest Path First, RFC 2328 for OSPFv2, RFC 5340 for OSPFv3) is a link-state protocol. Each router builds a complete topology map (Link-State Database / LSDB) and runs Dijkstra's SPF algorithm to compute shortest paths.

OSPF Area Design for DC
  • Area 0 (Backbone): Spine switches and inter-area links
  • Non-backbone areas: Each pod or rack group in its own area to limit LSA flooding
  • Stub / NSSA areas: Reduce routing table size on leaf switches by summarizing external routes
LSA Types
  • Type 1 (Router LSA): Every router advertises its links and costs within an area
  • Type 2 (Network LSA): DR (Designated Router) on multi-access segments advertises attached routers. Unnecessary on point-to-point links (use ip ospf network point-to-point to suppress DR election).
  • Type 3 (Summary LSA): ABR (Area Border Router) advertises inter-area routes. Spine switches acting as ABRs summarize leaf subnets.
  • Type 5 (External LSA): ASBR (Autonomous System Border Router) redistributes external routes into OSPF.
OSPF Limitations in Large DC Fabrics
  • SPF recalculation on every topology change floods the entire area
  • No policy control at individual router level (OSPF is a shortest-path algorithm; you cannot prefer one path for specific prefixes without hacks)
  • DR election on broadcast segments adds complexity (mitigated by point-to-point links)
  • Full LSDB replication in large areas consumes memory and CPU
  • ECMP limited by SPF cost calculation (all equal-cost paths must have identical metric)

BGP as the DC Routing Protocol

As detailed in Section 4, BGP with private ASNs per switch (RFC 7938) has become the dominant routing protocol in large-scale datacenter fabrics. Key operational advantages:

Unnumbered Interfaces

BGP unnumbered (supported in FRRouting/Cumulus/SONiC) establishes BGP sessions over IPv6 link-local addresses while carrying IPv4 NLRI. Benefits:

  • No /31 subnets to allocate, manage, or document for inter-switch links
  • Each switch needs only a single loopback IPv4 address (used as router-id and next-hop)
  • Dramatically simplifies IP address management in fabrics with hundreds of switches
  • Configuration is identical on every leaf-spine link, enabling zero-touch provisioning
Route Summarization at Spine

Spine switches can summarize routes from their connected leaves before advertising to other spines or super-spines. For example, if Leaf 0-7 (a pod) uses 10.0.0.0/24 through 10.0.7.0/24, the spine advertises a single 10.0.0.0/21 aggregate. This reduces the routing table size on super-spines and border routers, improving convergence time and reducing memory consumption.

BGP vs OSPF for Datacenter Fabrics

Criterion OSPF BGP (eBGP w/ private ASN)
Convergence speed Fast SPF recalculation (~ms), but LSA flooding can cause churn in large areas Slower inherent convergence (best-path recalculation), but BFD provides equivalent failure detection speed
Scalability Requires careful area design. Single area limited to ~200-300 routers before LSDB becomes problematic. Scales to thousands of switches. No area design needed. Each switch is its own AS.
Policy control Minimal. Can only manipulate link cost. No per-prefix policy at individual routers. Rich policy: route-maps, prefix-lists, communities on every session. Can filter, prefer, or reject any prefix at any hop.

Configuration complexity Moderate. Area assignment, network statements, DR tuning on broadcast segments. Low with unnumbered interfaces and peer-groups. Identical config on every leaf. Automation-friendly.
Troubleshooting Requires understanding LSDB, LSA types, SPF tree. show ip ospf database output is verbose. Straightforward. show bgp summary shows peer state at a glance. show bgp shows selected paths.

Bottom Line

For GPU cluster datacenter fabrics, BGP with private ASNs is the clear winner. It provides the scalability needed for large fabrics, the policy control needed for traffic engineering, and the operational simplicity that comes from uniform configuration across all switches. OSPF remains viable for smaller deployments (<50 switches) where the simpler conceptual model may be preferred, but offers no advantage at scale.

Knowledge Check: Network Architecture

Question 1: In a rail-optimized topology for a DGX H100 cluster with 8 GPUs per node, how are the GPU NICs connected to the leaf switches?

Correct. Rail-optimized topology assigns each GPU index to a separate "rail" (set of leaf switches). GPU 0 on every server in the pod connects to Rail 0 leaf switches, GPU 1 to Rail 1, etc. This means intra-rail AllReduce traffic (GPU N on Server A talking to GPU N on Server B) stays within a single rail's leaf-spine sub-fabric. Cross-GPU communication within a node uses NVSwitch/NVLink (900 GB/s on H100), not the network fabric. This design dramatically reduces cross-rail traffic and simplifies ECMP load balancing.

Incorrect. In a rail-optimized design, each of the 8 GPUs in a DGX node connects its NIC to a different leaf switch rail. GPU 0 across all servers connects to Rail 0 leaves, GPU 1 to Rail 1 leaves, etc. This aligns the network topology with NCCL's collective algorithms, keeping most AllReduce traffic within a single rail and using intra-node NVLink for cross-GPU communication.

Question 2: Why does RFC 7938 recommend using eBGP with private ASNs (one per switch) instead of OSPF or iBGP for datacenter spine-leaf fabrics?

Correct. The key advantages of eBGP with per-switch ASNs are: (1) no OSPF area design needed since each switch is its own AS, (2) no iBGP full-mesh or route reflectors since eBGP naturally propagates routes, (3) AS-path provides loop prevention, and (4) route-maps and communities enable per-hop policy control. The "bestpath as-path multipath-relax" knob enables ECMP even when paths traverse different ASNs. BFD is used alongside BGP to achieve sub-second failure detection, compensating for BGP's inherently slower convergence.

Incorrect. eBGP actually converges slower than OSPF by default (answer A is wrong). OSPF can be used as a VXLAN underlay (answer C is wrong). BGP does maintain per-prefix state (answer D is misleading). The correct answer is B: the combination of no area design, no full-mesh requirement, AS-path loop prevention, and rich policy control makes eBGP with private ASNs the simplest and most scalable approach for large datacenter fabrics.

Question 3: What is the "elephant flow" problem with ECMP in GPU training clusters, and how does NVIDIA's InfiniBand Quantum-2 address it?

Correct. NCCL generates a small number of very large, persistent flows during collective operations like AllReduce. With standard ECMP 5-tuple hashing, the birthday problem makes hash collisions likely even with moderate flow counts: with 32 spine paths and 8 flows, there is approximately a 75% chance of at least one collision. A collision means two elephant flows share one spine link (creating congestion) while another spine link sits idle. NVIDIA Quantum-2 switches implement adaptive routing at the hardware level, inspecting real-time port utilization and dynamically selecting the least-congested path for each packet. This per-packet decision eliminates hash collision hotspots and achieves near-theoretical bisection bandwidth.

Incorrect. Elephant flows are not about MTU/fragmentation (A) or short bursts (B) or multicast (D). They are large, long-lived data flows from NCCL collectives that persist for the duration of gradient synchronization. The problem is that ECMP's static 5-tuple hash can map multiple such flows to the same spine link, creating a hotspot. Quantum-2's adaptive routing solves this by making per-packet routing decisions based on real-time congestion, not static hashing.
