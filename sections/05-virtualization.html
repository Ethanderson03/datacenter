GPU Virtualization & Multi-Tenancy

From QEMU/KVM fundamentals through VFIO passthrough, SR-IOV, MIG partitioning, and cloud-native GPU orchestration for HPC and AI infrastructure.

1. QEMU/KVM Fundamentals

KVM — Kernel-based Virtual Machine

KVM is a type-1 (bare-metal) hypervisor implemented as a loadable Linux kernel module. When loaded, it turns the Linux kernel itself into a hypervisor. KVM exposes the /dev/kvm character device, which userspace programs use via ioctl() calls to create and manage virtual machines. KVM handles CPU virtualization (ring-based privilege separation using Intel VT-x VMCS / AMD-V VMCB structures) and memory virtualization (Extended Page Tables / Nested Page Tables for hardware-assisted guest-physical to host-physical address translation).

KVM operates by placing the host kernel in VMX root mode and guest code in VMX non-root mode. Sensitive instructions trigger a VM exit back to the host, where KVM handles the event and resumes the guest via VM entry. Modern hardware virtualization extensions minimize the frequency of VM exits, yielding near-native CPU performance.

QEMU — Quick EMUlator

QEMU is a userspace emulator and virtualizer. Running standalone, it uses dynamic binary translation (Tiny Code Generator / TCG) to emulate foreign CPU architectures. When paired with KVM, QEMU delegates CPU and memory virtualization to the kernel module and focuses on device emulation: disks (virtio-blk, virtio-scsi), network (virtio-net with vhost-net kernel acceleration), display (virtio-gpu, QXL), and other peripherals.

The virtio paravirtualized device framework avoids the overhead of emulating real hardware by using a shared-memory transport (virtqueues) between guest driver and host backend. vhost-net moves the virtio-net datapath into the kernel, and vhost-user moves it into a userspace process (e.g., DPDK/OVS) for even higher throughput.

VM Lifecycle

A virtual machine managed by libvirt follows a well-defined state machine:

define (persistent XML in /etc/libvirt/qemu/) → start (QEMU process launched, guest boots) → pause (vCPUs halted, memory preserved) → resume (vCPUs restarted) → shutdown (ACPI power button sent to guest) → destroy (QEMU process killed immediately). VMs can also be saved (memory snapshot to disk) and restored, or live-migrated to another host with minimal downtime (<100 ms for memory-converging migration).

QEMU Command Line — GPU Passthrough VM

The equivalent direct QEMU invocation (what libvirt generates under the hood):

# Launch a Q35-machine VM with KVM, 64 GB RAM, 16 vCPUs,
# one VFIO-passthrough GPU, and a vhost-accelerated virtio-net NIC.

qemu-system-x86_64 \
  -enable-kvm \
  -machine q35,accel=kvm,kernel-irqchip=split \
  -cpu host \
  -m 64G \
  -smp 16,sockets=1,cores=8,threads=2 \
  -drive file=/var/lib/libvirt/images/gpu-worker-01.qcow2,format=qcow2,if=none,id=disk0,cache=none,aio=native \
  -device virtio-scsi-pci,id=scsi0 \
  -device scsi-hd,drive=disk0,bus=scsi0.0 \
  -device vfio-pci,host=41:00.0,multifunction=on \
  -netdev tap,id=net0,vhost=on,queues=8 \
  -device virtio-net-pci,netdev=net0,mq=on,vectors=18 \
  -nographic \
  -monitor unix:/tmp/qemu-monitor.sock,server,nowait
Why Q35? The Q35 machine type emulates an Intel ICH9 chipset with native PCIe topology instead of PCI. This is required for proper IOMMU group separation when doing multi-device passthrough, and it supports PCIe-native hotplug. Always use Q35 for GPU passthrough workloads.

2. GPU Passthrough with VFIO

VFIO — Virtual Function I/O

VFIO is a Linux kernel framework that provides safe, userspace-driven device access with full IOMMU protection. Unlike the legacy pci-stub / UIO approach, VFIO enforces DMA isolation via the IOMMU, preventing a rogue VM from performing DMA attacks against host memory or other guests. VFIO exposes devices through /dev/vfio/ container and group file descriptors, which QEMU opens to map device BARs (Base Address Registers) and configure interrupts (MSI/MSI-X) for the guest.

IOMMU Groups

An IOMMU group is the smallest set of devices that the IOMMU can isolate from all other devices. All devices within a group share the same IOMMU domain and can DMA to each other's memory — therefore, all devices in a group must be passed to the same VM (or all remain on the host). This is dictated by the hardware topology and ACS (Access Control Services) capability of PCIe bridges.

In well-designed server platforms (DGX, HGX, Supermicro GPU servers), each GPU typically resides in its own IOMMU group because the root ports and PCIe switches implement ACS. Consumer motherboards often lump multiple devices into one group, requiring the pcie_acs_override kernel patch (not recommended for production).

Step-by-Step GPU Passthrough

Complete procedure for passing an NVIDIA GPU to a QEMU/KVM virtual machine:

# Add kernel parameters via GRUB
# For Intel:
sudo sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="/GRUB_CMDLINE_LINUX_DEFAULT="intel_iommu=on iommu=pt /' /etc/default/grub
# For AMD:
sudo sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="/GRUB_CMDLINE_LINUX_DEFAULT="amd_iommu=on iommu=pt /' /etc/default/grub

sudo update-grub && sudo reboot

# Verify IOMMU is active after reboot
dmesg | grep -i -e DMAR -e IOMMU
# Should see: "DMAR: IOMMU enabled" or "AMD-Vi: AMD IOMMUv2 loaded"

# Verify via /proc/cmdline
cat /proc/cmdline
# ... intel_iommu=on iommu=pt ...

The iommu=pt (passthrough) parameter tells the kernel to not put host devices behind IOMMU translation unless explicitly needed (by VFIO). This avoids DMA performance overhead for devices that remain on the host.

lspci -nn | grep -i nvidia

# Find the IOMMU group for this device

# 0000:41:00.0  (GPU)
# If other devices appear here, they must ALL be passed through together

# Script to enumerate all IOMMU groups
for d in /sys/kernel/iommu_groups/*/devices/*; do
  n=$(basename $(dirname $(dirname "$d")))
  echo "IOMMU Group $n: $(lspci -nns ${d##*/})"
done | sort -t: -k1 -n
# Check current driver binding
lspci -ks 41:00.0
# Kernel driver in use: nvidia
# Kernel modules: nvidia

# Get vendor:device ID for VFIO binding
lspci -ns 41:00.0
# 41:00.0 0302: 10de:2330 (rev a1)
# vendor=10de (NVIDIA), device=2330 (H100)

# Method 1: Runtime unbind/rebind (immediate, no reboot)
# Unbind from nvidia driver
echo 0000:41:00.0 | sudo tee /sys/bus/pci/devices/0000:41:00.0/driver/unbind

# Load vfio-pci module
sudo modprobe vfio-pci

# Bind to vfio-pci
echo 10de 2330 | sudo tee /sys/bus/pci/drivers/vfio-pci/new_id

# Verify
lspci -ks 41:00.0
# Kernel driver in use: vfio-pci

# Method 2: Persistent via modprobe config (survives reboot)
echo "options vfio-pci ids=10de:2330" | sudo tee /etc/modprobe.d/vfio.conf
echo -e "vfio-pci\nvfio\nvfio_iommu_type1\nvfio_virqfd" | sudo tee /etc/modules-load.d/vfio.conf

# Blacklist nvidia for this GPU if sharing system with other GPUs
# (use driver binding via BDF instead for selective passthrough)
sudo update-initramfs -u && sudo reboot
# Launch with QEMU directly
sudo qemu-system-x86_64 \
  -enable-kvm \
  -machine q35,accel=kvm,kernel-irqchip=split \
  -cpu host \
  -m 64G \
  -smp 16,sockets=1,cores=8,threads=2 \
  -drive file=disk.qcow2,format=qcow2,if=virtio \
  -device vfio-pci,host=41:00.0 \
  -netdev tap,id=net0,vhost=on \
  -device virtio-net-pci,netdev=net0 \
  -vnc :0

# Or use libvirt (recommended for production)
virsh start gpu-worker-01

# Inside the guest, the GPU appears natively:
nvidia-smi
# Shows H100 80GB as if bare metal
ACS (Access Control Services): For proper IOMMU isolation in multi-GPU servers, the PCIe root ports and downstream switches must support ACS. Without ACS, the IOMMU cannot distinguish peer-to-peer DMA between devices behind the same switch, forcing them into one IOMMU group. Server-grade platforms (DGX H100, HGX, Supermicro 4124GS) have proper ACS support. Verify with: lspci -vvv | grep -i "access control"

3. SR-IOV (Single Root I/O Virtualization)

What is SR-IOV?

SR-IOV is a PCIe specification that allows a single physical device to present itself as multiple virtual devices at the hardware level. The physical device exposes a Physical Function (PF) — the full-featured PCIe function with all configuration capabilities — and one or more Virtual Functions (VFs), which are lightweight PCIe functions with their own BARs, MSI-X vectors, and queue sets, but with reduced configuration space. Each VF appears as an independent PCIe device to the OS and can be assigned to a different VM via VFIO, achieving near-native I/O performance without software-based virtio emulation overhead.

PF vs VF Comparison

Attribute Physical Function (PF) Virtual Function (VF)
PCIe Config Space Full (Type 0, 4 KB standard + extended) Reduced (lightweight, limited caps)
BAR Resources All device BARs Subset (own queue memory region)
Driver on Host Full driver (e.g., mlx5_core) Can be bound to vfio-pci for passthrough
Management Controls VF creation, MAC assignment, VLAN, rate limit No management control; uses assigned resources
Typical Count 1 per port Up to 128-1024 per PF (device-dependent)

NIC SR-IOV with ConnectX-7

SR-IOV is extremely common for network adapters. The NVIDIA/Mellanox ConnectX-7 supports up to 1024 VFs per port, making it ideal for high-density VM hosting. Each VF gets a hardware queue pair, its own MAC address, optional VLAN tag, and configurable bandwidth limits — all enforced in hardware. 

# 1. Verify SR-IOV capability
lspci -vs 31:00.0 | grep -i "sr-iov"
#   Capabilities: [180] Single Root I/O Virtualization (SR-IOV)

# Total VFs supported
cat /sys/class/net/ens1f0/device/sriov_totalvfs
# 1024

# 2. Create 8 Virtual Functions
echo 8 | sudo tee /sys/class/net/ens1f0/device/sriov_numvfs

# Verify VFs appeared as PCIe devices
lspci | grep -i mellanox
# 31:00.0 Ethernet controller: Mellanox ConnectX-7 (PF)
# 31:00.1 Ethernet controller: Mellanox ConnectX-7 Virtual Function
# 31:00.2 Ethernet controller: Mellanox ConnectX-7 Virtual Function
# ... (8 VFs total)

# VFs also appear as network interfaces
ip link show ens1f0
# ... vf 0 MAC 00:00:00:00:00:00, vf 1 MAC 00:00:00:00:00:00 ...

# 3. Configure each VF (MAC, VLAN, bandwidth, spoofchk)
sudo ip link set ens1f0 vf 0 mac 52:54:00:aa:bb:01 vlan 100 rate 50000 spoofchk on
sudo ip link set ens1f0 vf 1 mac 52:54:00:aa:bb:02 vlan 200 rate 25000 spoofchk on
sudo ip link set ens1f0 vf 2 mac 52:54:00:aa:bb:03 vlan 100 rate 100000 spoofchk on
# rate is in Mbps (50000 = 50 Gbps)

# 4. Verify VF configuration
ip link show ens1f0
# vf 0: MAC 52:54:00:aa:bb:01, vlan 100, rate 50000Mbps, spoofchk on
# vf 1: MAC 52:54:00:aa:bb:02, vlan 200, rate 25000Mbps, spoofchk on

# 5. Bind VF to vfio-pci for passthrough to a VM
# Find VF's PCI BDF
VF0_BDF=$(readlink -f /sys/class/net/ens1f0/device/virtfn0 | xargs basename)
echo "$VF0_BDF"  # e.g., 0000:31:00.1

# Unbind from mlx5_core
echo "$VF0_BDF" | sudo tee /sys/bus/pci/drivers/mlx5_core/unbind

# Bind to vfio-pci
echo "vfio-pci" | sudo tee /sys/bus/pci/devices/$VF0_BDF/driver_override
echo "$VF0_BDF" | sudo tee /sys/bus/pci/drivers_probe

# 6. Pass VF to VM
qemu-system-x86_64 -enable-kvm -m 16G -smp 4 \
  -device vfio-pci,host=31:00.1 \
  -drive file=vm-disk.qcow2,format=qcow2,if=virtio \
  ...

# 7. Make VFs persistent across reboot (systemd-based)
# /etc/systemd/system/sriov-vfs.service
# [Unit]
# Description=Create SR-IOV VFs
# After=network-pre.target
# Before=network.target
#
# [Service]
# Type=oneshot
# ExecStart=/bin/bash -c 'echo 8 > /sys/class/net/ens1f0/device/sriov_numvfs'
# RemainAfterExit=yes
#
# [Install]
# WantedBy=multi-user.target

GPU SR-IOV

GPU SR-IOV support is limited compared to NIC SR-IOV. NVIDIA does not expose native SR-IOV on most GPU products. Instead, GPU sharing is achieved through:

  • NVIDIA vGPU (GRID): A mediated passthrough (mdev) framework where the NVIDIA host driver multiplexes GPU access across VMs via time-slicing or MIG-backed profiles. This uses the mdev (mediated device) framework rather than true SR-IOV VFs.
  • MIG + VFIO: On A100/H100, MIG instances can be exposed as individual compute instances and passed to VMs. This provides true hardware isolation (each instance has dedicated SMs and memory) without SR-IOV.
  • Intel Data Center GPU Flex (Arctic Sound-M): One of the few GPUs with true SR-IOV support, exposing up to 62 VFs per PF for cloud gaming and media workloads.
Best Practice for HPC/AI: In Superpod deployments, NIC SR-IOV (ConnectX-7 VFs) is heavily used to provide each VM or container with a dedicated high-performance network path. GPU access is typically full passthrough (VFIO) or MIG partitioning rather than SR-IOV.

4. NVIDIA MIG (Multi-Instance GPU)

What is MIG?

Multi-Instance GPU (MIG) is a hardware-level GPU partitioning technology available on NVIDIA A100, A30, and H100 GPUs (Ampere and Hopper architectures). MIG divides a single physical GPU into up to 7 isolated GPU instances, each with its own dedicated set of:

  • Streaming Multiprocessors (SMs) — compute engines for CUDA/Tensor Core operations
  • Memory controllers and corresponding HBM memory slice — guaranteed memory bandwidth
  • L2 cache partition — no cache interference between instances
  • Copy engines (CE) and decoders (NVDEC/NVJPG on supported profiles)
  • Unique GPU instance ID — each instance is addressable as a separate GPU device

MIG provides true hardware-level isolation: a workload running on one MIG instance cannot observe or interfere with workloads on other instances, including memory access patterns, cache behavior, and error conditions (ECC errors are scoped to the instance). This is fundamentally different from time-slicing, where workloads share all resources and can experience interference.

MIG Availability: MIG is only available on data center GPUs with Ampere or newer architecture: A100 (40GB/80GB), A30 (24GB), H100 (80GB), and H200 (141GB). It is not available on consumer GPUs (RTX 4090), professional GPUs (RTX 6000 Ada), or older data center GPUs (V100, T4). MIG requires the GPU to be in MIG mode, which requires a GPU reset or reboot to enable/disable.

MIG + Kubernetes

In Kubernetes, the NVIDIA GPU Operator and k8s-device-plugin can automatically discover MIG instances and expose them as schedulable resources. Each MIG slice appears as a distinct resource type in the node's capacity: 

# Pod spec requesting a specific MIG profile
apiVersion: v1
kind: Pod
metadata:
  name: inference-small
spec:
  containers:
  - name: model-server
    image: nvcr.io/nvidia/tritonserver:24.01-py3
    resources:
      limits:
        nvidia.com/mig-1g.10gb: 1    # Request one 1g.10gb MIG slice
---
# Node capacity (kubectl describe node gpu-node-01):
# Capacity:
#   nvidia.com/mig-1g.10gb:  7
#   nvidia.com/mig-2g.20gb:  0
#   nvidia.com/mig-3g.40gb:  0
#   nvidia.com/mig-7g.80gb:  0

The device plugin supports three MIG strategies configured via the mig.strategy Helm value:

  • none: MIG disabled; GPUs exposed as nvidia.com/gpu
  • single: All GPUs use the same MIG profile; resources named nvidia.com/gpu
  • mixed: Different MIG profiles per GPU; resources named nvidia.com/mig-{profile}

5. NVIDIA vGPU (GRID)

Software-Based GPU Virtualization

NVIDIA vGPU (branded under NVIDIA AI Enterprise and previously NVIDIA GRID) enables multiple virtual machines to share a single physical GPU. The NVIDIA Virtual GPU Manager runs on the hypervisor host (KVM, VMware ESXi, Citrix Hypervisor) and mediates access to the GPU hardware. Each VM runs a standard NVIDIA guest driver that communicates with the vGPU Manager through a paravirtualized channel.

vGPU uses the Linux kernel's mdev (mediated device) framework on KVM hosts. The vGPU Manager creates mdev instances (e.g., /sys/class/mdev_bus/0000:41:00.0/{uuid}) that are passed to QEMU as -device vfio-pci,sysfsdev=/sys/.... Each mdev instance represents a virtual GPU with a specific profile defining its framebuffer allocation and time-slice scheduling weight. 

# ===== Install vGPU Manager on KVM Host =====

Time-Slicing vs MIG-Backed vGPU

On GPUs that support MIG (A100, H100), vGPU can operate in two modes:

  • Time-sliced vGPU: The vGPU scheduler rapidly context-switches between VMs on a fixed time quantum (default: best-effort, or configurable equal-share / fixed-share). All VMs share the full SM array and memory bandwidth but get exclusive access during their time slice. Lower isolation, higher flexibility in profile sizes.
  • MIG-backed vGPU: Each vGPU instance is backed by a MIG GPU Instance. The VM gets dedicated SMs, memory, and L2 cache with hardware isolation. Provides the isolation benefits of MIG with the management benefits of the vGPU framework (licensing, live migration, monitoring via NVIDIA DCGM). 
# MIG-backed vGPU: first create MIG instances, then vGPU types map to them
sudo nvidia-smi mig -i 0 -cgi 19,19,19,19,19,19,19 -C

# vGPU types for MIG-backed mode appear with MIG profile in the name
ls /sys/class/mdev_bus/0000:41:00.0/mdev_supported_types/ | grep MIG
# nvidia-710   (H100-1g.10gb-C, compute profile for 1g.10gb MIG)
# nvidia-711   (H100-2g.20gb-C, compute profile for 2g.20gb MIG)

cat /sys/class/mdev_bus/0000:41:00.0/mdev_supported_types/nvidia-710/name
# NVIDIA H100-1g.10gb-C
Licensing: NVIDIA vGPU requires a license from the NVIDIA Licensing Server (NLS) or a Delegated License Service (DLS) appliance on-premises. Each concurrent vGPU session consumes one license. License types include NVIDIA AI Enterprise (full stack: vGPU + containers + frameworks + support) and vWS (Quadro Virtual Workstation). Without a valid license, vGPU operates in a degraded mode with reduced framebuffer and frame rate.

6. Cloud-Native Virtualization

KubeVirt — VMs as Kubernetes Pods

KubeVirt extends Kubernetes with a CRD (Custom Resource Definition) called VirtualMachine that allows traditional virtual machines to be managed alongside containers using the Kubernetes API. Under the hood, KubeVirt runs QEMU/KVM inside a pod (the virt-launcher pod), wrapping the VM lifecycle in Kubernetes primitives: the VM is scheduled like a pod, its storage is provided via PVCs, and its network via CNI plugins.

For GPU workloads, KubeVirt supports both GPU passthrough (VFIO) and MIG device assignment to VMs. The NVIDIA GPU Operator detects KubeVirt and configures the nodes appropriately, including loading VFIO drivers and configuring IOMMU for devices designated for VM passthrough.

When to Use Each Approach

Use Bare Metal When:
  • Maximum GPU performance is critical (large-scale training)
  • Multi-node NVLink/NVSwitch topologies (DGX SuperPOD)
  • Single-tenant, dedicated infrastructure
  • GPUDirect RDMA with no abstraction overhead
Use VMs (KVM + VFIO) When:
  • Strong multi-tenant isolation required (cloud providers)
  • Different OS or driver versions per tenant
  • Compliance requirements mandate VM-level separation
  • Legacy workloads need a full OS environment
Use Containers When:
  • Rapid scaling and deployment (inference serving)
  • CI/CD pipelines and dev/test environments
  • Kubernetes-native orchestration desired
  • High density with MIG or time-slicing
Hybrid Pattern: In production Superpod deployments, a common hybrid pattern is KubeVirt + NVIDIA GPU Operator: training jobs run as bare-metal Kubernetes pods on nodes with full GPU passthrough via the NVIDIA device plugin, while inference and multi-tenant workloads run inside KubeVirt VMs with MIG-backed vGPU slices — all managed by the same Kubernetes control plane. This provides operational consistency while matching the isolation model to each workload's requirements.

Knowledge Check

Question 1

An administrator wants to pass an NVIDIA H100 GPU to a QEMU/KVM virtual machine. They have enabled IOMMU and identified the GPU at PCI address 41:00.0, but when they check the IOMMU group, they find another device (an audio controller at 41:00.1) in the same group. What must they do?

Correct answer: B. All devices within an IOMMU group must be either assigned to the same VM or all remain on the host. The IOMMU group represents the smallest unit of isolation the hardware can enforce. NVIDIA GPUs commonly have an associated HDA (audio) controller as a separate PCIe function on the same device (multi-function device), and both must be passed through together. You cannot manually reassign IOMMU groups through sysfs — they are determined by hardware topology and ACS capability.

Question 2

A Kubernetes cluster operator has 4 NVIDIA H100 GPUs and needs to serve 20 concurrent small inference models, each requiring approximately 8 GB of GPU memory and 14 SMs. The models must have guaranteed performance isolation. Which approach is most appropriate?

Correct answer: B. MIG with the 1g.10gb profile on H100 provides 7 instances per GPU (28 total across 4 GPUs), exceeding the 20-pod requirement. Each 1g.10gb instance provides 16 SMs and 10 GB of HBM3 memory, meeting the workload's needs of ~14 SMs and ~8 GB. MIG provides hardware-level isolation (dedicated SMs, memory, and L2 cache), satisfying the guaranteed performance isolation requirement. Time-slicing (option A) does not provide isolation. vGPU (option C) requires additional licensing and has higher overhead for container-native workloads. Option D adds unnecessary VM overhead.

Question 3

What is the primary difference between SR-IOV Virtual Functions and NVIDIA vGPU mdev (mediated) devices?

Correct answer: B. SR-IOV Virtual Functions are true hardware-level PCIe functions created by the device itself. Each VF has its own PCIe configuration space, BAR registers, MSI-X vectors, and hardware queue pairs. The device's internal logic routes I/O to the correct VF without host software involvement on the data path.

In contrast, mdev (mediated devices) are a software framework where the host driver (e.g., NVIDIA vGPU Manager) intercepts and multiplexes guest access to a single physical device. The mdev device presented to the VM looks like a VFIO device, but behind the scenes, the host driver manages scheduling, memory partitioning, and context switching. This is why mdev-based vGPU requires a special host driver and license, while SR-IOV VFs work with the standard in-tree kernel driver.
