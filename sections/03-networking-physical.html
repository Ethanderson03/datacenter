03

Physical Networking & RDMA Fabric

InfiniBand, RoCE, GPUDirect, and the cabling that binds a GPU Superpod together

1. InfiniBand Deep Dive

What is InfiniBand? InfiniBand (IB) is a lossless, switched-fabric interconnect designed from the ground up for Remote Direct Memory Access (RDMA). Unlike Ethernet, which was originally a shared-medium broadcast protocol adapted over decades for datacenter use, InfiniBand was purpose-built for low-latency, high-bandwidth, credit-based communication between compute nodes. It provides kernel-bypass data transfers where application memory is registered with the NIC (called an HCA—Host Channel Adapter in IB terminology) and data moves directly between application buffers across the network without any CPU intervention on the data path.

InfiniBand Speed Generations

InfiniBand defines per-lane signaling rates. Links are assembled from 1x, 4x, or 12x lanes. In HPC/AI, 4x links are standard. Each generation roughly doubles the per-lane rate of the previous.

Link width notation: You will see ports labeled as 4X NDR or 4X HDR. This means 4 lanes at that generation's per-lane rate. A 4X NDR port delivers 4 × 100 Gb/s = 400 Gb/s signaling rate. 12x links exist in the spec but are extremely rare in modern deployments—4x is the universal standard for compute interconnects.

Subnet Manager (SM)

An InfiniBand fabric requires a Subnet Manager to function. Unlike Ethernet where switches independently learn MAC addresses and forward frames, IB is centrally managed. The SM discovers the fabric topology, assigns Local Identifiers (LIDs) to every port, computes forwarding tables (linear forwarding tables in each switch), and pushes them out. Without a running SM, no traffic flows.

OpenSM

The open-source reference implementation of the IB Subnet Manager, part of the OFED (OpenFabrics Enterprise Distribution) stack. OpenSM runs as a daemon on a management node (or on a switch with a management CPU). It supports multiple routing algorithms:

  • Min-Hop: Default algorithm; routes based on minimum hop count
  • Up/Down: Prevents routing loops in non-pure-fat-tree topologies by assigning direction to links
  • Fat-Tree: Optimized for fat-tree topologies; gives balanced dispersive routing across uplinks
  • DFSSSP: Deadlock-Free Single-Source Shortest Path for arbitrary topologies

OpenSM typically recalculates routes on topology changes (link up/down events). The -R flag sets the routing engine. Configuration lives in /etc/opensm/opensm.conf.

NVIDIA UFM (Unified Fabric Manager)

UFM is NVIDIA's commercial IB fabric management platform. It provides an embedded SM plus advanced features far beyond OpenSM:

  • Real-time telemetry: Per-port counters, error rates, congestion detection
  • Adaptive routing: Dynamically adjusts paths based on congestion signals
  • SHARP aggregation management: Configures in-network computing trees
  • Health monitoring: Detects BER (Bit Error Rate) degradation, flapping links, and cable faults before they cause job failures
  • Event-driven automation: REST API and webhook integrations for alerting/remediation
  • Topology visualization: Web GUI with full fabric map

In production Superpod deployments, UFM is essentially mandatory. It runs on a dedicated management node and communicates with the IB fabric via in-band management datagrams (MADs).

RDMA Core Concepts

RDMA communication is built on three fundamental kernel-bypass objects:

Queue Pairs (QP)

A QP is the fundamental communication endpoint. Each QP consists of a Send Queue (SQ) and a Receive Queue (RQ). Applications post Work Requests (WRs) to these queues. The HCA processes WRs asynchronously without kernel involvement. QP types include:

  • RC (Reliable Connected): Point-to-point, guaranteed delivery, in-order. Used for most HPC/AI traffic (NCCL collective operations, MPI point-to-point).
  • UC (Unreliable Connected): Point-to-point, no ACK/retransmit. Rarely used.
  • UD (Unreliable Datagram): One-to-any, message size limited to MTU. Used for management traffic, service discovery.
  • XRC (Extended Reliable Connected): Reduces QP count by sharing receive queues. Important for large-scale jobs where N×N QPs would exhaust HCA resources.

Completion Queues (CQ)

When the HCA finishes processing a WR, it posts a Completion Queue Entry (CQE) to the associated CQ. Applications poll the CQ (low-latency) or block on an event (lower CPU cost). A single CQ can serve multiple QPs. Completion status tells you if the operation succeeded or reports error codes (e.g., IBV_WC_REM_ACCESS_ERR for remote memory access violations).

Memory Regions (MR)

Before any RDMA transfer, memory must be registered with the HCA via ibv_reg_mr(). Registration pins the pages in physical memory (preventing swapout), creates a virtual-to-physical mapping in the HCA's page tables, and returns a pair of keys:

  • lkey (Local Key): Used by the local side to reference this memory in WRs
  • rkey (Remote Key): Shared with remote side to allow RDMA Read/Write into this region

Memory registration is expensive (kernel call, TLB shootdowns). Production codes pre-register buffers. Libraries like UCX and NCCL maintain registration caches to amortize cost.

The Verbs API (provided by libibverbs) is the standard programming interface for RDMA. It is intentionally low-level, giving applications direct control over HCA resources. The typical workflow:

  1. ibv_get_device_list() — enumerate HCAs in the system
  2. ibv_open_device() — open a device context
  3. ibv_alloc_pd() — create a Protection Domain (isolation boundary)
  4. ibv_reg_mr() — register memory
  5. ibv_create_cq() — create completion queue
  6. ibv_create_qp() — create queue pair
  7. Transition QP through states: RESET → INIT → RTR → RTS
  8. ibv_post_send() / ibv_post_recv() — post work requests
  9. ibv_poll_cq() — poll for completions

RDMA Operation Types

Operation Initiator Remote CPU Involved? Description
Send / Receive Sender Yes (must pre-post receive) Two-sided operation. Receiver must have a Receive WR posted. Data lands in receiver's buffer. Like traditional message passing.
RDMA Write Writer No One-sided. Writer specifies remote address + rkey. Data is placed directly into remote memory. Remote CPU is unaware. Used heavily by NCCL for GPU-to-GPU transfers.
RDMA Read Reader No One-sided. Reader pulls data from remote memory. Remote CPU is unaware. Useful for polling remote status or fetching shared state.
Atomic (CAS, Fetch&Add) Initiator No Performs atomic read-modify-write on remote memory. Compare-and-Swap (CAS) and Fetch-and-Add are the two standard atomics. Executed by the remote HCA, not the remote CPU.

InfiniBand uses Partition Keys (PKeys) to segment traffic on a shared fabric, analogous to VLANs on Ethernet. Each PKey is a 16-bit value. The high bit indicates full vs. limited membership:

  • Full member (0x8xxx): Can communicate with any member of the partition (full or limited)
  • Limited member (0x0xxx): Can only communicate with full members, not other limited members

Every IB port has a PKey table (up to 128 entries on ConnectX HCAs). The default partition is PKey 0x7FFF (limited) or 0xFFFF (full), and all ports must be members. The SM configures PKey tables based on the partitions.conf file.

Use cases in Superpod:

  • Separate storage traffic (Lustre/GPFS) from compute (MPI/NCCL) traffic
  • Isolate tenants in multi-tenant clusters
  • Restrict management traffic to admin partitions

Key difference from VLANs: PKeys provide membership enforcement at the HCA level. A node cannot forge PKey membership—the SM controls the PKey table, and the HCA enforces it in hardware.

InfiniBand vs Ethernet: Design Philosophy

Credit-Based Flow Control

InfiniBand is lossless by design. Every link uses credit-based flow control: a sender can only transmit when it has credits from the receiver, meaning the receiver has buffer space. This is fundamentally different from Ethernet, where packets are sent optimistically and may be dropped if a switch buffer overflows.

  • InfiniBand: Sender checks credits → transmits only if receiver has buffer space → zero packet drops on a healthy fabric. Congestion manifests as back-pressure (reduced throughput), not drops.
  • Ethernet: Sender transmits immediately → switch may drop if buffer overflows → relies on TCP retransmits for reliability. Drops are expected and handled by upper layers.

This is why InfiniBand achieves consistently lower latency: there is no retransmission overhead, no TCP slow-start, and the fabric provides hardware-level reliability guarantees. For RDMA workloads where a single dropped packet can stall a GPU collective operation across thousands of GPUs, this lossless guarantee is critical.

2. RoCE (RDMA over Converged Ethernet)

Why RoCE? RoCE enables RDMA semantics (kernel bypass, zero-copy, low latency) over standard Ethernet infrastructure. This lets organizations leverage existing Ethernet switches, cabling, and operational expertise while getting most of the performance benefits of RDMA. It is especially compelling when InfiniBand's specialized switch infrastructure is not available or when Ethernet is already deployed at scale.

RoCEv1 vs RoCEv2

Feature RoCEv1 RoCEv2
Encapsulation IB transport over Ethernet (EtherType 0x8915) IB transport over UDP/IP over Ethernet
Routing L2 only (same VLAN / broadcast domain) L3 routable (crosses subnets)
UDP Port N/A Destination port 4791; source port is a hash of QP/flow for ECMP
ECMP Support No (L2 only) Yes—switches hash on UDP source port for multi-path load balancing
Deployment Legacy, rarely used Industry standard for all modern RDMA-over-Ethernet deployments

In practice, RoCEv2 is the only version deployed today. The UDP encapsulation solved the critical limitation of RoCEv1 (L2-only), enabling ECMP across multi-tier leaf-spine fabrics.

Making Ethernet Lossless: PFC, ECN, and DCQCN

RDMA assumes a lossless transport. A single dropped packet causes an RDMA operation to fail at the transport level (unlike TCP, which retransmits transparently). Ethernet is inherently lossy, so achieving lossless behavior requires careful configuration of several mechanisms working together:

Priority Flow Control (PFC) — IEEE 802.1Qbb

PFC extends Ethernet pause frames to work on a per-priority basis (using 802.1p priority bits in the VLAN tag). When a switch input buffer for a given priority exceeds a threshold, it sends a PFC pause frame to the upstream sender for that priority only, leaving other priorities unaffected.

  • RoCEv2 traffic is typically assigned to priority 3 (configurable via DSCP-to-priority mapping)
  • PFC is enabled only for the RoCE priority class—other traffic classes (management, best-effort) remain lossy
  • PFC deadlock risk: If a circular buffer dependency forms across switches, PFC pause frames can cascade and deadlock the entire fabric. This is a real operational risk in leaf-spine topologies and requires careful buffer tuning and sometimes PFC watchdog timers.

PFC configuration example (NVIDIA Spectrum switch, Cumulus Linux):

# Enable PFC on priority 3 only
nv set qos pfc priority 3 enable on
nv set interface swp1-64 qos pfc priority 3 enable on

# Set PFC buffer thresholds
nv set qos pfc priority 3 xoff-threshold 80000
nv set qos pfc priority 3 xon-threshold 40000

Explicit Congestion Notification (ECN) — RFC 3168

ECN is the early-warning system that prevents PFC from ever firing under normal conditions. When a switch buffer occupancy exceeds a configured threshold (but before it is full enough to trigger PFC), the switch marks packets by setting the ECN bits in the IP header to CE (Congestion Experienced, binary 11).

  • ECN operates at L3 (IP header) and works per-packet
  • The receiver detects the CE marking and signals congestion back to the sender via a CNP (Congestion Notification Packet)
  • The sender reduces its injection rate in response
  • ECN thresholds must be set lower than PFC thresholds so that rate reduction kicks in before any pause is needed

DCQCN (Data Center QoS Congestion Notification)

DCQCN is the end-to-end congestion control algorithm used in RoCEv2 networks. It combines ECN marking at the switches with rate-limiting at the sender NIC. The algorithm:

  1. Switch marks packets with ECN when queue depth exceeds threshold (RED/WRED-style probabilistic marking)
  2. Receiver generates CNP (Congestion Notification Packet) back to sender when it sees CE-marked packets
  3. Sender reduces rate by a configurable factor (alpha) upon receiving CNP
  4. Sender gradually recovers rate using timer-based increase, byte-counter-based increase, or a combination (similar to TCP's AIMD but tuned for datacenter RTTs of ~10µs)

DCQCN parameters on ConnectX NICs are tuned via mlxreg or mlnx_qos. Key knobs include the rate-reduction factor, recovery timer interval, and the number of CNPs before rate cut. Mistuning causes either excessive PFC (thresholds too high) or unnecessary throughput reduction (thresholds too low).

InfiniBand vs RoCE: Side-by-Side Comparison

InfiniBand Characteristics

  • Latency: ~0.6–1.0 µs end-to-end (measured via ib_write_lat)
  • Lossless by design: Credit-based flow control at every hop. No configuration needed—it just works.
  • Subnet Manager: Central control plane computes all routes. Enables advanced features like adaptive routing, QoS SLs (Service Levels), and virtual lanes (VLs).
  • Fabric management: UFM provides deep telemetry, health monitoring, and topology awareness. SM detects and reroutes around failed links automatically.
  • In-network computing: NVIDIA SHARP performs MPI_Allreduce, MPI_Barrier, and other collectives inside the switch network, reducing traffic and latency for AI training.
  • Partitioning: PKeys enforced in hardware at the HCA for strong multi-tenant isolation.
  • Switches: Requires purpose-built InfiniBand switches (Quantum-2, etc.). These are not interchangeable with Ethernet switches.
  • Cabling: Same physical cables (QSFP-DD, fiber, DAC) but different electrical signaling. IB and Ethernet ports are not cross-compatible.

RoCE (v2) Characteristics

  • Latency: ~2–3 µs end-to-end (additional overhead from Ethernet framing, PFC/ECN processing)
  • Lossless via configuration: Requires PFC + ECN + DCQCN all configured correctly across every switch and NIC. A single misconfigured switch can cause drops and RDMA failures.
  • Routing: Standard IP/Ethernet routing (BGP, OSPF, ECMP). No special control plane needed, but ECMP hashing must be tuned for RoCE flows.
  • Fabric management: Uses standard Ethernet NOS (Cumulus, SONiC, EOS). Monitoring via SNMP, gNMI, streaming telemetry. Less IB-specific insight.
  • No in-network computing: SHARP is InfiniBand-only. RoCE relies entirely on host-side collectives (NCCL tree/ring algorithms).
  • Isolation: Standard Ethernet VLANs, ACLs, VRFs. Weaker than PKey enforcement but well-understood operationally.
  • Switches: Standard Ethernet switches (Spectrum-4, Tomahawk, Trident, etc.). Same switches can carry RoCE, TCP, and management traffic.
  • Operational familiarity: Most network engineers know Ethernet. IB requires specialized skills.

When to use which: NVIDIA Superpod reference architectures specify InfiniBand as the primary compute fabric due to its lossless guarantees, lower latency, adaptive routing, and SHARP in-network computing. RoCE is a strong choice when building on existing Ethernet infrastructure, when the cluster is smaller (<256 GPUs), or when Ethernet is mandated by organizational policy. Many large deployments use both: InfiniBand for GPU-to-GPU compute traffic and Ethernet for storage, management, and out-of-band access.

SHARP (Scalable Hierarchical Aggregation and Reduction Protocol) offloads collective operations from the hosts into the switch network itself. Instead of data traversing the full tree for an allreduce (up to root, then back down), intermediate switches perform partial reductions as data passes through:

  • Allreduce: Each leaf switch aggregates (sums/max/min) data from its connected nodes, sends the partial result up. Spine switches aggregate partials from leaf switches. The final result propagates back down. This reduces network traffic by up to 2x and latency by the depth of the tree.
  • Barrier: Implemented as a zero-byte allreduce. Completes in O(log N) switch hops instead of O(N) host-to-host messages.
  • Supported data types: FP16, BF16, FP32, FP64, INT32, INT64
  • NCCL integration: NCCL 2.12+ supports SHARP via the NCCL_COLLNET_ENABLE=1 environment variable. The SHARP plugin (libnccl-net-sharp.so) handles tree construction and in-network reduction.

SHARP is particularly impactful for large-scale LLM training where allreduce of gradient tensors dominates communication time. Benchmarks show 10–30% training throughput improvement on 256+ GPU jobs.

Adaptive routing on Spectrum-4 dynamically selects the least-congested ECMP next-hop for each packet, rather than using a static hash. Traditional ECMP hashes on the 5-tuple (src IP, dst IP, src port, dst port, protocol) to pick a single path per flow. This can lead to hash collisions where multiple large flows land on the same link while others are idle.

Spectrum-4's adaptive routing monitors real-time egress queue depths across all ECMP members and steers packets to the least-loaded path. This happens per-packet (or per-flowlet), providing much better load balancing for elephant flows common in RDMA/NCCL traffic patterns.

  • Per-packet adaptive routing: Maximum load balancing, but may cause packet reordering. Requires receiver-side reordering support (RoCE NICs handle this in hardware).
  • Flowlet-based adaptive routing: Only changes path during idle gaps in a flow, avoiding reordering. Good middle ground.

This is configured via Cumulus/SONiC as adaptive-routing on ECMP groups, and is one of the key advantages of Spectrum-4 over commodity Ethernet ASICs for RoCE deployments.

MPO/MTP (Multi-fiber Push On / Mechanical Transfer Push-on)

Multi-fiber connectors that bundle 8, 12, 16, or 24 fibers into a single ferrule. Used for parallel optics in high-speed links:

  • MPO-12: 12 fibers. Used for 40GbE (4 Tx + 4 Rx, 4 unused) and 100GbE SR4.
  • MPO-16: 16 fibers. Used for 400GbE SR8 (8 Tx + 8 Rx). Becoming the standard for 400G MMF deployments.
  • MPO-24: 24 fibers. Used for higher-density trunk cables and some 400G variants.
  • Polarity matters: MPO connectors can be Type A (straight), Type B (reversed), or Type C (pair-flipped). Mismatched polarity causes link failures. Always verify polarity type when connecting structured cabling.

LC (Lucent Connector)

Small form-factor duplex connector (2 fibers: Tx + Rx). The standard for SMF connections and SFP-based transceivers. LC connectors use a 1.25 mm ferrule and snap-in latch. At 400G, LC is used with DR4 and FR4 transceivers that use WDM (4 wavelengths over 2 fibers).

CS (Compact SN) Connector

A newer, smaller duplex connector designed for higher faceplate density. Used in some 400G and 800G transceiver designs where LC is too large. Same fiber alignment as LC but ~40% smaller footprint.

Breakout Cables

Breakout cables (also called fan-out or splitter cables) split a single high-speed port into multiple lower-speed ports. They physically separate the individual lane groups in the multi-lane transceiver into separate connectors:

  • 1×400G → 4×100G: Splits a QSFP-DD (8 lanes) into 4× QSFP28 (2 lanes each). Common for connecting 400G spine switches to 100G leaf switches or servers.
  • 1×400G → 2×200G: Splits a QSFP-DD into 2× QSFP56 (4 lanes each). Used for HDR InfiniBand connectivity from NDR switches.
  • 1×200G → 2×100G: Splits a QSFP56 (HDR) into 2× QSFP28 (HDR100). Common in mixed-generation IB fabrics.
  • 1×100G → 4×25G: Splits a QSFP28 into 4× SFP28. Classic breakout for connecting to 25GbE server NICs.

The switch port must support breakout mode in its configuration. Example on Cumulus Linux: nv set interface swp1 breakout 4x100G.

5. GPUDirect Technologies

The fundamental problem: In traditional data transfer, data from the network must be copied into host (CPU) memory, then copied again to GPU memory. For AI training workloads that transfer terabytes of gradient data per minute across thousands of GPUs, this double-copy through CPU memory creates a bottleneck that wastes PCIe bandwidth and adds latency. GPUDirect eliminates these copies.

GPUDirect RDMA: Data Flow Comparison

GPUDirect RDMA

GPUDirect RDMA enables the network adapter (ConnectX HCA) to directly read from and write to GPU memory (HBM) via PCIe peer-to-peer transactions, completely bypassing CPU host memory. This eliminates two memory copies per transfer and removes the CPU from the critical data path.

How it works:
  1. GPU memory is registered as an RDMA Memory Region, and the NIC is given the physical addresses (BAR mapping) of GPU HBM.
  2. When a remote RDMA Write targets a GPU buffer, the NIC DMAs the incoming data directly to GPU HBM via PCIe P2P.
  3. The CPU is not involved in the data transfer at all—it only sets up the operation (posts the WR, manages QP state).
Requirements:
  • NVIDIA GPU driver with nvidia-peermem kernel module loaded (replaces the older nv_peer_mem module)
  • PCIe topology: NIC and GPU must be on the same PCIe root complex (or use PCIe ACS override). In DGX H100, each ConnectX-7 is affinity-mapped to a specific GPU via the PCIe switch topology.
  • NCCL integration: NCCL automatically uses GPUDirect RDMA when available. Verify with NCCL_DEBUG=INFO—look for NET/IB : Using GPUDirect RDMA in logs.
Performance impact:

Without GPUDirect RDMA, inter-node GPU communication is limited by the CPU memory bandwidth (~200 GB/s on a Xeon) and requires two PCIe traversals (NIC→CPU, CPU→GPU). With GPUDirect RDMA, the path is NIC→GPU directly, achieving full PCIe Gen5 x16 line rate (~63 GB/s per direction, per NIC-GPU pair). On a DGX H100 with 8 NICs, this provides up to ~400 GB/s aggregate network injection bandwidth.

GPUDirect Storage (GDS)

GPUDirect Storage extends the direct-path concept to storage I/O. Instead of reading data from NVMe into CPU memory and then copying to GPU, GDS enables a direct DMA path from the NVMe controller (or NVMe-over-Fabrics target) to GPU HBM.

  • API: cuFile API (part of CUDA toolkit). Applications call cuFileRead()/cuFileWrite() instead of standard POSIX I/O.
  • Supported filesystems: ext4 (with O_DIRECT), XFS, Lustre, GPFS/Spectrum Scale, WekaFS, VAST Data
  • NVMe-oF: GDS works over NVMe over Fabrics (RDMA or TCP), enabling remote storage arrays to DMA directly to GPU memory.
  • Use case: Training data loading (checkpointing, dataset reading), especially for large models where loading terabytes of checkpoint data through CPU memory becomes a bottleneck.

GDS reduces checkpoint load time by 2–5x depending on storage backend and data sizes. It also frees CPU memory bandwidth for other tasks (preprocessing, CPU-side computation).

GPUDirect Peer-to-Peer (P2P)

GPUDirect P2P enables direct memory access between GPUs within the same node, without staging through CPU memory. The interconnect can be:

  • NVLink: In DGX H100, all 8 GPUs are connected via NVLink 4.0 (4th generation NVSwitch), providing 900 GB/s bidirectional bandwidth per GPU. This is the primary intra-node communication path and is roughly 7x faster than PCIe Gen5.
  • PCIe: If NVLink is not available (or for GPUs not directly connected via NVLink), P2P falls back to PCIe peer-to-peer. This is slower but still avoids the CPU memory copy.

NCCL automatically uses NVLink for intra-node collectives and GPUDirect RDMA for inter-node. The combination provides the hierarchical communication pattern needed for efficient data-parallel and tensor-parallel training.

Why this matters for SREs: When debugging NCCL performance issues, the first things to check are: (1) Is nvidia-peermem loaded? (lsmod | grep nvidia_peermem). (2) Are NICs affinity-mapped to the correct GPUs? (check nvidia-smi topo -m). (3) Is GPUDirect actually being used? (set NCCL_DEBUG=INFO and look for GDR/P2P messages). A single misconfigured NIC-GPU affinity can reduce training throughput by 20–40% due to cross-NUMA PCIe traffic.

Knowledge Check

Test your understanding of physical networking concepts for GPU Superpod infrastructure.

Question 1: InfiniBand Lossless Behavior

InfiniBand achieves lossless behavior by design. What is the fundamental mechanism that prevents packet drops on a healthy IB fabric, and how does this differ from how RoCE achieves lossless behavior on Ethernet?

Correct answer: B. InfiniBand's credit-based flow control is a link-layer mechanism where the receiver explicitly grants credits (buffer space) to the sender. A sender will never transmit without credits, so buffer overflow is physically impossible on a healthy link. This requires zero configuration—it is inherent to the protocol. RoCE, running over Ethernet, must retrofit lossless behavior using PFC (per-priority pause frames), ECN (congestion marking), and DCQCN (end-to-end rate control). All three must be configured correctly and consistently across every switch in the path. This is one of the primary operational advantages of InfiniBand over RoCE for RDMA workloads.

Question 2: Transceiver Form Factors and Breakout

You have a Quantum-2 (QM9700) InfiniBand switch with 64 ports of QSFP-DD (400Gb/s NDR). You need to connect to a legacy system that has ConnectX-6 adapters with QSFP56 ports (HDR, 200Gb/s). What is the correct approach?

Correct answer: B. NDR (400Gb/s) ports support 2x200G breakout mode, splitting the 8 lanes (100Gb/s each) into two groups of 4 lanes (200Gb/s each, matching HDR speed). A breakout cable with a QSFP-DD connector on the switch end and two QSFP56 connectors on the host end is the standard solution. This is a common scenario when upgrading spine switches to NDR while leaf or host connections remain at HDR. Option D is partially true—QSFP-DD cages do accept QSFP56 modules—but the port must still be reconfigured, and you get only one 200G connection instead of potentially two from the breakout approach.

Question 3: GPUDirect RDMA Troubleshooting

A user reports that their multi-node NCCL training job is running 35% slower than expected. You run NCCL_DEBUG=INFO and notice the logs say NET/IB : Using [Send/Recv] (no GPUDirect) instead of NET/IB : Using GPUDirect RDMA. What are the most likely causes, and what would you check first?

Correct answer: B. The most common cause of GPUDirect RDMA not being used is the nvidia-peermem module not being loaded. This module (which replaced the older out-of-tree nv_peer_mem module starting with CUDA 11.4 / driver 470+) registers a callback with the RDMA core subsystem so that ibv_reg_mr() can handle GPU virtual addresses. Without it, the NIC cannot access GPU memory and NCCL falls back to staging through host memory. The second most common cause is PCIe topology issues—if ACS (Access Control Services) is enabled in the BIOS or on PCIe bridges between the NIC and GPU, P2P DMA transactions are blocked. Check with lspci -vvv | grep ACSCtl. Option C is incorrect: GPUDirect RDMA works with both InfiniBand and RoCEv2. Option A is incorrect: NCCL does not fall back from GPUDirect due to congestion—it is a capability check at initialization time.
