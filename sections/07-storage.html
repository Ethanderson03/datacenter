Storage Architecture for GPU Clusters

From NVMe flash to parallel file systems — feeding data to thousands of GPUs without starving the pipeline

1. Storage Requirements for GPU Clusters

The Fundamental Problem

A single NVIDIA H100 GPU can consume data at roughly 3.35 TB/s of HBM3 memory bandwidth. A DGX H100 node with 8 GPUs provides 26.8 TB/s of aggregate memory bandwidth. When thousands of GPUs train in parallel, the storage subsystem must pre-stage, stream, and checkpoint data at rates that dwarf traditional enterprise workloads. If storage cannot keep up, GPUs sit idle — wasting thousands of dollars per hour in compute capacity.

Key Data Categories

Training Data

Large language model training corpora routinely reach petabyte scale. The Common Crawl dataset alone is ~400 TB compressed. Multimodal models combining text, images, audio, and video push datasets even larger. This data must be accessible to every GPU node with high throughput and low latency. Typical access patterns are sequential reads with large block sizes (1–16 MB), though data augmentation and random sampling can introduce random read patterns that stress IOPS capacity.

Checkpoint Storage

During training, model checkpoints are saved every N steps (commonly every 500–2,000 steps) as a fault-tolerance mechanism. A single checkpoint for a 70B-parameter model using mixed precision (FP16/BF16 weights + FP32 optimizer states) can consume 500 GB–1.5 TB per save. For a 175B-parameter model, checkpoints can exceed 3 TB. With distributed training, each rank may write its own shard simultaneously, creating massive burst write patterns. A 4,096-GPU training run writing checkpoints every 30 minutes can generate 50+ TB of checkpoint data per day.

The GPU Starvation Problem

Consider an 8-node DGX H100 cluster (64 GPUs). If each GPU processes micro-batches at 200 GB/s effective data consumption (after caching), the cluster needs 12.8 TB/s of data delivery. Even with aggressive caching in GPU HBM and host DRAM, the storage backend must sustain hundreds of GB/s to avoid pipeline stalls. This is why parallel file systems and NVMe-oF fabrics are non-negotiable for large-scale training.

2. NAS (Network Attached Storage)

NAS provides file-level storage over standard networking protocols. Clients mount remote filesystems and interact with files using familiar POSIX semantics (open, read, write, close). NAS is the simplest shared storage model but has inherent scalability limitations for HPC workloads.

pNFS (Parallel NFS) separates the metadata path from the data path. The MDS (Metadata Server) provides layout information that tells the client exactly which data servers hold which byte ranges of a file. The client then reads/writes directly to the data servers in parallel, bypassing the MDS for I/O.

Layout types:

  • Files layout: Data is striped across multiple NFS data servers. Each stripe unit maps to a specific server. Conceptually similar to RAID-0 across NFS servers.
  • Blocks layout: Client gets block-level addresses on shared block devices (SAN). Requires SAN connectivity from all clients.
  • Objects layout: Data stored as objects on OSD (Object Storage Device) targets. Least commonly deployed.
  • FlexFiles layout (RFC 8435): Extension of files layout with mirroring support for resiliency. Used by NetApp ONTAP.

Example Linux mount with pNFS:

# Mount with pNFS enabled (default in v4.1)
mount -t nfs4 -o vers=4.1,proto=tcp nfs-server:/export /mnt/data

# Verify pNFS is active
cat /proc/self/mountstats | grep -A5 "nfs4.1"

# Check layout statistics
cat /proc/fs/nfsfs/pnfs_layoutstats

When to Use NAS in GPU Clusters

  • Shared home directories: User home dirs, scripts, Jupyter notebooks, config files
  • Small shared datasets: Datasets under 1 TB with low concurrency
  • Model code repositories: Source code, experiment configs, environment definitions
  • NOT for: High-throughput training data delivery, checkpoint I/O, or scratch space

SMB/CIFS

SMB (Server Message Block) is primarily used in Windows environments. In HPC/GPU clusters running Linux, SMB is rarely the primary storage protocol. It may appear for:

  • Windows-based data preparation or visualization workstations accessing shared data
  • Integration with enterprise Active Directory environments
  • Samba servers providing cross-platform file access

SMB3 supports multichannel I/O, encryption, and direct RDMA (SMB Direct), but Linux NFS or parallel filesystems are almost always preferred for GPU cluster storage.

3. SAN (Storage Area Network)

A SAN provides block-level storage over a dedicated network. To the operating system, SAN LUNs (Logical Unit Numbers) appear as locally attached disks. Unlike NAS (which presents files), SAN presents raw blocks that must be formatted with a filesystem by the host.

Fibre Channel (FC)

FC Protocol Stack

Fibre Channel is a lossless, high-speed network protocol purpose-built for storage traffic. It operates over its own physical layer (optical fiber or copper twinax) with dedicated FC switches.

Key FC concepts:

  • WWN (World Wide Name): Unique 64-bit identifier for FC ports (analogous to MAC addresses)
  • Zoning: Logical segmentation of the FC fabric. Only devices in the same zone can communicate. Enforced at the switch level for security and stability.
  • LUN Masking: Storage-side access control determining which hosts can see which LUNs
  • Fabric Login (FLOGI): Process by which a node registers with the FC switch fabric
  • FCID: 24-bit address assigned to each port during FLOGI, used for frame routing

iSCSI (Internet Small Computer Systems Interface)

iSCSI Overview

iSCSI encapsulates SCSI commands inside TCP/IP packets, allowing block storage access over standard Ethernet infrastructure. This eliminates the need for dedicated FC HBAs (Host Bus Adapters) and FC switches.

# Discover iSCSI targets
iscsiadm -m discovery -t sendtargets -p 10.0.50.10:3260

# Login to a target
iscsiadm -m node -T iqn.2024-01.com.storage:lun0 -p 10.0.50.10 --login

# List active sessions
iscsiadm -m session -P3

# Multipath configuration (dm-multipath)
multipath -ll

SAN in GPU Clusters

Traditional SAN is uncommon in modern GPU HPC clusters. Block storage is increasingly accessed via NVMe-oF (NVMe over Fabrics) rather than FC or iSCSI. However, SANs may still appear as backend storage for parallel filesystem servers (e.g., Lustre OSS nodes connected to a shared FC SAN array for the OST backing store).

4. Parallel File Systems

The Critical Storage Layer for HPC/AI

Parallel file systems are the single most important storage technology for GPU supercomputer clusters. They stripe data across many servers, delivering aggregate throughput that scales linearly with the number of storage servers. Every major GPU Superpod deployment uses a parallel file system as the primary shared storage layer.

Lustre

Lustre Architecture

Lustre is an open-source parallel distributed file system originally developed by Cluster File Systems Inc. and now maintained by the OpenSFS community and DDN (DataDirect Networks). It is the dominant file system for HPC, used by the majority of the Top500 supercomputers, including Frontier (the first exascale system).

Core components:

  • MDS (Metadata Server): Handles all metadata operations — file names, directory structure, permissions, file layouts, striping information. Runs the MDT (Metadata Target) which stores the actual metadata on persistent storage (typically NVMe SSDs for performance).
  • MDT (Metadata Target): The storage backend for metadata. Each MDS serves one or more MDTs. Lustre 2.14+ supports DNE (Distributed Namespace) with multiple MDTs for metadata scalability. Remote directories can be striped across MDTs.
  • OSS (Object Storage Server): Handles all data I/O. Each OSS manages one or more OSTs. When a client reads or writes file data, it communicates directly with the appropriate OSS(es) in parallel.
  • OST (Object Storage Target): The actual storage devices backing file data. Typically a RAID array, JBOD, or NVMe pool formatted with ldiskfs (ext4-based) or ZFS. Each file is composed of one or more objects, each stored on a different OST.
  • MGS (Management Server): Stores configuration logs for the filesystem. Often co-located with the primary MDS. Clients contact the MGS first to obtain the filesystem configuration, then communicate with MDS and OSS directly.
  • Client: A kernel module (lustre.ko) implementing a POSIX filesystem. Clients mount Lustre and see a single unified namespace. The client handles striping — for a 4-stripe file, the client opens connections to 4 OSS servers simultaneously for parallel I/O.

GPFS / IBM Storage Scale (Spectrum Scale)

IBM Storage Scale Architecture

IBM Storage Scale (formerly GPFS — General Parallel File System) is a commercial parallel file system developed by IBM. Unlike Lustre's separate metadata and object servers, GPFS uses a shared-disk architecture where all nodes can potentially access all disks (typically via SAN). A token-based distributed locking manager coordinates access.

Key architectural features:

  • Shared-disk: Any node can read/write any disk block. No dedicated storage servers — all nodes participate in I/O.
  • Declustered RAID: Data protection is handled internally with RAID-like mechanisms spread across many disks, enabling faster rebuild times than traditional RAID.
  • Distributed Token Manager: Uses byte-range locking tokens to coordinate concurrent access. Tokens are granted and revoked dynamically.
  • Information Lifecycle Management (ILM): Policy engine for automatic data tiering based on age, access patterns, or file attributes. Data can migrate between NVMe, SSD, HDD, and tape tiers transparently.
  • Active File Management (AFM): Asynchronous replication and caching across geographically distributed clusters.
  • Data Encryption & Compression: At-rest encryption (AES-256) and inline LZ4/zlib compression.
# Create a GPFS filesystem
mmcrfs /dev/gpfs_data -F /tmp/disk_descriptors.txt -A yes -B 16M \
  -T /gpfs/data --metadata-block-size 256K -r 2 -R 3

# Mount the filesystem
mmmount /gpfs/data -a

# Check filesystem status
mmlsfs /gpfs/data
mmdf /gpfs/data         # Disk usage
mmhealth cluster show   # Cluster health

# Add disks to existing filesystem
mmadddisk /gpfs/data -F /tmp/new_disks.txt

# Create a fileset (logical partition)
mmcrfileset /gpfs/data training_run_42 --inode-space new
mmlinkfileset /gpfs/data training_run_42 -J /gpfs/data/runs/42

# Set quota
mmsetquota /gpfs/data:training_run_42 --block 50T:60T --files 10M:12M

# Policy-based tiering
mmapplypolicy /gpfs/data -P /etc/mmfs/tiering.policy
# Example policy file:
# RULE 'tier_cold' MIGRATE FROM POOL 'ssd' TO POOL 'hdd'
#   WHERE ACCESS_AGE > 30 DAYS
#   AND FILE_SIZE > 1GB

# Performance tuning
mmchconfig pagepool=32G -i    # Increase page cache
mmchconfig prefetchAggressiveness=2 -i
mmchconfig maxMBpS=20000 -i   # Max throughput per node

WekaFS (Weka.io)

WekaFS Overview

WekaFS is a modern, software-defined parallel file system built from the ground up for flash storage and GPU/AI workloads. It has rapidly gained adoption in GPU cloud infrastructure, notably at CoreWeave, Lambda Labs, and other AI-focused providers. Weka holds multiple IO-500 records for performance.

Key technical characteristics:

  • Architecture: Fully distributed — no separate metadata or data servers. Every Weka node handles both metadata and data. Uses a shared-nothing design with consistent hashing for data placement.
  • Protocol: Full POSIX compliance with a custom kernel-bypass client (DPDK-based). The client communicates directly with Weka backends, bypassing the Linux kernel network stack for minimal latency.
  • Performance: Latency as low as ~100–200 μs for small I/O. Throughput scales linearly — a 100-node Weka cluster can deliver over 2 TB/s of read throughput.
  • Data Protection: Distributed parity protection (similar to erasure coding, e.g., 16+2 or 6+2). No RAID controllers needed. Cluster-wide stripe with configurable protection levels.
  • Tiering: Native tiering from NVMe (hot) to S3-compatible object storage (warm/cold). Data is automatically moved based on access patterns. The S3 tier can be any S3 endpoint: AWS S3, MinIO, Ceph RGW, etc.
  • Snap-to-Object: Snapshots are created locally and can be asynchronously backed up to S3, enabling backup and disaster recovery without impacting primary performance.
  • Scale: Single namespace scales to exabytes of capacity and trillions of files.
  • GPU Integration: Supports NVIDIA GPUDirect Storage (GDS) for direct DMA transfers from NVMe storage to GPU memory, bypassing the CPU entirely.
# Install Weka agent on a backend node
curl -fsSL https://get.weka.io | sh

# Create a cluster (from management node)
weka cluster create backend-{01..20} --host-ips 10.0.10.1-10.0.10.20

# Add NVMe drives to the cluster
weka cluster drive add --host backend-01 /dev/nvme0n1 /dev/nvme1n1

# Create a filesystem with tiering to S3
weka fs create training_data --ssd-capacity 500TiB \
  --total-capacity 5PiB \
  --obs-name s3-backup \
  --data-drives 16 --parity-drives 2

# Mount on compute nodes (POSIX client)
mount -t wekafs backend-01/training_data /mnt/weka

# Mount with DPDK for kernel-bypass (lowest latency)
mount -t wekafs -o net=udp backend-01/training_data /mnt/weka
# Or with DPDK:
mount -t wekafs -o net=dpdk,num_cores=2 backend-01/training_data /mnt/weka

# Check cluster status
weka status
# Output:
#  WekaIO v4.3.2 (CLI build 4.3.2.1)
#  Cluster: gpu-storage (UUID: a1b2c3d4-...)
#  Status:  OK (20 backends, 100% healthy)
#  Capacity: 500 TiB SSD, 4.5 PiB Object Store
#  Throughput: ~1.8 TB/s (current), 2.1 TB/s (peak)

# Create a snapshot

BeeGFS

BeeGFS Architecture

BeeGFS (originally FhGFS) is an open-source parallel file system developed by ThinkParQ (Fraunhofer Institute). It emphasizes simplicity of deployment and ease of management while delivering high parallel performance.

Components:

  • Management Service (beegfs-mgmtd): Registry for all other services. Clients and servers contact it to discover the cluster topology. Lightweight, typically runs on a single node.
  • Metadata Service (beegfs-meta): Stores directory structure, file attributes, and stripe information. Can be distributed across multiple nodes for scalability. Metadata is stored on fast SSDs/NVMe for performance.
  • Storage Service (beegfs-storage): Stores actual file data in chunks. Files are striped across multiple storage targets (similar to Lustre OSTs). Each target is a local filesystem (XFS, ext4) on the storage node.
  • Client (beegfs-client): Kernel module providing POSIX access. Supports RDMA (InfiniBand, RoCE) and TCP transports.
  • BeeGFS-on-Demand (BeeOND): Ephemeral parallel filesystem using the local NVMe SSDs in compute nodes. Created for a job, destroyed when the job completes. Excellent for burst buffer / scratch space.
# Install BeeGFS (on all nodes)
wget -q https://www.beegfs.io/release/beegfs_7.4/dists/beegfs-rhel9.repo \
  -O /etc/yum.repos.d/beegfs.repo
yum install -y beegfs-mgmtd beegfs-meta beegfs-storage beegfs-client beegfs-utils

# Initialize management service
/opt/beegfs/sbin/beegfs-setup-mgmtd -p /data/beegfs/mgmt

# Initialize metadata service (pointing to management node)
/opt/beegfs/sbin/beegfs-setup-meta -p /data/beegfs/meta -s 1 -m mgmt-node

# Initialize storage service
/opt/beegfs/sbin/beegfs-setup-storage -p /data/beegfs/storage -s 1 -i 101 -m mgmt-node

# Configure client
/opt/beegfs/sbin/beegfs-setup-client -m mgmt-node
echo "connUseRDMA = true" >> /etc/beegfs/beegfs-client.conf

# Mount
mount -t beegfs beegfs_nodev /mnt/beegfs

# Set stripe pattern (4 targets, 2MB chunks)
beegfs-ctl --setpattern --chunksize=2M --numtargets=4 /mnt/beegfs/training/

# Show stripe info

# Check storage targets

5. Object Storage

Object storage organizes data as discrete objects in a flat namespace, each identified by a unique key. Unlike POSIX file systems, there is no directory hierarchy — objects are stored in buckets (containers) and accessed via HTTP-based APIs (primarily S3-compatible). This model scales massively and is ideal for storing large, immutable datasets.

S3-Compatible Object Storage

The Amazon S3 API has become the de facto standard for object storage. Key operations:

  • PUT: Upload an object (with metadata and tags)
  • GET: Download an object (supports byte-range requests for partial reads)
  • DELETE: Remove an object
  • LIST: Enumerate objects in a bucket (prefix-based filtering)
  • Multipart Upload: Upload large objects in parallel chunks (required for objects >5 GB)

Object storage is not POSIX: no open/close semantics, no byte-range writes to existing objects (objects are immutable once written), no hard/soft links, no rename (copy + delete instead), and no append operations (with some vendor extensions). Applications must be designed for PUT/GET patterns.

MinIO

MinIO is an open-source, S3-compatible object store designed for high performance on commodity hardware. It is written in Go and is commonly used as the object storage tier behind parallel file systems or as a standalone data lake.

Performance: A single MinIO server with NVMe drives can deliver 35+ GB/s GET and 12+ GB/s PUT throughput. Distributed MinIO across 32 nodes with NVMe can achieve 325+ GiB/s GET.

# Deploy MinIO distributed cluster (4 nodes, 4 drives each)
minio server http://minio-{1...4}/data/disk-{1...4}

# Configure mc (MinIO Client)
mc alias set gpu-store http://minio-1:9000 ACCESSKEY SECRETKEY

# Create a bucket for training data
mc mb gpu-store/training-data

# Upload a dataset with parallelism
mc cp --recursive /local/datasets/imagenet/ gpu-store/training-data/imagenet/ \
  --storage-class REDUCED_REDUNDANCY

# Set lifecycle policy (delete old checkpoints after 30 days)
mc ilm rule add gpu-store/checkpoints --expire-days 30 --prefix "epoch-"

# Enable bucket versioning
mc version enable gpu-store/model-artifacts

# Server-side encryption with KMS
mc encrypt set sse-s3 gpu-store/sensitive-data

# Replicate to remote site
mc replicate add gpu-store/training-data --remote-bucket backup-store/training-data

# Erasure coding: MinIO uses Reed-Solomon EC by default
# With 16 drives: EC:8 means 8 data + 8 parity (can lose 8 drives)
# With 16 drives: EC:4 means 12 data + 4 parity (can lose 4 drives)

6. Ceph

Ceph: Unified Distributed Storage

Ceph is an open-source distributed storage system that provides three storage interfaces on top of a single unified storage backend (RADOS):

  • RBD (RADOS Block Device): Block-level storage. Thin-provisioned virtual disks. Used extensively as Kubernetes PersistentVolumes, OpenStack Cinder volumes, and VM disks.
  • CephFS: POSIX-compliant distributed file system. Supports multiple active MDS servers for metadata scalability. Can serve as a parallel file system for HPC (though not as performant as Lustre/Weka for extreme-scale workloads).
  • RGW (RADOS Gateway): S3-compatible and Swift-compatible object storage interface. Production-grade S3 implementation used by many organizations.

CRUSH Algorithm

Controlled Replication Under Scalable Hashing (CRUSH)

CRUSH is the heart of Ceph's data distribution. Instead of maintaining a centralized lookup table mapping objects to storage devices (which would be a bottleneck), CRUSH uses a deterministic algorithmic approach. Any client can compute the exact location of any object given:

  1. The object name (hashed to a placement group, or PG)
  2. The CRUSH map (a hierarchical description of the storage topology: data centers, racks, hosts, OSDs)
  3. The CRUSH rules (placement policies, e.g., “place 3 replicas in 3 different racks”)

This eliminates any single point of failure for data location lookups and enables Ceph to scale to thousands of OSDs. When OSDs are added or removed, CRUSH deterministically rebalances the minimum amount of data.

MGR (Manager) Provides monitoring, metrics (Prometheus), dashboard, orchestration, and module framework. 2 (active + standby) Moderate CPU and RAM
RGW (RADOS Gateway) S3/Swift-compatible HTTP gateway. Stateless — scale horizontally behind a load balancer. 2+ (behind LB) CPU-bound for encryption; RAM for caching

7. NVMe and NVMe-oF

NVMe (Non-Volatile Memory Express)

NVMe is a storage protocol designed specifically for flash (and future persistent memory) storage, running over the PCIe bus. It replaced the legacy AHCI/SATA protocol stack which was designed for spinning disks and had fundamental architectural limitations.

Feature AHCI (SATA) NVMe
Command Queues1 queue, 32 commands65,535 queues, 65,536 commands each
Interrupt HandlingSingle interrupt, coalescingMSI-X per-queue interrupts, direct CPU steering
Latency~6–10 ms (HDD), ~100 μs (SSD via SATA)<10 μs (PCIe Gen4/5 NVMe)
IOPS (peak)~100K (SATA SSD)1–2M+ IOPS per device
Bandwidth~550 MB/s (SATA III)7–14 GB/s (PCIe Gen4/Gen5 x4)
CPU OverheadHigh (register-based, complex state machine)Low (streamlined, minimal CPU cycles per I/O)

A single PCIe Gen5 x4 NVMe SSD (e.g., Samsung PM1743, Kioxia CM7) delivers ~14 GB/s sequential read and ~2M+ random IOPS. A DGX H100 node with 8 local NVMe SSDs provides ~100 GB/s of local storage bandwidth.

NVMe-oF (NVMe over Fabrics)

Extending NVMe Across the Network

NVMe-oF extends the NVMe protocol over network fabrics, allowing a host to access remote NVMe devices with near-local performance. The remote NVMe namespaces appear as local /dev/nvmeXnY devices on the initiator host.

Transport Underlying Network Added Latency Throughput Pros
NVMe/RDMA InfiniBand or RoCEv2 <10 μs Wire-speed (200–400 Gb/s) Lowest latency; zero-copy; kernel bypass
NVMe/TCP Standard Ethernet ~30–80 μs Near wire-speed (100–400 GbE) No special NIC needed; works over any TCP/IP network
NVMe/FC Fibre Channel ~20–50 μs 32–128 Gb/s per port Leverages existing FC infrastructure

NVIDIA GPUDirect Storage (GDS)

Direct Path: NVMe to GPU Memory

GPUDirect Storage (GDS) enables a direct DMA data path between NVMe storage (local or NVMe-oF) and GPU memory, completely bypassing the CPU and system memory. Without GDS, data must follow this path: NVMe → CPU/DRAM bounce buffer → GPU HBM. With GDS, the path becomes: NVMe → GPU HBM (via PCIe peer-to-peer or network RDMA).

Benefits:

  • Eliminates CPU bounce buffer copies (reduces latency by 30–50%)
  • Frees CPU cores for other work (data preprocessing, scheduling)
  • Reduces system memory bandwidth consumption
  • Achieves up to ~12 GB/s per GPU from local NVMe
  • Supported by cuFile API (part of CUDA toolkit) and integrated into major frameworks via kvikio library

8. Storage Networking

Separation of Concerns: Dedicated Storage Fabric

In GPU Superpod deployments, the storage network is physically separate from the GPU compute fabric (InfiniBand for NCCL/collective communication). Storage typically runs over a dedicated Ethernet network (100GbE or 200GbE) to avoid interference with GPU-to-GPU traffic. Some deployments use a dedicated InfiniBand partition for storage (LNET over o2ib), while others use separate Ethernet switches entirely.

ZFS for Storage Servers

ZFS is a combined volume manager and file system that provides data integrity, compression, and flexible storage management. It is used as the backend filesystem on Lustre OST/MDT devices and on Ceph OSD nodes.

# Create a ZFS pool with RAID-Z2 (double parity, similar to RAID 6)

  /dev/nvme0n1 /dev/nvme1n1 /dev/nvme2n1 /dev/nvme3n1 \
  /dev/nvme4n1 /dev/nvme5n1 /dev/nvme6n1 /dev/nvme7n1

# Add L2ARC read cache

# Add SLOG (ZIL) for synchronous write acceleration

BlueField DPU for Storage Offload

NVIDIA BlueField DPU: Storage Acceleration

The NVIDIA BlueField-3 DPU (Data Processing Unit) integrates an ARM CPU complex, ConnectX-7 network adapter, and hardware accelerators onto a single SmartNIC. For storage, it offloads:

  • NVMe-oF Target: The DPU can present local NVMe devices as NVMe-oF targets without consuming host CPU cycles. NVIDIA SNAP (Software-defined Network Accelerated Processing) exposes NVMe namespaces as virtio-blk devices to VMs/containers.
  • Encryption: AES-XTS hardware acceleration for at-rest and in-flight storage encryption at line rate (400 Gb/s).
  • Compression: Hardware LZ4/Deflate compression engine for inline data reduction.
  • Erasure Coding: Hardware-accelerated Reed-Solomon encoding for distributed parity without host CPU overhead.
  • Storage virtualization: Multi-tenant NVMe namespace isolation with SR-IOV and hardware-enforced QoS.

Knowledge Check: Storage Architecture

Question 1

In a Lustre parallel file system, what is the primary role of the OSS (Object Storage Server), and how does file striping improve performance?

Correct answer: B. The OSS (Object Storage Server) is responsible for storing and serving file data through its OSTs (Object Storage Targets). When a file is striped with lfs setstripe -c 4, the file is divided into fixed-size chunks that are distributed round-robin across 4 different OSTs, potentially on 4 different OSS nodes. The Lustre client opens parallel connections to all relevant OSS servers simultaneously, achieving N-times the throughput of a single server (where N is the stripe count). This is the fundamental mechanism that allows Lustre to deliver hundreds of GB/s of aggregate bandwidth. The MDS (not the OSS) handles metadata, and striping is about parallelism, not redundancy or compression.

Question 2

What is the key advantage of NVMe-oF/RDMA over NVMe-oF/TCP for GPU cluster storage, and how does GPUDirect Storage (GDS) further optimize the data path?

Correct answer: A. NVMe-oF/RDMA adds less than 10 microseconds of latency to local NVMe access by using RDMA (Remote Direct Memory Access), which bypasses the kernel network stack entirely and performs zero-copy transfers directly between NIC and application buffers. NVMe/TCP, while more universally deployable over standard Ethernet, must traverse the full TCP/IP stack, adding 30–80 μs of latency. GPUDirect Storage takes this further by establishing a direct DMA path from the NVMe device (local or remote via NVMe-oF) to GPU HBM memory. Without GDS, data follows: NVMe → system DRAM (bounce buffer) → GPU HBM (two copies, consuming CPU cycles and memory bandwidth). With GDS: NVMe → GPU HBM (one transfer, zero CPU involvement).

Question 3

A team is designing storage for a 512-GPU training cluster. Training data is 2 PB, and checkpoints of 800 GB are written every 20 minutes. They need at least 400 GB/s sustained read throughput and POSIX semantics. The data should automatically tier to S3 when cold. Which storage architecture is most appropriate?

Correct answer: B. WekaFS is the best fit for this workload profile. The requirements include: (1) 400 GB/s sustained throughput, which WekaFS can achieve with linear scaling across NVMe-backed nodes; (2) POSIX semantics, which WekaFS fully supports; (3) automatic S3 tiering, which is a native built-in feature of WekaFS (not a bolt-on script); and (4) the scale of 512 GPUs with 2 PB of data and frequent large checkpoint writes. Option A (pNFS) cannot realistically deliver 400 GB/s and has no native S3 tiering. Option C (MinIO) does not provide POSIX semantics, which are required for most ML training frameworks that use standard file I/O. Option D (CephFS) could technically work but would struggle to consistently deliver 400 GB/s at the required latency — CephFS is not optimized for GPU/HPC workloads at this scale, and 3x replication for 2 PB would require 6 PB of raw capacity. WekaFS with its kernel-bypass client, flash-optimized architecture, and native tiering is purpose-built for exactly this use case.
